# 观微后端工程质量审查报告

> 审查范围：`backend/` 全部 Python 源码（55 个文件）
> 审查维度：安全性、架构设计、代码质量、测试覆盖

---

## 总评

项目架构分层合理（routes → services → agents → pipeline），DEV_MODE 环境保护机制设计到位，测试基础设施（事务回滚、工厂模式）质量不错。但在安全、并发、工程规范三个维度存在需要优先处理的硬伤。

---

## P0 — 必须立即修复

### 1. 多个 API 端点完全无鉴权

- `POST /api/v1/models/set-provider` — 任何人可切换 LLM 提供商，重定向流量到恶意端点
- `POST /api/v1/evidence` — 任何人可向任意 report 注入伪造证据
- `GET /api/v1/metrics/*` — 系统指标和业务数据裸奔

建议：删除 `routes.py` 中的 `set-provider`（admin 里已有鉴权版本），evidence 创建加认证，metrics 移到 admin 路由。

### 2. CORS 配置安全漏洞

```python
allow_origins=["*"],
allow_credentials=True,
```

`*` + credentials 是经典反模式。生产环境必须配置明确的 origin 白名单。

### 3. Collector Agent 过滤逻辑反转

```python
if any(blocked not in url.lower() for blocked in [...]):
    continue
```

条件几乎永远为 True（URL 不可能同时包含所有被屏蔽域名），导致几乎所有搜索结果被过滤掉。应为 `if any(blocked in url.lower() for blocked in [...])`。这是核心求证功能，等于搜集 Agent 基本不工作。

### 4. Moderator 正则表达式全部无效

```python
r"国家领导人[姓名]+"    # [姓名] 是字符类，匹配"姓"或"名"
r"中[南海│央]"          # │ 是 Unicode U+2502，不是正则的 |
r"[杀│人]"              # 同上
```

所有敏感词正则都用了 `[]` 字符类 + `│`（全角竖线），语法上不会报错但完全不会按预期匹配。等于内容审核只剩 LLM 一条线在撑。

### 5. 审核失败默认放行

`_moderation_node` 中，Agent 执行异常时 `passed=True, action="通过"`。安全审核崩溃 = 所有内容通过，策略过于宽松。安全场景应 fail-closed（默认拦截）。

### 6. 依赖无版本锁定

`requirements.txt` 所有包无 `==` 或 `>=` 约束。任何 breaking change 可直接搞崩生产。

## P1 — 高优先级

### 安全类

| 问题 | 位置 | 说明 |
|------|------|------|
| JWT 密钥硬编码 | `auth.py:34` | DEV 模式下用固定字符串，运维误用 DEV_MODE 即全量沦陷 |
| `int(user_id)` 无异常保护 | `auth.py:71` | 非法 payload 触发 500 而非 401 |
| Token 7 天有效 + 无吊销 | `auth.py:21` | 泄露后无法撤销，admin 用户尤其危险 |
| 注册接口无速率限制 | `user_routes.py` | login 做了限流，register 遗漏 |
| 密码无强度校验 | `schemas.py:11` | 可注册密码为 `"1"` 的账户 |
| SSRF 风险 | `web_search.py` | `fetch_page` 可抓取任意 URL 包括内网地址 |
| 演示参数暴露生产 API | `routes.py:28-31` | `demo_crash_trigger` / `crash_probability` 不应出现在生产模型 |

### 功能 Bug

| 问题 | 位置 | 说明 |
|------|------|------|
| 积分类型标记错误 | `guess_service.py` | 佐证点赞/提交的积分 `record_type="guess_correct"`，应为 `"evidence_upvoted"` 等 |
| 批量揭瓜不结算积分 | `admin_routes.py:396-414` | 单个揭瓜有积分结算，批量揭瓜只改状态没结算 |
| 注销用户仍可正常使用 | `auth.py` | `get_current_user` 不检查注销状态 |
| 猜瓜并发竞态 | `guess_service.py` | 先查后插，无唯一约束保护，TOCTOU |
| `participant_count` 非原子更新 | `guess_service.py` | `+= 1` 并发丢失，应用 `update().values()` |

### 架构类

| 问题 | 说明 |
|------|------|
| Commander 对非 LLM 节点切换 provider | Collector 用 httpx 不用 LLM，切换 provider 对它无效 |
| `_switch_to_backup` 修改全局状态 | 影响所有并发请求，不是请求级隔离 |
| 配置分散 | `DEV_MODE`、`SECRET_KEY`、`DATABASE_URL` 散落各文件，应用 `pydantic-settings` 统一管理 |
| 种子数据 import 时执行 | `main.py` 顶层执行 seed，每次 import 都有副作用 |

## P2 — 中优先级

### 性能

| 问题 | 位置 | 说明 |
|------|------|------|
| 外键全缺索引 | `models.py` | 所有 ForeignKey 列无索引，数据增长后全表扫描 |
| N+1 查询 | `melon_service.py`、`admin_routes.py` | 瓜列表循环查 User，证据列表循环查 User |
| Redis `delete_pattern` 用 KEYS | `cache.py` | O(N) 全库扫描，应改用 SCAN |
| 同步 Redis 阻塞事件循环 | `cache.py` | 在 async FastAPI 中用同步 redis 库 |
| 分页 size 无上限 | `admin_routes.py` | 可传 `size=999999`，内存耗尽 |

### 代码质量

| 问题 | 说明 |
|------|------|
| `datetime.utcnow()` 已弃用 | Python 3.12+ PEP 587，应用 `datetime.now(timezone.utc)` |
| `declarative_base()` 已弃用 | SQLAlchemy 2.0 风格应用 `DeclarativeBase` |
| 错误响应格式不统一 | 三套格式并存：全局处理器 / Pydantic 模型 / 直接 dict |
| `print()` 替代 `logger` | cache.py、seed.py 等用 print，生产无法统一管控 |
| HTML 解析用正则 | `web_search.py` 应使用 beautifulsoup4 |
| LLM 客户端无关闭 | `AsyncOpenAI` 只增不减，无 `aclose()` |
