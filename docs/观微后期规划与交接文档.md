## 二、当前状态与限制

### 2.1 能跑起来，但还不是"上线状态"

目前项目处于**可演示的 Prototype 阶段**：

- 前端所有功能可用，但数据全部来自 Mock（内存/本地生成）
- 后端代码写好了，但**尚未部署**，数据库是本地 SQLite
- LLM 调用走 Groq 免费层，有速率限制（30 RPM）
- 没有真实的用户注册体系（自动登录 demo 账号）
- 图片用的是外部 URL，没有自己的图片存储

### 2.2 直接发到群里让同学们测，会遇到这些问题

| 问题 | 影响 |
|------|------|
| 没有部署，同学们打不开 | 致命 — 只能本地跑 |
| 数据是假的，所有人看到一样的内容 | 体验差 — 没有"真实社区"的感觉 |
| 用户系统形同虚设，大家没法注册自己的账号 | 无法区分用户，无法形成竞争 |
| Groq 免费层 30 RPM，500 人同时用会超限 | AI 功能全部报错 |
| 辩论房间是内存态，刷新就没了 | 多人无法进入同一房间 |
| 没有内容审核兜底，UGC 可能出问题 | 合规风险 |

## 三、500–1000 人测试需要做的事（按优先级排序）

### P0 — 不做就没法测

#### 3.1 部署上线

**前端部署**（最简单，1 天内）
- 方案 A：Vercel / Netlify — 零配置，`git push` 自动部署，免费额度足够 1000 人
- 方案 B：Cloudflare Pages — 国内访问相对友好，免费
- 方案 C：腾讯云 COS + CDN — 国内速度最快，但需要备案域名

**后端部署**（1–2 天）
- 轻量方案：一台腾讯云/阿里云轻量应用服务器（2C4G，约 50 元/月）
- 数据库：SQLite → PostgreSQL（用 Supabase 免费层或服务器本地装）
- 进程管理：PM2 或 Docker Compose

**域名与备案**
- 如果选国内服务器，需要域名备案（1–2 周）
- 如果选海外服务器（Vercel/Cloudflare），不需要备案但国内访问可能慢

#### 3.2 真实用户系统

当前状态：自动登录 demo 账号，所有人是同一个用户。

需要做的：
- 手机号/微信登录（或至少邮箱注册）
- 用户数据持久化（积分、段位、历史记录）
- 登录态管理（Token 过期、刷新）

最低成本方案：先用邮箱注册，不做短信验证。后端已有注册/登录接口，只需把数据库换成 PostgreSQL 并部署。

#### 3.3 真实数据填充

当前状态：12 条固定 Mock 瓜，所有人看到一样的。

需要做的：
- 准备 30–50 条真实"瓜"（结合时事热点）
- 写一个后台脚本或管理页面，方便后续运营添加新瓜
- 瓜的封面图需要自己存储（上传到 OSS 或服务器本地）

#### 3.4 LLM 配额管理

当前状态：Groq 免费层 30 RPM，500 人同时用必崩。

方案选择：

| 方案 | 成本 | 说明 |
|------|------|------|
| Groq 付费 | ~$5/月起 | 提高 RPM 限制 |
| 硅基流动（SiliconFlow）| 按量计费 | 国内服务，DeepSeek/Qwen 都有，价格低 |
| 阿里云百炼 | 免费额度 + 按量 | Qwen 系列，国内速度快 |
| 书生（InternLM）| 免费 9000 token/月 | 用户已有开发者计划名额，但额度很少 |
| 混合策略 | 推荐 | 核心功能用付费 API，非核心用免费层 |

建议：AI 评判和辩论生成用付费 API（硅基流动或阿里云百炼），情绪检测和方言翻译可以用本地规则引擎兜底。

### P1 — 不做能测，但体验差

#### 3.5 实时通信（WebSocket）

当前辩论房间是内存态，每个人看到的数据不一样。需要：
- WebSocket 连接，让同一房间的人看到实时发言
- 房间状态持久化（Redis 或数据库）
- 断线重连机制

#### 3.6 图片存储

当前封面图是外部 URL。需要：
- 对象存储（阿里云 OSS / 腾讯云 COS / Cloudflare R2）
- 图片上传接口
- 缩略图生成

#### 3.7 内容审核加强

当前只有前端敏感词过滤。500 人 UGC 需要：
- 后端接入内容审核 API（阿里云内容安全 / 腾讯云天御）
- 图片审核（OCR + 鉴黄）
- 举报处理流程

### P2 — 锦上添花

#### 3.8 打包 APK

用 Capacitor 把 Web 应用包成安卓 APK：
- `npm install @capacitor/core @capacitor/cli @capacitor/android`
- `npx cap init` → `npx cap add android` → `npx cap sync`
- 用 Android Studio 构建 APK
- 预计 1 天工作量

#### 3.9 数据埋点与监控

- 接入 Umami / Plausible（隐私友好的网站分析）
- 错误监控：Sentry（免费层 5000 事件/月）
- 后端日志：结构化日志 + 简单的管理面板

#### 3.10 运营工具

- 管理后台：添加/编辑/下架瓜
- 数据看板：DAU、辩论参与率、求证准确率
- 推送通知：新瓜上线、辩论结果通知

## 四、推荐执行路线

### 第一阶段：最小可测版本（3–5 天）

目标：让同学们能打开链接、注册账号、看到真实内容、参与辩论。

```
Day 1: 前端部署到 Vercel/Cloudflare Pages
Day 1: 后端部署到轻量服务器，数据库换 PostgreSQL
Day 2: 打通前后端联调，真实注册/登录
Day 2: 准备 30+ 条真实瓜数据，导入数据库
Day 3: 图片存储接入 OSS
Day 3: LLM API 切换到付费方案（硅基流动）
Day 4: 辩论房间 WebSocket 化 + 状态持久化
Day 5: 端到端测试，修 bug
```

### 第二阶段：体验优化（1–2 周）

- 内容审核接入
- 打包 APK（方便不会用浏览器的同学）
- 数据埋点
- 根据第一波反馈修 bug、调 UI

### 第三阶段：运营增长（持续）

- 管理后台
- 每周更新 3–5 条新瓜（结合热点）
- 辩论赛事运营（周赛、月赛）
- 段位排行榜激励

## 五、技术债务清单

| 编号 | 债务 | 影响 | 建议处理时机 |
|------|------|------|-------------|
| T1 | `withFallback` 的 `backendAvailable` 状态在多 Tab 间不同步 | 切换 Tab 可能误用 Mock | 部署后去掉 Mock 逻辑 |
| T2 | 辩论房间数据在内存，刷新丢失 | 多人无法协同 | 第一阶段必须解决 |
| T3 | 前端路由有些 `navigate(-1)` 残留 | 可能产生导航死循环 | 已全部改为显式路由，但需回归测试 |
| T4 | DeviceFrame 缩放依赖 `window.innerHeight`，窗口 resize 不响应 | 调整窗口大小后比例失调 | 加 resize 监听 |
| T5 | Mock 数据中瓜的 ID 是字符串，后端期望数字 | 联调时可能类型不匹配 | 联调时统一 |
| T6 | Groq API Key 硬编码在前端 | 安全风险 | 部署后移到后端代理 |
| T7 | 没有 E2E 测试 | 改代码容易引入回归 | 核心流程补测试 |
| T8 | Tailwind v4 的 `@theme` 自定义色在部分场景不生效 | 偶发样式问题 | 遇到时修 |

## 七、给接手同学的话

这个项目的核心设计理念是**"过程即娱乐"**——不是简单地告诉你一个消息是真是假，而是把求证和辩论的过程变成一场好看的"瓜"。AI 在主观题上的随机性不是 bug，是 feature：每场辩论都是独一无二的，这就是看点。

前端已经能完整跑通所有流程（Mock 模式下），你可以先 `npm run dev` 本地体验一下，重点看：
1. 瓜田 → 点击瓜 → 辩论房间的完整链路
2. PC 端三栏布局 + 右上角"真机模拟"切换安卓视图
3. 工具箱的几个取证功能

如果要部署上线给更多人用，优先看第三节的 P0 部分。有任何架构上的疑问，可以参考 `观微开发思路复盘.md`，里面有完整的设计决策记录。
