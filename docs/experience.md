## 一、写作方法与阅读指南

## 二、国外真实案例深度复盘

### 2.1 Klarna：从“全 AI 替换”到“返聘人工 + AI 混合”

#### 初始设计与成功指标
- 2024 年初上线 OpenAI 驱动的 AI 助手。
- 首月处理 230 万对话，解决时间从 11 分钟降至 <2 分钟。
- 对外宣称等效 700+ 人工，年收益约 4000–6000 万美元。
- 设计假设：大多数客服请求是 routine（订单状态、退换货政策、物流查询），可用单一大模型 Agent 端到端解决。

#### 为什么最初看起来成功
- 高频、标准化问题占比高，模型在 pattern-matching 和 FAQ 上表现极好。
- 平均指标（解决时间、成本、deflection rate）非常好看。
- 演示场景中，AI 回答流畅、准确，符合“降本”叙事。

#### 真实失效点（2024 下半年–2025）
1. **长尾/情感/边缘案例摧毁体验**
   - 纠纷、欺诈、hardship（经济困难请求延期还款/退款）、复杂售后协商等场景，AI 给出“技术正确但情感空洞”的回答。
   - 客户感到被敷衍，投诉激增，重复联系率回升。
2. **自信幻觉（confidently wrong）**
   - 对未知或边界问题，AI 以高置信度给出错误答案，客户按错误指引操作后问题恶化。
3. **上下文丢失导致人工兜底体验崩盘**
   - 升级到人工时，人工客服看不到完整对话上下文，客户需要重复解释。
   - 人工成为“擦屁股”角色，而非高价值服务者。
4. **过度优化成本，忽视 CSAT 和品牌风险**
   - 平均指标好，但长尾案例的负面口碑和合规风险被低估。

#### CEO 的公开复盘
- 承认“太注重成本，质量下降，我们走太远了”。
- 2025 年开始重新雇佣人工客服。
- 从“AI 替代”转向“AI routine + 人工 high-stakes”的混合模式。

#### 架构调整（不是简单加人，而是重构路由与兜底）
1. **分层路由 + confidence gate**
   - 低置信度、复杂意图、情绪化信号自动升级，不再让 AI 猜测。
   - 加“intent clarification”步：AI 先问澄清，而非直接给答案。
2. **完整上下文传递**
   - 升级时携带全对话历史 + AI 已尝试路径 + 客户情绪标签。
   - 人工接手时能看到“AI 为什么卡住了”。
3. **人类 VIP 定位**
   - 把人工服务重新定位为“premium differentiator”，处理情感、纠纷、高风险判断。
   - AI 处理 80–90% routine，人工 owns 剩余 10–20% 但决定品牌忠诚度。
4. **知识新鲜度与外部验证**
   - 对可能过期的知识加 freshness check。
   - 对政策类问题引入规则引擎复核。

#### 可复用模式：Klarna 弧线
> 全 AI 替换 → 平均指标好但长尾崩盘 → 返聘人工 → 混合模式 + 分层路由 + 上下文传递。

这是客服 Agent 设计中最值得反复参考的完整弧线。它说明：**平均解决率不能代表生产可用性，长尾失败率才是品牌/合规的杀手。**

### 2.2 Salesforce Agentforce：平台级客服 Agent 的真实曲折

#### 背景
- Agentforce 是企业级客服 Agent 中较成熟的平台方案之一。
- 支持工具调用、Data Cloud RAG、Agent Graph 编排、human escalation。
- 内部“Customer Zero”部署处理 150 万+ 支持请求；SDR Agent 生成管道。

#### 早期失败
1. **数据不一致导致 30% “I don’t know”率**
   - 知识库陈旧、营销页与产品实际行为不一致，Agent 无法给出可靠答案。
   - 客户反复问同一个问题，体验差。
2. **guardrails 过严导致合法查询被拒答**
   - 早期采用过度 restrictive/prescriptive prompts，试图用大量否定句限制模型。
   - 结果：合法问题（如“如何集成竞品工具”）也被拒答。
3. **从 prescriptive 到 goal-oriented**
   - 调整后不再罗列“你不能做什么”，而是定义“目标是什么 + 边界在哪里 + 什么情况下升级”。
   - 模型在约束内自主规划，拒答率下降，解决率上升。

#### 架构调整
- 深度 CRM 集成：Agent 直接读取客户、订单、工单、合同数据。
- Data Cloud 作为统一数据层，减少孤岛。
- 迭代降低 “I don’t know” 率：从数据治理、知识库更新、prompt 重构三面入手。

#### 真实效果还是演示？
- 150 万+ 内部请求是真实生产负载。
- 但早期 30% “I don’t know” 说明：**平台能力强，不代表部署后就能直接用；数据治理是前置条件。**

### 2.3 Siemens + Amazon Connect：语音自助的边界

#### 成功点
- 90% 呼叫自主处理。
- 消除 DTMF 菜单，改为自然语言路由。
- 等待时间下降，首解率提升。

#### 为什么适合 Siemens
- 场景以信息查询、路由、状态更新为主，动作空间明确。
- 与 Amazon Connect 深度集成，语音 pipeline、ASR/TTS、后台系统调用统一。

#### 未公开的边界
- 复杂投诉、跨部门协调、情感安抚仍转人工。
- 高自主率建立在“任务范围狭窄”和“后台系统成熟”之上，不是通用 AI 客服模板。

### 2.4 Swiggy + Databricks：RAG 规模化的真实条件

#### 成功点
- 企业规模 RAG 处理数千并发。
- 亚秒延迟，100% 自动化常见查询。

#### 关键设计
- Hybrid 检索（向量 + 关键词 + 图）+ reranking + 引用来源。
- 数据来源统一、更新机制明确。

#### 与失败案例的对比
- Salesforce 早期也做 RAG，但因知识库治理差而失败。
- Swiggy 的成功说明：**RAG 的技术组件成熟，但生产效果取决于数据管道和知识治理。**

### 2.5 MavenAGI / Tripadvisor：高自主解决的代价

#### 成功点
- 93% 自主解决，解决时间降 60%。
- 票据成本从 $40 降至 $8。

#### 前提条件
- 高度结构化的售后/预订流程。
- 工具调用（查订单、改订、退款）有明确 schema 和幂等设计。
- 失败路径清晰， escalation 快速。

#### 不能外推的场景
- 如果业务以开放式咨询、情感协商、政策解释为主，93% 自主解决不可复现。

### 2.6 多 Agent 编排：三种模式的适用边界与失败率

#### 生产中的三种模式

| 模式 | 适合场景 | 失败形态 |
|------|---------|---------|
| **Pipeline / 串联** | 确定性流程（检索 → 回答；查订单 → 更新状态） | 单点故障传播；未知问题卡住；灵活性差 |
| **Supervisor / 层级** | 动态任务分解（复杂 case 路由到不同子 Agent） | Supervisor 成瓶颈；>6–10 子 Agent 时 context bloat、路由幻觉 |
| **Parallel / 并发** | 独立子任务可并行（多系统查询、多源验证） | 结果冲突、合并困难、无效并行浪费、成本高 |

#### 真实失败数据
- **handoff 失败占多 Agent 系统失败的 62%**。
  - Agent A 总结后传给 B，B 收到不完整/过时/不同版本现实。
  - 导致决策冲突、重复工作、死锁、无限循环。
- **UC Berkeley MAST 研究**：多 Agent 框架失败率 86.7%，主因：
  - context bleed（污染）
  - context explosion（历史爆炸）
  - context drift（陈旧状态）
- **ChatDev “软件公司”模拟**：正确率仅 33%。

#### 为什么单 Agent + 工具往往更优
- 70% 自称需要 multi-agent 的场景，实际用单 Agent + 多工具即可解决。
- 成本低 1/3，延迟低，调试简单。
- 只有当任务需要真正并行、需要独立验证、需要不同角色辩论时，multi-agent 才值得引入。

#### 架构演进：从点对点消息到共享内存
- 早期失败：点对点 message passing，race condition、stale data、延迟叠加。
- 调整后：共享内存 / 事件溯源架构。
  - 所有 Agent 访问统一 context layer（immutable events）。
  - 显式序列化状态 + “handoff justification”（为什么转这个 Agent）。
- 加 timeout、circuit breaker、最大迭代限制。
- 高价值流程用 staged（延迟换可靠性）；并行时加聚合与验证 Agent。

#### 真实案例：invoice 提取
- Consensus/debate 模式：错误率从 4.2% 降至 0.7%。
- 但代价：需要多数投票或独立合成 Agent，成本高、延迟高。
- 仅适合高价值、低容错、可并行的任务。

### 2.7 安全与治理：不是后期，是生产前提

#### 关键风险（OWASP Agentic AI Top 10）
- Prompt injection（直接/间接）
- 过度权限 / 工具滥用
- 目标劫持
- 内存投毒
- 供应链攻击
- 级联失败

#### 真实失败
- 早期 Agent 继承广权限或共享凭证，prompt injection 后导致数据泄露、未授权交易。
- Orphaned agents 长期存活，成为后门。
- 无 inventory 导致 shadow AI。
- **88% 组织报告过 AI 安全事件，但仅少数有全面控制。**

#### 架构设计
- 每个 Agent 唯一身份 + owner + 生命周期管理。
- Least privilege：tool + action + resource 三级作用域。
- 短时 token、just-in-time 访问、运行时 PEP/PDP 策略检查。
- 不可变审计日志；高风险动作必须 human approval。
- Red teaming 模拟 novel 攻击。

#### 不可信输入的边界（数据 vs 指令）
- 很多事故不是内部越权，而是**外部抓取的内容被当成指令执行**。
- 网页、PDF、用户上传文档中混入“忽略上述指令”等文本，被下游 Agent 当真。
- 这跟权限边界是两个维度：权限管的是“能不能做”，这条管的是“会不会把不该听的话当成命令听”。
- 设计要点：
  - 外部/不可信来源的内容进入 LLM 前，用分隔符/框架语言明确标记为“数据”而非“指令”。
  - 不要默认“检索到的内容是善意的”；对检索结果做 sanitize 和来源验证。
  - SearchAgent 抓取的公网内容直接喂给 ResearchAgent 时，必须有隔离层。

#### 可逆性分级（与 human approval 配合）
- 发一条微博和删一条数据库记录都可能被判为“高风险”，但可逆性不同，审批松紧应不同。
- 不要只按“风险等级”一刀切。
- 设计要点：
  - 每类动作标注可逆性。
  - 不可逆动作应有更高的确认门槛：双人确认、冷却期、快照备份、rollback 预案。

#### 行业差异
- 金融、医疗、政府：强制 human-in-the-loop，权限粒度到字段级；对不可信输入和不可逆动作最严格。
- 电商、内容平台：可接受更高自动化，但需审计、回滚与内容安全过滤。

### 2.8 效率与成本：Token 经济的真实账单

#### 问题
- Multi-agent token 消耗可达聊天机器人的 5–30x。
- 无限循环案例：11 天 $47k。
- 上下文重复发送占账单 62%。
- 未建模总成本：推理 + 集成 + 治理 + 人工复核。

#### 优化实践
- **分层模型路由**：
  - 小/快/便宜模型：routine（占 70–90% token）。
  - 中型模型：复杂推理、工具规划。
  - 顶级模型：高风险决策、情感分析、长上下文合成、合规审查。
  - 开源/本地：敏感数据。
- 缓存 + 提示压缩 + 批处理。
- Per-session / daily token 上限 + 循环断路器。
- 测 cost per successful task，而非 per token。
- 多路由可降 40–80% token 成本。

#### 为什么去年 all-in 最强模型，今年转向路由
- 去年模型能力不足，必须用最强模型兜底。
- 今年主流模型能力普遍够用，经济性和专业性成为主导。
- 同时规避单一供应商 downtime 和政策风险。

### 2.9 可扩展性、集成、可观测性

#### 可扩展性
- 从试点到生产，AI 24/7 连续查询打破传统 observability 的峰谷假设。
- 86% 企业需升级基础设施。
- 实践：Serverless/弹性伸缩、云原生容器、负载均衡、渐进 rollout、资源隔离。

#### 集成
- 42% 企业依赖 8+ 数据来源。
- 工具非幂等导致状态腐败。
- 57% 企业遇 hallucination/confidently wrong，根因常是数据碎片/不洁。
- 实践：数据治理优先、middleware、工具 schema 验证、幂等设计、RAG + 结构化工具调用。

#### 可观测性
- 传统监控不适合 AI：非确定性、长循环、多步推理、黑箱决策。
- 实践：
  - OpenTelemetry 端到端 trace，loop 级 span（goal / tool / decision）。
  - LLM-as-Judge。
  - 行为信号：confidence、tool 异常、输出漂移。
  - 外部遥测，而非仅 Agent 自报。
  - 从 Day 1 设计，不是后期补丁。

### 2.10 可靠性、合规与伦理

#### 错误复合
- 7 步流程，单步错误率 10% → 整体成功率 <50%。
- 重复运行成功率大幅下降。

#### 合规
- EU AI Act 高风险系统要求人类监督、审计轨迹。
- 影子 AI、无问责是普遍问题。

#### 实践
- Checkpoint + 重试；结构化输出 + guardrails。
- Human-in/on/out-of-loop 分层。
- 合成数据测试 + 红队。
- Agent registry；政策编码进系统，非文档。
- 狭窄范围 + 明确 P&L 指标启动。

#### 行业报告
- MIT：95% 试点无显著 P&L 影响。
- Gartner：预测 2027 年前 40%+ agentic 项目被取消。
- 70–95% 试点无显著 P&L 影响。

### 2.11 模型路由与“现实世界设计”趋势

#### 2026 年的主流共识
- 没有单一模型垄断企业级 Agent。
- 多供应商、多模型路由是标配。
- 模型是“发动机”，胜负在“底盘 + 变速箱 + 安全系统”（编排、工具、知识、安全、可观测）。

#### 真实案例
- **Salesforce Atlas Reasoning Engine**：根据任务复杂度、成本、延迟动态选模型。
- **Siemens / AWS**：Amazon Connect + Bedrock 路由多个基础模型。
- **JPMorgan 等金融企业**：混合 OpenAI + Anthropic + 自有/开源，顶级模型只用于最终合成或高置信场景。

#### 趋势转变
- 去年（2025 及以前）：重点在让模型更聪明（prompt engineering、long context、reasoning）。
- 今年（2026）：模型基础能力够用，瓶颈在连接现实世界。
  - 工具调用 / function calling 成熟。
  - 动作执行（API、数据库写、workflow 触发）。
  - 持久状态与编排。
  - 治理、可观测、human-in-loop。

#### 架构启示
- 把“模型”当作可替换的推理引擎。
- 核心是可复用的 orchestration + memory + tools + governance 层。
- Token 变便宜后，会增加 verification Agent、simulation step、更激进的多 Agent，但胜负仍在集成深度和可靠性。

## 三、国内真实开发经验深度复盘

### 3.1 京东言犀：电商客服的渐进演进

#### 业务背景
- 京东自营 + 第三方商家，SKU 海量，售后流程复杂（退换货、价保、发票、优惠券叠加）。
- 大促期间流量洪峰极高，对延迟和可用性要求苛刻。

#### 演进路径
1. **规则 + CRF 时代**
   - 早期基于正则和 CRF 做意图识别。
   - 问题：当对话轮次超过 7 轮，CRF micro-F1 从 0.91 跌至 0.74，意图漂移率升至 18.6%。
2. **BERT+BiLSTM+CRF**
   - 在 120 万条人工标注对话上训练。
   - 50 万条测试集：micro-F1 0.903，意图漂移率 6.2%，误触发率 2.1%。
   - 引入动态窗口机制处理平均 312 token 的长对话。
3. **大模型时代**
   - 2024 双 11 咨询量 36 亿，同比增长 157%。
   - 名创优品案例：在线客服机器人准确率 >97%，独立接待率 >70%，降低 40% 服务成本。

#### 为什么这样设计
- 电商客服的核心是**意图准确 + 状态跟踪 + 工具调用**。
- 早期用轻量模型控制成本；复杂/long-tail 逐步交给大模型。
- 增量学习机制：每 2 小时收集置信度 <0.6 的样本，人工复核后触发训练。

#### 真实效果
- 大促真实负载下的 36 亿咨询量是生产指标，非演示。
- 但高准确率建立在：海量标注数据、领域微调、严格的状态管理、与订单/库存系统深度集成。

### 3.2 阿里小蜜 / 店小蜜：平台级客服的分层架构

#### 阿里小蜜（平台客服）
- 2016 双 11：单日接待 >632 万消费者，对话轮次 >1800 万，承载服务请求占比 95%。
- 2019 双 11：承接淘宝天猫 97% 在线服务需求，等效 8.5 万名人工客服，全天 3 亿在线咨询，解决率 70%。

#### 店小蜜（商家客服）
- 为商家节省近一半服务人力。
- 支持“意图分流”：基于买家实时意图（如“我要退款”）转接到售后组，而非按页面入口固定分流。

#### 架构差异：平台 vs 商家
- **平台客服**： knowledge 集中，与交易/物流/支付系统深度集成，适合统一大模型 + 工具调用。
- **商家客服**： knowledge 分散（各店铺商品、活动、话术不同），需要多租户隔离、商家自定义知识库、模板化配置。

#### 为什么店小蜜强调意图分流
- 传统按页面入口或订单状态分流是“规则前置”，客户真实意图可能与入口不一致。
- 基于实时意图分流减少转接等待，提高一次解决率。

### 3.3 携程 TripGenie：多语言旅游客服的集成挑战

#### 业务背景
- Trip.com 面向 39 个国家，24 种语言，35 种货币。
- 旅游场景信息准确性要求高，涉及航班、酒店、景点、签证、当地政策。

#### 与微软 Azure OpenAI 合作
- 用 GPT-4 Turbo / GPT-4o / GPT-4 Turbo with Vision。
- 机酒核心业务自助率 >70%。

#### 2025 年披露的真实生产指标
- 大模型客服机器人承接 80% 常见咨询。
- 89% 咨询 20 秒内接起，90% 问题 20 分钟内解决。
- IM 对话实时翻译 5 秒内多语种双向互译。

#### 多语言客服的真实复杂性
- 上海珍宝酒店案例：接入携程 AI 多语言客服后，日均外国游客咨询处理时间从半天缩短至 30 分钟。
- 前提是：酒店信息已结构化、多语言知识库已维护、与预订系统打通。

#### 与电商客服的行业差异
- 旅游客服的**知识更新频率高**（航班变动、酒店房态、当地政策），对 RAG 新鲜度要求更高。
- 旅游客服的**动作链更长**：查询 → 预订 → 改签 → 退款 → 通知，跨系统协调复杂。
- 旅游客服的**语言多样性**是核心约束，必须做多语言模型或统一多语言引擎。

### 3.4 金融客服与 Agent：招行、平安、度小满

#### 度小满：语音机器人先行
- 智能语音机器人准确率约 90%，无感率 99%。
- 贷后管理场景：成本 <人工 1%，效率 >人工 5 倍。
- 2019 年上线方言版（重庆话、山东话、河南话、东北话）。
- 2023 年成为文心一言首批生态合作伙伴。

#### 为什么金融从语音机器人切入
- 贷后管理流程标准化强，动作空间明确（确认还款、协商方案、记录承诺）。
- 语音渠道是主要触达方式，ROI 可量化。
- 高合规场景仍需人工复核与录音审计。

#### 招商银行：从智能助手到开源金融大模型
- 2024 年发布国内银行业首个开源百亿参数金融大模型“一招”。
- “小招”智能助手客户满意度 92%，人工替代率 35%。
- 2025 年 AI 应用场景 856 个，全年替代人工 >1556 万小时，等效 >8000 人全职效率。

#### 平安银行：大规模场景落地
- 2024 年末大模型应用场景 >200 个；2025 年末 >390 个。
- AI 代码生成量占比 >30%。
- “AI+T+Offline”服务模式；消保审查智能化覆盖率约 91%。
- 平安集团部署智能体超 5 万个，覆盖 11 万员工。

#### 工商银行：数字员工
- 基于大模型的网点员工智能助手全年处理业务量 3.2 亿笔。
- 部署 13 个综合型数字员工及 1000+ RPA 机器人。

#### 金融行业的特殊约束
1. **合规与可解释性**：监管要求可追溯、可解释，黑箱模型无法直接用于高风险决策。
2. **数据隐私**：客户资产、交易、征信数据必须隔离，很多场景只能用本地/私有化模型。
3. **容错极低**：一条错误利率、错误理赔指引可能引发法律风险。
4. **因此**：金融 Agent 普遍采用“强规则 + 大模型辅助 + 人工终审”的三层架构，而非端到端自主。

### 3.5 国内企业落地失败教训（不是演示，是生产翻车）

#### 共性现象
- 演示惊艳，上线两周后幻觉率飙升（有案例达 30%）。
- 85% 企业落地 AI 工具集中于 Text2SQL 与客服，但多沦为“高级问答机”。
- 75% AI 项目未能跨越试点阶段；近两年 1178 个项目陷入“AI 墓地”。

#### 案例 1：生鲜电商“鲜小达 Pro”
- 目标：替代 60% 初级售后坐席（日均约 2.1 万单的退款/换货/配送问题）。
- 结果：召回率仅 38%，人工转场成本升 17%，用户投诉率升 2.2%。
- 第 35 天叫停。
- 根因：
  - 场景定义缺失，想做“万能 Agent”。
  - Prompt engineering 补丁化，system prompt 超过 2000 token。
  - Multi-agent 分工模糊，内耗严重。
  - 数据闭环断裂，Agent 池静态，无法在线学习。

#### 案例 2：某 SaaS 多 Agent 客服系统
- 正确率从 85% 降至 72%。
- 延迟从 2s 涨至 12s。
- 用户投诉涨 40%。
- Token 成本从 5000 元/天涨至 30000 元/天（翻 6 倍）。
- 根因：简单问题也走多 Agent 路由，协作税过高。

#### 案例 3：多 Agent 生产环境的经典故障
- **共享状态静默覆写**：Agent A 读 v1，Agent B 读 v1，A 写 v2，B 基于 v1 写回，A 的工作被覆盖，不报错。
- **AutoGen GroupChat 无限循环**：两个 Agent 分歧，第三个和稀泥，循环到 max_rounds 截断，输出半成品。
- **CrewAI 版本不一致**：审代码 Agent、安全扫描 Agent、测试 Agent 各自基于不同代码版本工作。

#### 根因归纳
1. 知识库结构混乱、检索机制失效、缺乏仲裁层。
2. 数据未打通：客服处理退换货需在订单/库存/售后三个系统切换。
3. 多 Agent 分工模糊、状态管理缺失、协作税高（CrewAI token 膨胀 3–5x）。
4. 低代码平台强绑定模型与切片算法，丧失迭代能力。
5. 把“演示效果”当成“生产可用性”。

### 3.6 国内 Agent 平台与跨 Agent 协作趋势

#### 平台化
- 阿里云、腾讯云、火山引擎均推出低代码/无代码 Agent 开发平台。
- 典型能力：可视化编排、RAG、工具调用、模型路由、人工兜底。
- 风险：强绑定模型与切片算法，长期可能锁死技术路线。

#### 跨 Agent 协作
- 支付宝“阿宝”通过 AHA（Agent Hub-Access）协议与阶跃、OPPO 等 AI 终端互通。
- 不同 Agent 在用户授权下调用服务技能，而非模拟点击 App。
- 连接出行、本地生活、支付等真实服务。

#### 与国外的差异
- 国内更强调**超级 App 内的服务闭环**（支付宝、微信、抖音）。
- 国外更强调**企业级 SaaS 集成**（Salesforce、ServiceNow、AWS）。
- 国内对**多语言/方言**需求更突出（旅游出海、语音贷后）。
- 国内对**低代码/可视化编排**接受度更高，但锁定风险也更高。

## 四、架构设计启示

### 4.1 不同行业的核心约束差异

| 行业 | 核心约束 | Agent 设计重点 | 混合模式 |
|------|---------|---------------|---------|
| 电商 | 高并发、SKU 海量、售后流程标准化 | 意图识别、状态跟踪、工具调用、大促弹性 | AI 处理 routine，人工处理纠纷/情感 |
| 旅游 | 多语言、知识高时效、动作链长 | RAG 新鲜度、多语言模型、跨系统编排 | AI 处理查询/改签，人工处理突发事件 |
| 金融 | 合规、可解释、隐私、容错极低 | 强规则、权限隔离、审计、人工终审 | AI 辅助，高风险必须 human approval |
| 制造业/IT 服务 | 工单系统、SLA、知识库分散 | 与 ITSM/ERP 集成、结构化输出、escalation | AI 分类/推荐，人工执行关键变更 |

### 4.2 编排模式选择决策树

```
任务是否可分解为独立子任务？
  ├─ 否 → 单 Agent + 工具
  │
  └─ 是 → 子任务之间是否需要强顺序？
           ├─ 是 → Pipeline / 串联
           │        └─ 是否步骤多、容错要求高？→ 加 checkpoint + retry
           │
           └─ 否 → 子任务是否可并行且结果可独立验证？
                    ├─ 是 → Parallel + 聚合/投票 Agent
                    │        └─ 是否高价值/低容错？→ Debate/Consensus
                    │
                    └─ 否 → Supervisor / 层级
                             └─ 子 Agent > 6–10？→ 考虑分层 Supervisor 或共享内存重构
```

### 4.3 真实效果评估框架

不要只看：
- 解决率
- 成本降低
- 响应时间

必须看：
- 长尾失败率（按问题类型、情绪强度、客户价值分层）
- 人工兜底质量（上下文传递完整性、首次解决率）
- 幻觉/错误率（事实性、逻辑性、知识边界、指令遵循）
- 错误复合率（多步任务整体成功率）
- 成本 per successful task（含人工复核、客户投诉、品牌损失）
- 可回滚性（坏变更多久能恢复）

### 4.4 混合模式设计：不是“AI 失败再找人”，而是“人机分工”

#### 三种 human involvement
- **human-in-the-loop**：每一步或关键步骤需人工审批（金融高风险交易）。
- **human-on-the-loop**：AI 自动执行，人类实时监控、可介入（客服实时监听）。
- **human-out-of-the-loop**：完全自动，事后审计（routine 查询）。

#### Klarna 混合模式的启示
- AI 不是“退而求其次”，而是被明确分工处理已知/重复任务。
- 人类不是“擦屁股”，而是处理情感、判断、创新解决方案。
- 边界由 confidence、情绪、风险、业务规则共同决定，而非单一阈值。

### 4.5 生命周期、幂等性与并发控制

这是多 Agent 系统中最容易被忽视、但最难排查的问题类别。特点是：**不报错——系统显示成功，数据却是错的或丢的。**

#### 三类真实问题

1. **并发写入同一份共享状态**
   - 例：platform_contents 被 6 个并行 Agent 竞态覆盖。
   - 选项：
     - 最终一致：谁后写谁赢（简单但可能丢数据）。
     - 隔离：每个 Agent 写独立分区，最后由聚合 Agent 合并。
     - 乐观锁 / 版本控制：写前检查版本，冲突时重试或升级。
   - 原则：不要假设“每个节点都显示完成”就意味着结果正确。

2. **同一个任务被重复提交**
   - 来源：用户手滑、前端重连重发、上游重试、消息队列 duplicate delivery。
   - 选项：
     - 幂等键（idempotency key）：相同 key 的第二次请求被识别并复用结果。
     - 去重窗口：在 N 秒内相同 payload 只执行一次。
     - 状态机：只有特定状态（pending）才允许开始执行，重复提交返回已有任务。

3. **僵尸“进行中”状态**
   - 例：run 端点没有防重入，执行体崩溃或网络断开后，任务永远卡在 running。
   - 选项：
     - 心跳机制：执行体定期上报，超时未上报则标记为失败/超时。
     - 租约（lease）：任务执行需要持有有时效的锁，失联后锁过期，可被重新调度。
     - 终态兜底：定期扫描长时间 running 的任务，人工或自动推进到失败/重试。

#### 设计原则
- 每个任务必须有唯一 ID、明确状态机、可观测的执行轨迹。
- 共享状态写入必须有版本或锁，不能依赖“每个 Agent 都友好”。
- 超时、重试、幂等、终态兜底是同一套生命周期管理，不能分开设计。

## 五、用于“全面思考 Skill”的核查清单

设计 Agent 系统前，逐条确认：

### 问题边界
- [ ] 这个 workflow 的范围是否足够狭窄？
- [ ] 哪些问题是已知/重复的，哪些是未知/新颖的？
- [ ] 未知问题出现时，系统会如何表现（猜测 / 澄清 / 升级）？

### 数据与知识
- [ ] 知识库是否经过治理？更新频率如何？
- [ ] RAG 召回是否有 confidence 分数？低置信度如何处理？
- [ ] 敏感知识是否有权限隔离？

### 架构选择
- [ ] 为什么不用单 Agent + 工具？多 Agent 的真实收益是什么？
- [ ] 多 Agent 之间的状态如何共享？是消息传递还是共享内存？
- [ ] 是否有 timeout、circuit breaker、最大迭代？
- [ ] 降级策略是什么（并发失败 → 串联 fallback；Supervisor 负载高 → peer handoff）？

### 生命周期与幂等性
- [ ] 并发写入同一份共享状态时，是最终一致（谁后写谁赢）还是有隔离/聚合机制？
- [ ] 同一个任务被重复提交（用户手滑、前端重连重发、上游重试）时，系统如何识别并拒绝/复用？
- [ ] 一个任务卡在“进行中”但执行体已经失联（进程崩溃/网络断开），是否有兜底机制把它标记为终态？
- [ ] 共享状态写入是否有版本或锁，而不是假设每个 Agent 都友好？

### 模型与成本
- [ ] 是否使用了模型路由？不同任务用什么模型？
- [ ] 成本如何按 successful task 计算？
- [ ] 是否设置了 token/session/每日预算？
- [ ] 关键路径上是否有为了“锦上添花”功能而阻塞主流程的 LLM 调用？

### 安全与治理
- [ ] 每个 Agent 是否有唯一身份和权限边界？
- [ ] 高风险动作是否需要 human approval？
- [ ] 每类动作的可逆性如何？不可逆动作是否有比可逆动作更高的确认门槛？
- [ ] 外部/不可信来源的内容进入 LLM 前，是否明确用分隔符/框架语言标记为“数据”而非“指令”？
- [ ] 系统是否假设了“检索到的内容是善意的”？如果假设错了，最坏情况是什么？
- [ ] 是否有不可变审计日志？
- [ ] 是否做过 red teaming？

### 可观测性
- [ ] 是否能追踪每个 loop 的 goal / tool / decision？
- [ ] 是否有 LLM-as-Judge 或独立验证 Agent？
- [ ] prompt/策略版本是否可追溯？出问题时能否确认当时用的是哪个版本的 prompt？
- [ ] 是否有实时异常检测与自动回滚？

### 人机协同
- [ ] 升级到人工时，上下文是否完整传递？
- [ ] 人类处理后的反馈是否回流优化 AI？
- [ ] 客户是否能明确知道当前是 AI 还是人工？

### 效果验证
- [ ] 是否有一份固定的回归评测集（覆盖已知场景），每次改动前后跑一遍对比？
- [ ] 是否有按问题类型、情绪、客户价值分层的成功率？
- [ ] 是否区分了演示效果与生产效果？
- [ ] 是否监控了错误复合率和长尾失败率？
