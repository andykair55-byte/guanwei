# 工作间改造方案

> 核心命题：不是帮用户写文章，而是帮用户完成从"碎片化信息"到"结构化观点"的全过程。

---

## 一、重新定义工作间

### 先想清楚：AI 在观微里的角色到底是什么

市面上的 AI 产品走的路大多是**"省时间"**——拍照识别卡路里、AI 写周报、一键生成 PPT。本质都是"替用户工作"，用户是被动的、被服务的。用完就走，没有情感连接。

观微的逻辑不是这个。

用户来观微不是为了"省时间"，而是为了**参与**。辩论输了赌气用工具查资料——这不是"省时间"，这是"我咽不下这口气"。自发式的，有情感驱动的。

如果工作间做成了"AI 帮你写文章"，那它就是一个任务。用户会想：我上了一天班，还要来你这写作业？

**所以工作间的设计基调应该是：**

| 不是 | 而是 |
|------|------|
| 省时间（AI 替你做） | 让想做的事变得可行（AI 给你工具） |
| 完成任务（我要写一篇文章） | 情绪驱动（我不服 / 我想搞清楚 / 这个太有意思了） |
| 用完就走 | 自发地反复回来 |
| 降低门槛只是"变快" | 降低门槛是为了让用户能做他本来做不了的事 |

> **核心原则：用户进来不是因为"需要写点什么"，而是因为"我觉得这个不对劲，想弄清楚"。工作间是帮他把"弄清楚"这件事变得可行的工具。**

### 现在是什么

一个"AI 骨架生成 + 快速发布"的单页编辑器。用户进来 → 输入主题 → 生成骨架 → 写 → 发布。和市面上的 AI 写作工具没有本质区别。

### 应该是什么

一个**信息组织工作台**。用户在这里完成的不只是写作，而是：

```
碎片化信息（看到的/听到的/想到的）
  → 收集/归拢
    → 识别关联/发现矛盾
      → 形成结构
        → 输出观点
```

AI 的角色不是替用户写，而是帮用户整理、帮用户发现关联、帮用户核查——**用户始终是思考的主体**。

### 和目标用户（大学生）的契合

大学生最典型的信息场景：刷到一条热搜/看到一个帖子/听到一个观点 → 觉得有点道理但也说不上来哪里不对 → 想分析清楚但不知道怎么下手。

这不是"不会写"的问题。手上已经有碎片了——几张截图、几个链接、几段引文、脑子里几个模糊的想法。缺的是把这一堆东西理清楚的工具。

工作间就是做这件事的。

---

## 二、完整工作流

不再是一条线走到底，而是四个阶段来回穿插：

```
触发 → [收集箱] ↔ [写作板] ↔ [发布] → 反馈
                ↓            ↑
             [核查台]   (引用/关联)
```

用户可以在任何阶段停下来改天继续（草稿自动保存），也可以跳过不需要的阶段。

### Stage 0：触发——不是任务入口，是情绪入口

用户打开工作间的动机不是"我要写一篇文章"，而是：

| 情绪 | 场景 | 进入工作间后 |
|------|------|-------------|
| **不服** | 辩论输了 / 看到一个观点觉得不对但说不出哪里不对 | 查资料、找论据、准备反击 |
| **好奇** | 看到一个瓜，想知道更多 | 收集各方信息，理清楚来龙去脉 |
| **分享欲** | 发现一个很有意思的角度，想让更多人看到 | 整理成一篇有信息密度的帖子 |
| **求证欲** | 看到一条消息，怀疑是假的 / 想确认是不是真的 | 收集证据、交叉验证 |
| **记录** | 看到一段有用的信息，怕忘了 | 存到收集箱，以后再说 |

**入口设计**：不再只有"写佐证"这种功能性按钮，而是根据场景触发不同情绪文案：

| 入口 | 当前文案 | 新文案 |
|------|---------|--------|
| 瓜田详情页 | "写佐证" | "这个瓜有蹊跷？来分析一下" / "我也想说两句" |
| 辩论房 | 无（跳转到工作间） | "没发挥好？去查资料" / "孔子说得不对？去找证据" |
| 社区帖子 | "写分析" | "这个角度有意思，我也来写写" |
| 首页 "+" | 无（新建） | "今天又刷到了什么？" |

### Stage 1：收集箱（Collect）

用户面对一个话题时，信息的来源不只有一种。收集箱的碎片可以来自多个入口，不是只有"刷到了存起来"。

#### 碎片的四种来源

| 来源 | 含义 | 例子 |
|------|------|------|
| **手动收藏** | 用户主动保存的 | 刷到一篇好文章→存链接；看到一条评论→截图；想到一个观点→随手记 |
| **上下文关联** | 从其他模块启动时自动带入的 | 从瓜田"来分析一下"→自动携带瓜标题+描述+已有佐证；辩论输了"去查资料"→自动携带辩论主题+双方论点 |
| **AI 主动采集** | 信息全景 agent 自动搜索的 | 用户输入主题后，AI 扫描全网相关文章、讨论、报道（详见 Stage 1.5） |
| **外部导入** | 从观微之外带入的 | 浏览器插件剪藏、系统分享、粘贴链接自动解析 |

每条碎片进入收集箱时，携带来源类型标记：

```typescript
interface Fragment {
  id: string
  type: 'link' | 'quote' | 'image' | 'note' | 'voice'
  content: string
  source?: string            // 来源 URL / 出处
  sourceTitle?: string       // 来源标题
  author?: string            // 原始作者
  entryPoint: 'manual' | 'context' | 'ai_scan' | 'external'
  // ^ 标记这条碎片是怎么进来的
  contextOrigin?: string     // 如果是从其他模块带来的，记录来源
  // 如：'melon:xxx' / 'debate:xxx' / 'community:xxx'
  credibility?: number
  tags: string[]
  collectedAt: number
  usedInDraft?: string
}
```

#### 手动收藏

用户主动保存的碎片是最基础、最常见的入口：

| 收集方式 | 操作 | 自动处理 |
|---------|------|---------|
| 粘贴链接 | Ctrl+V 一个 URL | 自动提取标题+摘要+关键段落 |
| 选中文本→发送 | 选中→右键→"发送到工作间" | 自动附带来源 URL 和选中时间 |
| 粘贴截图 | Ctrl+V 图片 | 存入图片，OCR 文字提取（v2） |
| 随手记 | 直接打字 | 纯文本 |
| 语音输入 | 语音转文字 | 文字存入（v2） |

#### 上下文关联（被低估的重要入口）

这是工作间区别于普通编辑器的关键之一。用户不是在真空中创作的——他一定是从某个上下文过来的。这个上下文本身就是最好的初始收集素材。

| 从哪来 | 自动带入的碎片 | 用户打开工作间时已经有什么 |
|--------|--------------|--------------------------|
| 瓜田详情页"来分析一下" | 瓜标题、瓜描述、现有佐证（3-5条） | 不需要重新输入主题，直接开始 |
| 辩论房"没发挥好？去查资料" | 辩论主题、双方论点、裁判点评 | 带着"反驳目标"进入，不是空白页 |
| 社区帖子"这个角度有意思" | 原帖标题、正文、热门评论 | 引用原文，不需要切屏来回看 |
| 求助帖"我来帮你看看" | 求助问题、已有回答 | 带着"解决什么问题"进入 |

**上下文关联碎片在收集箱中用特殊的标签标记**，比如显示"[来自瓜田：XX公司裁员]"。

#### 收集箱的交互

- **列表视图**：按时间倒序，每条显示类型图标+摘要+来源
- **筛选**：按类型/按标签/按来源
- **搜索**：全文搜索
- **选中后操作**：标记来源、加标签、删除、发送到写作板

#### 手机端收集

用户最常做的是"刷到一条 → 存一下"。手机端需要一个极低成本的收集入口：

- 方案 A：系统分享菜单 → "分享到观微" → 自动进入收集箱
- 方案 B：小程序/浮窗，一键保存

v1 先做粘贴收集，系统分享留到 v2。

### Stage 1.5：信息全景（Information Landscape）——竞品分析 Agent

用户在收集了几条碎片之后，最需要的一件事是：**"现在关于这个话题，大家都在说什么？"**

竞品分析 agent 做的事不是商业分析的"竞品"，而是**信息全景扫描**——自动搜索和分析"关于这个话题，已经有哪些声音了"。

#### 它能回答三个问题

| 问题 | 价值 |
|------|------|
| **这个事谁在说？** | 识别信息源：官方/媒体/KOL/社交平台，各自什么立场 |
| **他们都说了什么？** | 提取各方核心论点，归类不同角度 |
| **什么还没人说？** | 识别信息缺口——这就是用户能贡献的独特价值 |

#### 在工作间的触发时机

不是用户一进工作间就自动扫描——那样太吵了。而是：

1. **用户收集了 3 条以上碎片后** → 右侧面板底部出现一条小提示："已收集 X 条信息，要看看到目前大家都在怎么说这个话题吗？"
2. **用户主动点击"信息全景"按钮** → 放在右侧面板顶部，像"📊 全景"图标
3. **用户用碎片生成了骨架之后** → 自动检查：骨架里的要点是否"已有大量覆盖"，标记"这个角度已经有 5 篇分析了"或"这个角度目前还没有人提"

#### 扫描结果长什么样

```
┌────────────────────────────────────────┐
│  📊 信息全景                            │
│                                         │
│  关于"XX 裁员"这个话题：                 │
│                                         │
│  🔵 官方/企业方（2 条）                  │
│    ├─ 公司声明：结构调整，涉及 5% 员工    │
│    └─ 内部信：补偿方案 N+3               │
│                                         │
│  🟢 媒体报道（4 条）                     │
│    ├─ 36氪：行业寒冬还是个案？            │
│    ├─ 虎嗅：裁员的真实原因                │
│    └─ 晚点：内部员工采访                  │
│                                         │
│  🟡 社交平台讨论（6 条）                  │
│    ├─ 脉脉：匿名员工爆料                  │
│    ├─ 知乎：如何看待 XX 裁员              │
│    └─ 微博：热搜讨论                      │
│                                         │
│  ⚪ 信息缺口                             │
│    └─ 没有对比同行业其他公司的裁员情况      │
│    └─ 没有对未来趋势的分析                 │
│                                         │
│  [刷新] [引用到写作板]                    │
└────────────────────────────────────────┘
```

#### 这跟用户收集的碎片是什么关系

用户手动收集的碎片是"用户自己看到的"。

信息全景是"AI 帮用户看到的"——用户可能根本没刷到某些角度，全景帮用户补上视野盲区。

两者在写作板的右侧面板并列展示：

```
右侧面板：
┌───────────┐
│ 📦 我的收集 │  ← 用户手动收集的（精准但可能片面）
├───────────┤
│ 📊 信息全景 │  ← AI 自动扫描的（全面但需要筛选）
├───────────┤
│ 🤖 AI 建议  │  ← 基于以上两者的智能建议
└───────────┘
```

#### 这一条链就是产品的核心壁垒

大多数 AI 产品只能做到"输入→输出"。
观微工作间能做到的链条是：

```
用户收集（主观视角）
  + 全景扫描（客观信息环境）
  = 用户看清全局 → 找到自己能贡献的角度 → 产出高质量内容
```

这不是"写得更快"，而是**"写得比别人更清楚自己在写什么"**——这正是信息素养的体现，也是其他平台做不到的。

### Stage 2：写作板（Organize + Write）

写作板是工作间的核心工作区。

它不再是"一个输入框+一个文章标题"，而是一个**左右分栏的工作台**：

```
┌──────────────────────────────────────────────────────┐
│  左侧：写作区            │  右侧：参考面板            │
│                          │                          │
│  ┌──────────────────┐    │  📦 收集箱               │
│  │ 标题              │    │  ┌──────────────────┐   │
│  │                   │    │  │ □ 链接1           │   │
│  │ 第一节            │    │  │ □ 引文2           │   │
│  │  ├─ 要点1         │    │  │ □ 随手记3         │   │
│  │  ├─ 要点2         │    │  └──────────────────┘   │
│  │  └─ 自定义内容    │    │                          │
│  │                   │    │  🤖 AI 建议             │
│  │ 第二节            │    │  ┌──────────────────┐   │
│  │  ...              │    │  │ 这些碎片似乎都在  │   │
│  │                   │    │  │ 讨论X的同一方面   │   │
│  │                   │    │  │ [分组合并]        │   │
│  └──────────────────┘    │  └──────────────────┘   │
│                          │                          │
└──────────────────────────────────────────────────────┘
```

#### 工作方式不再是"写→改"，而是"收集→拖→组→写"

具体操作流：

```
1. 从右侧收集箱选中碎片 →
2. 拖到左侧写作区的某个段落 →
3. 碎片以引用块的形式嵌入 →
4. 用户在引用块周围写自己的分析 →
5. 写完后在右侧标记"已引用"
```

#### AI 能力嵌入点

写作板上的 AI 不是"生成文字"的按钮，而是**针对用户当前阶段提供不同帮助**：

| 用户当前在做什么 | AI 提供什么 |
|----------------|------------|
| 收集了几条碎片，还没开始写 | "这些碎片都指向同一个结论" / "这几条信息之间有矛盾" |
| 正在写某个段落 | "你写到这里，以下是相关的收集项" |
| 写了段落，标注了某条未经验证的声明 | "这条声称 X，以下是网络上的其他来源" |
| 觉得写得不够有力 | "以下是一些你可以补充的角度" |

所有 AI 建议都是**建议**——用户可以选择接受、忽略、修改。

#### 骨架生成2.0

当前的骨架生成是"输入主题 → AI 出提纲"。新版本应该是：

**基于已经收集的碎片，AI 帮助用户识别结构和关联。**

用户打开写作板时，可能已经收集了 5-10 条碎片。这时候 AI 可以做的不是凭空生成提纲，而是：

1. 分析这些碎片都在讨论什么
2. 识别碎片之间的关联（相同的方面/矛盾的观点/互补的信息）
3. 建议一个结构框架

这样生成的骨架是扎根在实际材料上的，不是 AI 的自由发挥。

> **当前版本**：主题→AI 生成骨架
> **新版本**：碎片集合→AI 识别关联→建议结构→用户调整→形成骨架

### Stage 3：核查台（Verify）

核心信息写完之后，用户需要确认自己写的东西是可靠的。

核查台是一个**侧边面板**，不是单独页面：

- 自动扫描未经验证的声明（基于 `needsVerification` 标记）
- 每个标记显示一个状态下：「待核查 / 核查中 / 已通过 / 发现矛盾」
- 点击"核查" → 发起 AI 求证（复用现在的 verify 系统）
- 核查通过后自动生成引用标注

核查通过后，在最终输出中，每条已经核实的声明旁边会显示可信度标记，增加社区读者对内容的信任。

### Stage 4：发布 + 反馈（Publish + Impact）

发布不再是"点完就结束"。

发布时：

- 选择发布目标：瓜田佐证 / 社区帖子 / 同时发布
- 如果是佐证，自动关联到瓜，并显示"你选择了 XX 立场"
- 发布前显示最终预览，包含所有核查标记和引用来源

发布后：

- 在个人中心→我的创作 里看到所有已发布内容
- 每篇内容显示：阅读量、点赞数、评论数、**作为佐证时改变了多少人的判断**
- 这个"影响力数据"是正反馈循环的关键——用户能看到自己写的东西真的有价值

## 三、v1 实现范围

"从能用到好用"——v1 只做"能用"，不做"完美"。

### v1 包含

| 模块 | 范围 | 说明 |
|------|------|------|
| 收集箱 | ✅ 基础版 | 粘贴链接/文本收集，列表展示，搜索，删除 |
| 写作板 | ✅ 基础版 | 左右分栏（参考面板），引用块嵌入，分段编辑 |
| 骨架生成2.0 | ✅ 基于碎片 | 不再是纯 prompt 生成，而是基于已收集材料 |
| AI 建议 | ⚠️ 有限版 | 只做"碎片关联识别"，不做段落级 AI 建议 |
| 核查台 | ✅ 基础版 | 侧边面板，一键核查，结果标记 |
| 反馈数据 | ✅ 基础版 | 阅读量 + 点赞 + 佐证影响人数 |
| 多入口触发 | ✅ 必须 | 瓜田/社区/新建三个入口 |
| 手机端收集 | ❌ v2 | 仅保留粘贴收集 |
| 拖拽交互 | ❌ v2 | 先做"点击引用"，不做拖拽 |
| 系统分享 | ❌ v2 | 仅保留手动粘贴 |

### v1 不做

- 拖拽排序/归类（用按钮操作替代）
- 语音收集
- 图片 OCR
- 富文本编辑器（用 Markdown 编辑器替代）
- 协作/分享功能
- 版本历史

### 技术影响

| 模块 | 需要新建/改造 |
|------|-------------|
| 收集箱 | 新建 `fragmentStore.ts`（Zustand + persist） |
| 写作板 | 改造 `AgentWorldPage.tsx` → 左右分栏布局 |
| 骨架生成2.0 | 改造 `creationService.ts` → 增加基于碎片的生成路径 |
| 引用面板 | 新建组件 `ReferencePanel.tsx` |
| 核查台 | 新建组件 `VerificationSidebar.tsx`，复用现有 `verify` service |
| 反馈数据 | 改造 `userStore.ts` 或新建 `impactStore.ts` |
| 多入口触发 | 修改 `MelonDetailPage.tsx` 和 `CommunityPage.tsx` |
| 编辑器 | 用一个轻量 Markdown 编辑器替代 textarea |

## 四、设计原则：自发式 > 任务式

这个原则决定了工作间里每一个交互的朝向，不是锦上添花的理念。

### 用户怎么进来的，决定了他怎么用

| 进入方式 | 心态 | 用户会 | 留存概率 |
|---------|------|-------|---------|
| "我要写一篇文章" | 任务式，有压力 | 要么写到一半放弃，要么写完了也不想来第二次 | 低 |
| "我不服，我要搞清楚" | 自发式，有情绪驱动 | 主动收集信息，有动力完成，完成了有成就感 | 高 |
| "这个很有意思，我想记一下" | 自发式，无压力 | 存了就走了，但下次还会来存 | 中→高 |

**工作间的交互设计必须保护"自发式"心态，不能把它变成"任务式"。**

### 具体怎么做

#### 1. 永远不要求用户"一次完成"

- 写了一半关掉 → 自动保存，下次打开还在原地
- 收集了 10 条碎片一条都没用上 → 没问题，碎片留在收集箱，以后可能用得上
- 生成了骨架但不想写了 → 骨架保留，下次打开可以继续

**反面案例**：当前工作间只保存了"一个草稿"，新建草稿会覆盖旧的。这意味着用户每次打开工作间都有"要么写、要么丢失"的压力。

#### 2. AI 建议是被动的，不是主动打扰的

- 用户收集了几条碎片，AI 不弹窗说"要不要生成骨架？"
- 而是在用户看不到的地方（右侧面板底部）显示"已识别 X 条碎片之间的关联"
- 用户想用的时候自己点过去

**反面案例**：弹窗"检测到你有 X 条碎片，是否一键生成文章？"——这跟"你的电脑有 X 个问题需要清理"一样烦人。

#### 3. 收集箱本身就有价值，不一定要"产出"

用户存了 50 条碎片，一条都没发布出去 → 这也是一种使用方式。

收集箱本身就是一个"信息剪报"——用户需要的时候来翻翻看，不需要的时候放着就行。不一定非要"用完→产出→清空"。

#### 4. 情绪出口用情绪文案，功能按钮用中性文案

- "写佐证" → "这个瓜不对劲？来分析一下"
- "发布" → "我觉得想清楚了" / "先发出去看看大家的反应"
- "保存" → 保留"已保存"，但加一句"下次接着写"

## 五、用户故事验证

用三个典型用户故事检验设计是否自洽：

### 故事 A：大学生刷到一条热搜

```
1. 在瓜田看到"XX 公司裁员"的瓜
2. 点进去看了佐证，觉得"这个事情没那么简单"
3. 点击"写佐证"
4. 自动带入瓜标题 → 直接进入写作板（跳过收集箱）
5. 从右侧收集箱看到之前收集过的几条相关链接
6. 快速引用链接+写几段分析
7. 一键发布佐证
8. 第二天看到"你的佐证影响了 23 人的判断"
```

### 故事 B：想写一篇深度分析但没有头绪

```
1. 平时刷手机看到相关文章都存到收集箱（十几条链接+随手记）
2. 打开工作间 → 新建 → 从收集箱开始
3. AI 识别碎片关联："这些碎片都提到同一个核心矛盾"
4. 接受 AI 建议 → 自动生成骨架
5. 调整章节结构，删除多余的
6. 每章从收集箱引用对应材料
7. 核查台标出 3 条待核查声明 → 一键核查
8. 发布到社区，帖子自动带上"基于 X 条来源"的标签
9. 帖子被点赞 → 看到"因你的分析，有 5 人改变了看法"
```

### 故事 C：只是随手记一个想法

```
1. 首页 → "+" → 新建工作间
2. 标题写了一半，正文只写了一句话
3. 关掉 → 自动保存到草稿箱
4. 三天后打开 → 看到草稿箱里有 3 个未完成的草稿
5. 继续写其中一个 → 从收集箱补充材料 → 发布
```

三个故事说明了同一个逻辑：**不要求用户一次完成，不要求用户"先想清楚再写"。收集→组织→输出可以跨天跨周，每个阶段都可以暂停。**

## 五、和产品飞轮的关系

```
瓜田（消费） → 看到佐证 → 觉得"我也可以写"
  → 工作间（生产） → 收集碎片 → 组织 → 写 → 核查 → 发布
    → 佐证/社区帖子（沉淀）
      → 其他用户看到 → 被影响 → 点赞/评论
        → 作者看到反馈 → 正反馈 → 继续参与
```

工作间把 "消费→生产" 这个转化成本降到了最低。用户不需要从零开始写，只需要"把我已经有的碎片理清楚"。

这就是产品的核心飞轮闭环：**吃瓜（低门槛） → 看到佐证（接触高质量信息） → 打开工作间（AI 辅助降低生产门槛） → 产出高质量内容 → 获得正反馈 → 继续吃瓜。**

## 六、需要确认的几个设计决策

1. **收集箱位置**：我倾向于放在写作板右侧面板，而不是独立页面。这样用户在写作时可以随时看/引用，不需要来回跳转。同意吗？

2. **骨架生成2.0的触发时机**：应该是在用户有几条碎片之后"建议生成"，还是作为一个单独的"生成骨架"按钮放在写作板顶部？我偏向后者——用户自己控制何时生成，但生成时会基于已收集的碎片而不是纯 prompt。

3. **编辑器选型**：v1 是继续用 textarea 加 markdown 支持，还是换一个轻量级编辑器？我倾向保留 textarea 但加 markdown 预览——换编辑器有学习成本，而且 textarea 配合右侧面板已经够用了。

4. **核查台的"一键核查"是否消耗免费额度**：如果是，收集箱里的碎片数量可能很多，免费额度够用吗？这件事会直接影响用户的使用意愿。
