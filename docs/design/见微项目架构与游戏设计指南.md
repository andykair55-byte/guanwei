# 见微项目架构与游戏设计指南

## 1. 项目概述

见微是一个"理性吃瓜"的信息求证社区，核心使命是**以娱乐无痛的方式激发用户的求证欲**。用户本身具备思考和判断能力，但常常被从众心理、信息噪音、营销号带节奏等因素影响。见微提供一个轻松的环境，让用户在吃瓜娱乐中自然而然地养成求证习惯。

### 核心价值主张

> **不是简单的"猜对游戏"，而是让用户在娱乐中养成求证习惯。**

> **不是"教你思考"，而是"帮你求证"——AI是工具，不是导师。**

- **娱乐属性**：吃瓜热点内容，满足好奇心，降低参与门槛
- **工具属性**：AI辅助资料整理、逻辑收敛、复盘，不替代人类思考
- **社区属性**：用户分享判断依据，形成理性讨论氛围，避免无意义辩论
- **安全属性**：官方证实消息可查验，不碰红线；反诈科普，关怀弱势群体

### 痛点分析

| 痛点 | 说明 | 见微的解决方案 |
|------|------|--------------|
| **信息噪音** | 水军、营销号、不良媒体带节奏 | AI一键求证 + 多源交叉验证 |
| **从众心理** | 评论区观点直接影响主观判断 | 佐证机制让用户自己找证据 |
| **资料整理难** | 整理证据链耗时费力，容易被攻击 | AI搜集Agent自动搜集，可视化展示 |
| **理性用户少** | 讨论氛围差，不愿与拿不出证据的人辩论 | 佐证排行榜，优质内容获得曝光 |
| **依赖AI决策** | 过于依赖AI结论，缺少独立思考 | AI仅提供工具和证据，最终判断由用户做出 |
| **反诈推广难** | 强制推广效果差，下载后删除 | 娱乐化科普视频，猜错惩罚机制 |

### 为什么不用其他求证工具

| 工具 | 局限性 | 见微的差异化 |
|------|--------|-------------|
| 抖音求真 | 依附短视频生态，内容碎片化 | 独立社区，专注求证，深度内容 |
| 腾讯求真 | 企业背书，官方感强，互动性弱 | 娱乐化设计，用户参与感强 |
| 搜索引擎 | 信息分散，需要用户自行筛选 | AI自动搜集整理，证据链完整 |
| 官方媒体 | 权威性强，但互动性差，更新慢 | 用户参与，实时讨论，社区共创 |

### 特殊人群关怀

**青少年群体**：
- 针对青少年的科普视频（常识教育、网络安全、诈骗防范）
- 家长可查看孩子的猜瓜记录，了解孩子关注的内容
- 年龄分级内容过滤

**老年人群体**：
- 大字模式、语音朗读功能
- 针对老年人的科普视频（投资风险、电信诈骗、健康谣言）
- 一键求助功能，家人可收到求助提醒

**反诈合作**：
- 与官方媒体号、反诈APP合作制作案例视频
- 反诈科普视频作为猜错惩罚内容
- 与反诈APP联动，用户举报诈骗信息可获积分奖励

## 2. 系统架构

### 2.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| **前端** | React 19 + TypeScript | 移动端优先，支持 PC 端自适应 |
| **状态管理** | Zustand | 轻量级状态管理 |
| **样式** | TailwindCSS 4 | 原子化 CSS，零配置 |
| **路由** | React Router DOM 7 | SPA 路由 |
| **后端** | FastAPI | 高性能异步框架 |
| **Agent 编排** | LangGraph | 多 Agent 协作系统 |
| **数据库** | SQLite → PostgreSQL | MVP 用 SQLite，验证期切换 |
| **ORM** | SQLAlchemy | Python ORM |
| **LLM** | 多模型支持 | DeepSeek/Groq/Claude/OpenAI 等 |

## 3. 功能模块详解

### 3.1 底部 Tab 导航

| Tab | 路径 | 核心功能 |
|-----|------|---------|
| **瓜田** | `/melon` | 热点吃瓜信息流，大图卡片，分类筛选 |
| **社区** | `/community` | 帖子社区，瀑布流，公益/求助/热帖 |
| **发布** | `/publish` | 发布内容，AI预检，类型选择 |
| **求证** | `/verify` | AI一键求证，Agent协作可视化 |
| **我的** | `/profile` | 个人中心，段位，积分，工具箱 |

### 3.2 瓜田信息流

**核心功能**：
- 大图单列卡片流（抖音风格）
- 分类筛选：娱乐/科技/生活科普/社会热点/历史/财经
- 下拉刷新
- 每个卡片包含：封面图、标题、参与人数、热度标签

**卡片状态**：
- `pending`：待开奖，显示"立即猜"按钮
- `revealed`：已开奖，显示"已开奖"标签和结果

### 3.3 猜瓜详情页

**核心流程**：

```
进入详情 → 查看内容 → 选择真/假 → 提交佐证（可选）→ 等待开奖 → 查看结果
```

**关键交互**：
- 真/假判断按钮（大按钮，显眼）
- 佐证输入框（选填，激发用户寻找证据的欲望）
- 当前参与人数和双方比例
- 开奖后显示：AI实锤报告 + 佐证排行榜 + 段位进度反馈

### 3.4 佐证机制（核心差异化）

**设计理念**：不是猜对就行，而是让用户主动寻找证据。人类本身具备思考能力，需要的是激发求证欲。

**佐证卡片内容**：
- 用户名、头像
- 判断方向（真/假标签）
- 佐证内容
- 点赞/踩数

**奖励机制**：
- 提交佐证：+5 积分
- 佐证被点赞：+2 积分/赞
- 佐证被选为最佳：+50 积分 + 段位经验×1.5

**最佳佐证判定**：综合点赞数、相关性、准确率

### 3.5 AI 一键求证

**核心流程**：

```
输入内容 → AI审核 → 搜集Agent搜索 → 验证Agent交叉验证 → 分析Agent生成报告
```

**可视化展示**：
- Agent进度节点动画（搜集→验证→分析）
- 实时对话流（显示搜索进度）
- 证据时间线（按时间顺序展示证据）
- 最终报告（可信度等级、关键证据、倾向性判断）

**免费额度**：每日免费 3 次，积分可兑换（50积分/次）

### 3.6 社区

**帖子类型**：
- 普通帖子：用户分享的吃瓜/讨论
- 公益贴：带有公益标签，专属标识
- 求助贴：需要求证/帮助的内容
- 热帖：高互动量帖子

**布局**：双列瀑布流

### 3.7 发布

**功能**：
- 类型选择：吃瓜爆料/公益求助/求助求证
- 封面图上传
- 正文输入
- 话题标签
- AI智能审核预检

**交互**：点击底部中间"+"按钮，发布页从底部向上滑出全屏

### 3.8 个人中心

**核心展示**：
- 头像、昵称
- 段位（核心展示）+ 段位进度条
- 积分余额
- 猜瓜统计（正确率/总次数）
- 成就徽章列表

**工具箱入口**：横向滚动工具卡片
- 一键求证（50积分/次）
- 深度调查（200积分/次）
- 谣言粉碎机（免费）

## 4. 游戏设计系统

### 4.1 积分系统

#### 获取规则

| 行为 | 积分 | 每日上限 | 说明 |
|------|------|---------|------|
| 每日登录 | +10 | 1次 | 签到奖励 |
| 猜对瓜 | +20~50 | 200 | 根据难度梯度 |
| 提交佐证 | +5 | 50 | 鼓励用户寻找证据 |
| 佐证被点赞 | +2/赞 | 100 | 正反馈激励 |
| 佐证被选为最佳 | +50 | - | 优质内容奖励 |
| 发布高质量内容 | +20~100 | - | 需审核认定 |
| 邀请好友注册 | +50 | - | 好友首次猜瓜后发放 |

#### 消耗场景

| 用途 | 消耗 | 说明 |
|------|------|------|
| 兑换求证次数 | 50积分/次 | 每日免费3次用完后 |
| 深度调查 | 200积分/次 | 更全面的多Agent分析 |
| 帖子曝光度加成 | 100积分/次 | 提升帖子曝光 |
| 头像框/称号装饰 | 100-1000积分 | 个性化装饰 |
| 公益捐赠 | 任意 | 公益功能 |

#### 核心规则
- **猜错不扣积分**，仅需看1分钟科普视频
- **每日获取上限200**，防止刷分和沉迷
- 积分形成闭环：`猜瓜赚积分 → 佐证获积分 → 积分换工具 → 工具帮猜瓜 → 猜瓜更准`

### 4.2 段位成长体系

#### 7级段位数值

| 段位 | 等级 | 猜对次数 | 准确率 | 参与次数 | 预估耗时 |
|------|------|---------|--------|---------|---------|
| 吃瓜群众 | Lv.0 | 0 | - | 0 | 初始 |
| 瓜田新手 | Lv.1 | 10 | 25% | 30 | 1-2周 |
| 鉴瓜学徒 | Lv.2 | 30 | 35% | 80 | 1个月 |
| 瓜田侦探 | Lv.3 | 80 | 45% | 180 | 3个月 |
| 鉴瓜达人 | Lv.4 | 200 | 50% | 400 | 6个月 |
| 鉴瓜大师 | Lv.5 | 400 | 55% | 800 | 12个月 |
| 见微先知 | Lv.6 | 800 | 60% | 1500 | 1.5年+ |

#### 升级判定逻辑

三个条件必须同时满足：
1. 累计猜对次数 >= 下一级要求
2. 准确率 >= 下一级要求
3. 累计参与次数 >= 下一级要求

#### 加速机制

- **佐证加速**：佐证被选为最佳，获得段位经验×1.5
- **禁止VIP加速**：保持公平，段位只能靠真实能力提升

#### 升级反馈

- 全屏升级动画
- 新段位名称和图标展示
- 段位进度变化提示

### 4.3 成就徽章

| 称号 | 条件 | 展示位置 |
|------|------|---------|
| 鉴瓜大师 | 累计猜对50次 | 头像框、评论区 |
| 火眼金睛 | 连续猜对10次 | 成就栏 |
| 谣言粉碎机 | 累计识别50个假消息 | 成就栏 |
| 最佳佐证 | 佐证被选为最佳5次 | 成就栏 |

### 4.4 猜错处理

- 不扣积分，保护用户体验，避免用户因失败而离开
- 弹出科普小视频（约1分钟），**这不是惩罚，而是学习机会**
- 视频内容与被猜内容主题相关，针对性强
- 观看倒计时结束后可关闭，不强制
- **科普视频分类**：
  - **反诈系列**：与官方媒体号、反诈APP合作制作案例
  - **常识系列**：针对青少年和老年人的基础常识（如网络安全、健康谣言）
  - **投资风险系列**：针对老年人的投资陷阱科普
  - **工作风险系列**：针对年轻人的求职诈骗科普
- **设计理念**：在轻松的环境中学习，比真正被骗后再后悔的"惩罚"要温和得多；无痛学习，无隐私侵扰

## 5. 数据模型

### 5.1 用户模型

```typescript
interface User {
  id: string;
  nickname: string;
  avatar: string;
  rank: Rank;           // 段位
  points: number;       // 积分
  totalGuesses: number; // 总参与次数
  correctGuesses: number; // 正确次数
  badges: Badge[];      // 成就徽章
  createdAt: string;
}
```

### 5.2 瓜模型

```typescript
interface Melon {
  id: string;
  title: string;
  description: string;
  coverImage: string;
  category: MelonCategory;
  difficulty: 1 | 2 | 3;
  trueCount: number;      // 选真人数
  falseCount: number;     // 选假人数
  totalParticipants: number;
  revealTime: string;     // 开奖时间
  status: 'pending' | 'revealed';
  result?: boolean;       // 开奖结果
  report?: Report;        // 实锤报告
  likeCount: number;
  commentCount: number;
  evidenceCount: number;
  isLiked: boolean;
  createdAt: string;
}
```

### 5.3 佐证模型

```typescript
interface Evidence {
  id: string;
  userId: string;
  userNickname: string;
  userAvatar: string;
  melonId: string;
  guessId: string;
  direction: boolean;     // true=真, false=假
  content: string;
  upvotes: number;
  downvotes: number;
  isBest: boolean;       // 是否最佳佐证
  createdAt: string;
}
```

### 5.4 实锤报告模型

```typescript
interface Report {
  id: string;
  melonId: string;
  timeline: TimelineItem[];      // 信息时间线
  evidenceChain: EvidenceItem[]; // 证据链
  keyDoubts: string[];          // 关键疑点
  tendency: string;             // 倾向性判断
  tendencyDirection: boolean;    // 倾向真/假
  disclaimer: string;           // 免责声明
  generatedAt: string;
}
```

## 7. 多模型支持

### 7.1 支持的模型提供商

**国产模型（7家）：**
- DeepSeek
- 智谱 GLM
- 通义千问
- Kimi（月之暗面）
- 豆包（火山引擎）
- MiniMax
- 阶跃星辰 StepFun

**国外模型（4家）：**
- OpenAI
- Anthropic Claude
- Google Gemini
- Groq

**自定义模型：**
- 任意 OpenAI 兼容接口

### 7.2 切换机制

- 默认主模型通过 `LLM_PROVIDER` 环境变量设置
- 支持运行时通过 API 切换
- 主模型调用失败自动降级到备用模型
- 所有模型统一使用 OpenAI 兼容 SDK 调用

| 并发写入 | 单写多读 | 多写多读 | PostgreSQL |
| 数据量 | < 1GB | 无上限 | PostgreSQL |
| 运维成本 | 零 | 低 | SQLite MVP，PG 验证期 |
| 迁移成本 | - | 1小时 | 可接受 |

**迁移方案**：
1. 配置 PostgreSQL Docker 容器
2. 修改 `DATABASE_URL` 环境变量
3. 运行迁移脚本 `backend/scripts/migrate_db.py`
4. 配置连接池（最小 5，最大 20）

### 8.3 缓存策略

## 9. 证据链体系

### 9.1 数据结构

证据链是 AI 求证的核心产出，支持多源证据的可信度评估。

```python
class EvidenceChain:
    id: int                 # 证据 ID
    report_id: int          # 所属报告
    source_url: str         # 来源 URL
    source_type: str        # 来源类型：official/media/social/forum
    credibility_level: int  # 可信度等级（1-5星）
    timestamp: datetime     # 证据时间戳
    content_summary: str    # 内容摘要
    relevance_score: float  # 相关性评分（0-1）
```

### 9.2 可信度基准

| 来源类型 | 可信度基准 | 说明 |
|---------|-----------|------|
| official | 5 | 官方来源，最高可信度 |
| media | 4 | 媒体报道，较高可信度 |
| social | 2 | 社交平台，较低可信度 |
| forum | 1 | 论坛来源，最低可信度 |

### 9.3 综合评分公式

```
综合评分 = 可信度等级 × 相关性评分 × 时间衰减因子

时间衰减因子：
- 7天内：1.0
- 7-30天：0.8
- 30天以上：0.5
```

### 9.4 API 端点

| 端点 | 方法 | 说明 |
|------|------|------|
| `/evidence` | POST | 创建证据链项 |
| `/evidence/report/{id}` | GET | 获取报告的证据链 |
| `/evidence/melon/{id}` | GET | 通过瓜 ID 获取证据链 |
| `/evidence/report/{id}/top` | GET | 获取可信度最高的 N 条证据 |

## 9. 核心设计原则

1. **前期快后期慢**：新手 1-2 周就能升级，顶级按年计，稀缺感=价值感
2. **猜错不惩罚**：保护用户体验，用科普视频替代扣分
3. **佐证引导思考**：不是猜对就行，而是让用户学会如何思考
4. **公平性**：禁止 VIP 加速段位，段位只能靠真实能力提升
5. **正反馈激励**：佐证被点赞/选为最佳获额外奖励，形成学习闭环
6. **国产优先**：默认使用国产模型，7家国产模型平等对待

## 10. 关键技术决策

### 10.1 Mock 数据策略

- 前端 `api.ts` 实现了 `withFallback` 机制
- 先尝试真实后端，失败自动切换到 Mock
- 已确认后端可用后直接走真实请求
- 已确认后端不可用后直接走 Mock
- 首次探测仅在配置了 `VITE_API_BASE_URL` 时发生

### 10.2 跨平台适配

- 移动端：原始布局，底部 Tab 导航
- PC 端：Twitter 风格三栏布局（左侧导航 + 中间内容 + 右侧面板）
- 最大宽度限制：移动端内容区最大宽度 480px

### 10.3 动画策略

- 使用 CSS transition + transform 实现动画
- 避免重 JS 动画，保证 60fps
- 卡片入场：渐入 + 轻微上移（stagger 80ms）
- 图片加载：模糊→清晰渐变（blur-up）
- 弹窗：背景模糊渐入 + 内容缩放弹出
