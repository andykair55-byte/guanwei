# 技术栈调整方案

## 1. 需求概述

根据项目现状和未来规划，需要调整技术栈以满足以下需求：

| 需求 | 目标 |
|------|------|
| **用户规模** | 500-1000 日活用户，同时在线 100-200 |
| **跨平台支持** | Android 移动端 + PC 端 |
| **性能要求** | 页面响应 < 500ms，API 响应 < 200ms |
| **可扩展性** | 支持未来扩展到 10,000+ 日活 |
| **开发效率** | 快速迭代，维护成本低 |

## 2. 当前技术栈分析

### 2.1 前端

| 技术 | 当前状态 | 评估 |
|------|---------|------|
| React 19 | ✅ 使用中 | 成熟稳定，生态完善 |
| TypeScript | ✅ 使用中 | 类型安全，减少 bug |

**评估结论**：前端技术栈无需调整，React + TypeScript + TailwindCSS 是当前最成熟的组合，完全满足跨平台需求。

### 2.2 后端

| 技术 | 当前状态 | 评估 |
|------|---------|------|
| FastAPI | ✅ 使用中 | 高性能，文档自动生成，适合 API 开发 |
| SQLite | ✅ 使用中 | 开发方便，但不适合多用户并发 |
| SQLAlchemy | ✅ 使用中 | 成熟 ORM，迁移成本低 |
| LangGraph | ✅ 使用中 | 多 Agent 编排标准方案 |

**评估结论**：后端框架无需调整，但数据库需要从 SQLite 切换到 PostgreSQL。

## 3. 技术栈调整方案

### 3.1 后端技术栈（推荐方案）

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|---------|
| **框架** | FastAPI | 0.115+ | 高性能异步，自动文档，Python 生态丰富 |
| **语言** | Python | 3.11 | LTS 版本，性能稳定，AI/ML 支持好 |
| **数据库** | PostgreSQL | 15+ | 成熟稳定，支持并发，适合 10,000+ 用户 |
| **ORM** | SQLAlchemy | 2.0+ | 成熟 ORM，无缝迁移，支持异步 |
| **缓存** | Redis | 7+ | 热点数据缓存，会话管理，减少 DB 压力 |
| **认证** | JWT | PyJWT | 无状态认证，适合分布式扩展 |
| **Agent 编排** | LangGraph | 0.2+ | 微软开源，多 Agent 协作标准方案 |
| **LLM SDK** | openai | 1.0+ | 统一 API，支持多模型提供商 |
| **部署** | Uvicorn + Gunicorn | 最新 | 生产级 ASGI 服务器 |
| **容器** | Docker | 24+ | 环境一致性，方便部署 |

### 3.2 移动端方案（Android）

#### 方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **React Native** | 代码复用，开发效率高 | 性能稍差，依赖原生模块 | ⭐⭐⭐⭐ |
| **Flutter** | 性能好，UI 一致性高 | 学习曲线，生态不如 RN | ⭐⭐⭐ |
| **原生开发** | 性能最好，功能完整 | 开发成本高，维护困难 | ⭐⭐ |
| **WebView** | 零开发成本 | 体验差，功能受限 | ⭐ |

#### 推荐方案：React Native

**理由**：
1. 代码复用率高（与 Web 端共享 60-80% 逻辑）
2. 开发效率高，一套代码覆盖 Android + iOS
3. 生态成熟，社区活跃
4. 支持热更新，快速迭代

**技术栈**：
- React Native 0.75+
- TypeScript
- TailwindCSS React Native
- React Navigation
- Zustand（状态管理共享）

#### 替代方案：Flutter（备选）

如果团队有 Flutter 经验或对性能有更高要求，可以选择 Flutter。但考虑到当前 Web 端已经是 React 技术栈，React Native 是更平滑的迁移方案。

### 3.3 PC 端方案

当前 Web 端已经支持 PC 端自适应（三栏布局），无需额外开发。

**关键优化**：
- 响应式设计：移动端单列，PC 端三栏
- 桌面端功能增强：快捷键支持、拖拽操作、多标签页
- 性能优化：代码分割、懒加载、缓存策略

## 6. 关键技术决策

### 6.1 认证方案

**推荐方案**：JWT 无状态认证

```
用户登录 → 返回 JWT Token → 后续请求携带 Token → 服务端验证 Token → 返回数据
```

**优点**：
- 无状态，支持水平扩展
- 无需 Session 存储
- 移动端友好

**安全措施**：
- Token 过期时间：2 小时
- 刷新 Token：7 天
- HTTPS 传输
- 强密钥签名

### 6.2 缓存策略

**推荐方案**：Redis 缓存

| 缓存类型 | Key 设计 | TTL | 说明 |
|----------|---------|-----|------|
| 瓜列表 | `melons:{category}:{status}` | 5分钟 | 热点数据缓存 |
| 用户信息 | `user:{id}` | 30分钟 | 用户资料缓存 |
| 用户统计 | `user_stats:{id}` | 10分钟 | 积分/段位缓存 |
| 实锤报告 | `report:{melon_id}` | 永久 | AI报告缓存 |

### 6.3 LLM 调用策略

**推荐方案**：多模型负载均衡 + 自动降级

```python
# 主模型 → 备用模型 → 降级模型
# DeepSeek → Groq → Claude
```

**成本控制**：
- 默认使用成本较低的模型（如 DeepSeek）
- 复杂任务使用更强大的模型
- 设置调用频率限制
- 缓存重复请求结果
