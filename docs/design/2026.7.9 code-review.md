---
  二、需求侧清晰度

  目标用户明确度：🟡 中

  从策划案来看，产品的目标用户是"喜欢刷内容、辨真假的网民"，但缺少更精细的分层：
  - 到底是面向"闲下来刷一刷的泛娱乐用户"，还是"关心信息真伪的理性用户"？这两类人的使用路径完全不同——前者是 TikTok 式沉浸刷，后者是带着问题来搜证
  - 策划案里的"AI 创作助手"降的是创作门槛，但目标用户是否同时是创作者？如果"普通用户只消费不创作"，那"Ai 创作助手 P0"的优先级判断可能需要重新审视

  刚需程度评估：🟡 部分可信

  "猜真假 + 下注"这个核心玩法在海外有原型（Metaculus / Manifold），国内竞品不直接。但存在一个结构性问题：
  - 流量入口需要"有持续供给的优质内容"——这又依赖 AI 创作助手
  - AI 创作助手需要 LLM 稳定、成本可控——这在当前 Groq 免费额度用完后的场景还没验证
  - 结论：核心闭环在每个环节都有单点依赖，任何一个瓶颈都会让产品"转不动"

  隐藏复杂度

  1. LLM 调用层有两套，互相不知道

  server/src/lib/llm.ts  → Groq SDK（TS）
  backend/services/llm.py → ?? （Python）

  这两套是什么关系？如果 Python 后端也调 LLM，那两套之间是"任选其一"还是"互相备份"？当前前端有 4 个 service 硬编码了 Groq 调用（aiJudgeService / aiSummaryService / aiTranslatorService /  refereeService），而策划案里又说要"重构 4 个 hardcode Groq 的 service → callLLM()"——说明 LLM 层本身就在重构中，架构还没稳定。

  2. Engine/Mirror 的角色不明确

  engine/ 下有一个 5485 行的 Python 引擎，目录结构完整（consciousness / cortex / abilities / world / scenarios），像是一个独立的 AI agent 框架。但它跟 backend/ 是什么关系？是用于"AI
  斗蛐蛐"的对战引擎，还是一个同学单独实验的项目放在同一个仓库里？如果它跟产品直接相关但没有被主后端的 pipeline/agents 引用，那就是一条独立的研发线，有重复造轮子的风险。

  3. mock 依赖的切换时机不清晰

  前端有完整的 mock 层（mockApi.ts 14KB + mockData.ts 28KB），VITE_API_BASE_URL 配置后自动切后端。但后端 API（admin_routes / melon_routes / user_routes / routes）只有 4
  个文件，每个端点的覆盖率未知。从 mock 切换到真实后端的"逐模块对接"在计划中是 Phase 3，但 Phase 1 前端界面做完后，如果后端还没就位，UI 改得再多也还是 mock 数据。

  MVP 建议

  现有策划案的 MVP 范围合理（瓜田 + AI 创作助手 + AI 竞技场 + 用户体系），但有两点值得注意：

  - AI 创作助手定义成 P0 但本身是"难"复杂度——如果 LLM 层重构在 Phase 2 才做，Phase 1 做的"AI 创作助手页面"在演示时效果可能只能展示 UI 布局，演示价值打折
  - 辩论系统 P1 但真人辩论的并发控制是隐藏复杂度——"只做 AI 斗蛐蛐"是对的，真人辩论模式应该明确推迟

  七、优先修复清单

  按风险从高到低：

  🔴 1. 确定 LLM 层的架构选型

  现在有两套 LLM 调用层（server/src/lib/llm.ts + backend/services/llm.py）和 4 个硬编码 Groq 的前端 service。在继续开发之前，必须先定清楚：
  - 哪个是"唯一的 LLM 入口"？建议统一到 Python 后端，前端只通过 API 调
  - 重构 4 个 hardcode Groq service 的计划要排上日程，不然每次改模型都要改 4 个文件

  🔴 2. 明确 Engine/Mirror 和后端的关系

  如果 Mirror 是独立的实验项目，应该移到单独的目录或仓库；如果是产品依赖的组件，应该补上被主后端引用的接口。一个仓库里有两套后端 Python 代码而不说明关系，对团队协作是隐患。

  🔴 3. 补反滥用设计

  对一个内容质量社区来说，发布后才发现有用户用水贴刷积分，比 UI 不够好看严重得多。至少需要：
  - 发布频率限制（每人/分钟最多 X 条）
  - 举报需要审核机制（不能举报即下架）
  - 积分获取设上限（每人每天最多赚 N 分）

  🟡 4. 拆分超大页面

  AdminPage（62KB）和几个 30KB+ 的页面应该按功能拆成子组件或子路由。现在的体量不利于多人协作修改。

  🟡 5. 为 Phase 2-3 做后端接口契约

  Phase 1 做完 UI 后，如果后端 API 的请求/响应格式跟 mock 不一致，Phase 3 的"逐模块对接"会变成逐模块重写。建议先定下每个核心端点的 OpenAPI 契约，mock 和真实后端都用同一份契约。
