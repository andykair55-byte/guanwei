## 一、需求侧清晰度

### 1.1 核心矛盾：用户到底来"玩"还是来"求真"

整个产品存在一个根本性的定位撕裂——"瓜田"暗示娱乐消遣，"求证"暗示严肃核查。11份策划案没有一份正面回答这个问题：大学生刷手机时，到底是在什么情境下打开观微？

**瓜田系统**的目标用户描述是"18-26岁大学生，每天刷手机3h+"——这不是用户画像，是人口统计。这些学生**现在**在哪里吃瓜？微博热搜、豆瓣小组、微信群、抖音评论区。为什么他们要离开这些零成本场景来观微？策划案没有回答"当前替代方案为什么不够用"。

**佐证系统**假设用户愿意花时间为一个瓜写带来源引用的证据。但目标用户恰恰是"注意力碎片化"的群体——这两个假设互相矛盾。真正愿意写论证的人去的是知乎和维基百科，不是吃瓜App。

**辩论系统**需要6个真人同时在线、在同一时间参与同一话题。凑齐6个水平相当、时间匹配、愿意辩论的大学生——匹配难度极高。如果有人中途离开（断电、断网、室友叫吃饭），辩论就废了。

### 1.2 真正成立的需求

经过逐模块审视，以下需求是**真实成立的**：

- **AI斗蛐蛐/AI竞技场**：用户不需要在线、不需要凑人、纯观赏性、娱乐性强。"AI互搏+围观下注"是一个被验证过的病毒传播模型（类似AI象棋、AI围棋直播）。这是观微最有可能引爆的方向。
- **事件时间线收束**（MEMORY.md中记录的产品方向重定义）：比"猜真假"更成立。痛点真实——热点事件碎片化、记忆黑洞、监督权无工具。核心模型"互相蒸馏"（AI生成初始时间线，用户贡献证据纠正）有清晰的价值闭环。
- **求证工具集（EXIF+情绪检测）**：作为独立小工具有使用价值，但不应作为产品核心。

### 1.3 需求侧判定

| 模块 | 需求真实度 | 判定理由 |
|------|-----------|---------|
| 瓜田 | ⚠️ 中等 | "吃瓜"是真需求，"主动佐证"大概率伪需求 |
| 佐证系统 | ❌ 弱 | 假设用户像学术研究者一样提供证据，不符合目标用户行为 |
| 辩论系统 | ⚠️ 中等 | 结构化辩论有价值，但6人同时在线的要求过于苛刻 |
| AI竞技场 | ✅ 强 | 零参与门槛、纯观赏、娱乐性强、病毒传播潜力高 |
| 热点系统 | ✅ 强 | 信息聚合是真需求，但依赖内容供给 |
| 求证工具集 | ⚠️ 中等 | 单工具有价值，但使用频率极低 |
| 社区系统 | ❌ 弱 | 没有独立存在的理由，必须依附于内容 |
| 笔记系统 | ❌ 弱 | 对hackathon项目是过度设计 |
| 用户体系 | ✅ 必要 | 段位+积分是留存基础设施 |
| 管理后台 | ✅ 必要 | 运营基础设施 |

## 二、可行性边界

### 2.1 逐模块复杂度标注

| 模块 | 策划案描述的功能 | 实际复杂度 | 隐藏复杂度 |
|------|----------------|-----------|-----------|
| 瓜田 | 瀑布流+猜真假+佐证+开奖+推荐流 | 中等→**极难** | "推荐流基于协同过滤"需要大量用户行为数据，hackathon不可能有；开奖判定机制未设计 |
| 佐证 | 5种类型+AI质量评分+实时检测+引用体系 | **极难** | AI质量评分每一个子项（客观性检测、情绪化检测、重复检测）都是独立NLP课题；来源可信度判定是独立项目 |
| 辩论 | 3v3实时+AI评分+候补+房主+禁言+乐观锁 | **极难** | AI评分每场144次LLM调用；实时通信方案未设计；凑不齐人是生死问题 |
| AI竞技场 | AI角色+自动辩论+裁判+弹幕+下注 | 中等 | 不依赖后端可运行，是可行性最高的模块 |
| 热点 | 热搜+热度算法+事件详情+时间线 | 中等→**难** | 热点数据从哪来？爬虫？API？手动录入？热度算法权重无数据支撑 |
| 求证工具 | 6个工具+智能路由+聚合报告 | **极难** | 反向搜图需要图片搜索引擎；多源验证需要爬取+提取+比对；洗稿检测需要embedding模型 |
| 社区 | 4种社区类型+权限+管理 | 中等 | 功能本身不难，但冷启动无内容就是空壳 |
| 笔记 | Notion式块编辑器+动态引用 | **难** | 块编辑器本身开发量大；动态引用需要维护引用链一致性 |
| 用户体系 | 7段位+15徽章+12种积分来源+8种消费+关注体系 | 中等→**难** | 关注流内容聚合是feed系统；积分经济平衡需要数值模拟；Notion式自定义主页过度设计 |
| 管理后台 | 数据看板+用户管理+审核+导出 | 中等 | 已有基础实现，主要依赖后端API完善 |

### 2.2 被严重低估的隐藏复杂度

**1. 内容供给——所有策划案的集体盲区**

11份策划案没有一份讨论"瓜从哪来"。瓜田没有瓜就是空页面，热点没有热点就是空列表。内容供给方案是整个产品的生死问题：
- 靠运营手动录入？运营成本多高，能维持多久？
- 靠爬虫自动抓取？版权风险、内容质量、时效性
- 靠UGC？冷启动期没有用户就没有内容
- 靠LLM生成？内容可信度谁来保证？

**2. LLM调用成本——完全没有评估**

多个模块依赖LLM调用（AI评分、AI质量检测、一键求证、时间线构建、聚合报告、辩论评分），但没有一份策划案评估过：
- 每次调用消耗多少token？日均成本多少？
- 延迟多少秒？用户体验如何？
- LLM服务不可用时如何降级？
- 书生模型9k token/月/30rpm的限额够不够用？

**3. 实时通信——辩论和社区的隐含需求**

辩论系统需要多用户实时同步（发言、投票、状态变化），但策划案没有讨论WebSocket/SSE/轮询方案。前端用Zustand管理内存状态，但多用户同步完全缺失。

### 2.3 MVP裁剪建议

以"黑客松可演示的最小闭环"为锚点：

**保留（P0）**：
- 瓜田核心循环：瀑布流 + 猜真假 + 基础佐证列表（纯文字） + 手动开奖
- AI竞技场：AI自动辩论 + 裁判评分 + 弹幕 + 围观（不依赖后端，独立可运行）
- 基础用户体系：段位 + 简化积分 + 基础个人主页
- 管理后台：已有基础，保持

**降级（P1）**：
- 热点系统：手动录入热点 + 关联瓜 + 基础时间线（不做自动抓取）
- 求证工具：只保留EXIF分析 + 情绪检测 + LLM文字求证，其余标记coming soon
- 辩论系统：简化为AI斗蛐蛐（已有基础），真人辩论降级为demo模式

**砍掉（P2+）**：
- 社区系统（无内容的空壳）
- 笔记系统（Notion式块编辑器开发量巨大）
- 佐证系统的AI评分、引用体系、实时检测
- 用户体系的关注流、徽章系统、Notion式自定义主页
- 协同过滤推荐流

### 3.2 "一句话模块"红旗

以下模块被一句话带过但实际需要大量开发：

- 瓜田"推荐流：基于订阅领域+协同过滤" → 一个完整的推荐系统工程
- 佐证"AI实时质量检测" → 每次按键触发LLM？成本和延迟不可控
- 用户体系"Notion式模块化自定义" → 拖拽排序+自定义布局的前端复杂度
- 求证"反向搜图" → 图片特征提取+搜索引擎，不是前端组件能做的
- 辩论"高并发方案：乐观锁" → 没有真实后端，前端mock的乐观锁无意义

### 3.3 装饰性参考产品

每份策划案都有一个"设计参考"表格，但大部分参考是装饰性的：
- TweetDeck已经停止运营
- Clubhouse已经基本消亡
- 法庭证据规则与佐证系统没有可比性
- GitHub PR Review与任何用户内容系统都不相似
- 把Redis和产品设计产品并列在参考表中

这是典型的AI生成痕迹：用"知名产品"增加可信度，但没有深入分析具体借鉴了什么。

## 五、优先修复清单

按风险从高到低排序：

### 🔴 P0 — 不解决则产品不成立

**1. 内容供给方案**
所有策划案的最大盲区。瓜田没有瓜、热点没有热点、社区没有帖子——整个产品就是空壳。必须明确：MVP阶段的内容从哪来、谁来维护、更新频率多少。建议：LLM生成+人工审核的混合模式，准备50-100条种子内容。

**2. 开奖判定机制**
瓜田核心玩法的闭环缺失。谁来判定真假？判定标准是什么？如果AI判定，准确率如何保证？如果人工判定，运营团队从哪来？建议：MVP阶段用人工判定（管理员在后台操作），AI判定作为v2。

**3. AI竞技场完善**
这是观微最有可能引爆的功能——零参与门槛、纯观赏、娱乐性强、不依赖后端。但当前策划案颗粒度不够：缺少AI角色的具体prompt设计、对战流程状态机、弹幕防刷、下注经济系统。建议：作为P0优先完善。

### 🟡 P1 — 影响MVP可演示性

**4. LLM成本与降级方案**
多个模块依赖LLM，但成本和延迟完全未评估。书生模型9k token/月限额可能不够。需要：评估每个LLM调用的token消耗、设计降级方案（LLM不可用时功能如何退化）、考虑缓存策略减少重复调用。

**5. 防刷基础机制**
所有策划案的防刷停留在"频率限制"。至少需要：设备指纹（而非IP，大学NAT环境不可靠）、行为异常检测（短时间内大量相同操作）、积分获取上限。

**6. 积分经济数值模拟**
积分通胀则无意义，通缩则无人花。需要至少跑1个月/3个月的积分收支预测，确保经济系统平衡。

### 🟢 P2 — 提升方案完整性

**7. 上线后异常应对预案**
LLM服务中断、后端API超时、并发冲突、数据不一致——每个核心模块都需要降级方案。

**8. 新手引导与首次体验**
首次进入的用户需要被引导完成"猜第一个瓜"的完整循环，否则流失率极高。

**9. 辩论凑不齐人的降级策略**
是否允许AI填充空位？是否支持2v2？是否允许异步辩论（每人24小时内发言）？

---

> 审查结论：11份策划案在"功能描述"层面做得不错，但在"落地可行性"层面存在系统性缺陷——内容供给缺失、LLM成本未评估、过度设计、异常场景未覆盖。建议以"AI竞技场+瓜田核心循环+基础用户体系"为MVP三角，其余模块降级或砍掉。
