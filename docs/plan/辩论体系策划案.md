# 见微「辩论体系」策划案

> 版本：v3.0
> 状态：可落地执行
> 最后更新：2025-05

---

## 一、模块定位

**辩论体系是观微的观点碰撞场——MVP聚焦AI斗蛐蛐（归入AI竞技场模块统一实现），真人辩论降级为demo模式供hackathon展示，支持灵活人数（1v1到3v3），允许AI填充空位，辩论结束后内容沉淀为可消费的文章。**

---

## 二、MVP范围定义

### 做什么

| 功能 | 说明 |
|------|------|
| 真人辩论（简化版） | 灵活人数：1v1、2v2、3v3均可开始，不强制凑齐6人 |
| AI填充空位 | 人不够时，空位由AI自动补位发言 |
| 辩论大厅 | 列表展示进行中的辩论房间 |
| 辩论流程 | 创建房间→等待→辩论（最多4轮）→评分→结束 |
| AI评分 | 辩论结束后LLM对每位辩手评分，选出MVP和获胜方 |
| 观众弹幕 | 围观用户可发弹幕互动 |
| 辩论内容沉淀 | 辩论结束后生成总结文章，可被其他用户浏览 |
| AI斗蛐蛐 | 归入AI竞技场模块，本模块不重复实现 |

### 砍掉清单

| 砍掉项 | 原因 |
|--------|------|
| 房主审批候补 | MVP不需要，直接加入即可 |
| 禁言/踢出/拉黑 | hackathon阶段无此需求，人工管理即可 |
| 候补系统 | 人不够用AI填充，不需要候补排队 |
| 乐观锁 | 没有真实后端，mock数据不需要版本控制 |
| 6轮制 | 缩短为4轮，减少LLM调用成本和用户等待时间 |
| 缺席检测+补位 | AI自动填充空位，不需要检测缺席 |
| 举报系统 | MVP阶段人工管理 |
| 段位/积分系统 | 归入用户体系，辩论模块不独立实现 |
| 辩论回放 | 内容沉淀用总结文章替代，不做完整回放 |

AI斗蛐蛐（两个AI自动对战，用户围观下注）归入**AI竞技场模块**统一实现。本模块只保留真人辩论的简化版，当真人不够时用AI填充空位。两者的区别：

| 维度 | 真人辩论（本模块） | AI斗蛐蛐（AI竞技场模块） |
|------|-------------------|------------------------|
| 参与者 | 真人为主，AI补位 | 全部AI |
| 发言方式 | 真人手动输入 | AI自动生成 |
| 用户角色 | 辩手 + 观众 | 纯观众 + 下注 |
| 房间创建 | 用户创建 | 系统自动创建 |

## 五、状态流转

### 5.1 房间状态机

```
WAITING → DEBATING → SCORING → SUMMARY → ENDED
```

| 状态 | 含义 | 允许操作 |
|------|------|----------|
| WAITING | 等待辩手加入 | 加入房间、房主开始辩论 |
| DEBATING | 辩论进行中 | 发言、发弹幕 |
| SCORING | LLM评分中 | 等待（展示"评分中..."） |
| SUMMARY | 总结生成完成 | 查看结果、查看总结 |
| ENDED | 归档 | 仅查看 |

### 5.2 轮次状态

```
每轮：OPEN（可发言）→ TIME_UP（时间到）→ NEXT_ROUND 或 SCORING
```

- 每轮限时90秒
- 每轮每方按座位顺序依次发言（每人限1条，150字以内）
- 4轮结束后进入SCORING

### 5.3 发言顺序（以2v2为例）

```
第1轮：正方1 → 反方1 → 正方2 → 反方2
第2轮：反方1 → 正方1 → 反方2 → 正方2（交替先手）
第3轮：正方1 → 反方1 → 正方2 → 反方2
第4轮：反方1 → 正方1 → 反方2 → 正方2
```

空位由AI自动按顺序发言。

## 六、AI评分方案

### 6.1 评分调用

辩论结束后，将所有发言记录打包发给Groq，一次性评分。

```python
SCORING_PROMPT = """你是一个辩论赛评委。请根据以下辩论记录，对每位辩手评分。

## 辩题
{topic}

## 辩论记录
{messages_formatted}

## 评分要求
对每位辩手给出0-10分的评分，评分维度：
- 逻辑性：论证是否严密
- 说服力：是否能打动听众
- 事实依据：是否有事实支撑
- 表达质量：语言是否清晰有力

## 输出格式（严格JSON）
{
  "scores": [
    {"seat": "正方1", "score": 8, "comment": "逻辑清晰，但缺乏数据支撑"},
    ...
  ],
  "winner": "affirm|negate|draw",
  "mvp": "正方1",
  "summary": "100字以内的辩论总结"
}"""
```

### 6.2 Token预算

| 项目 | 估算 |
|------|------|
| 输入 | 4轮 x 4人 x 150字 = 2400字中文 ≈ 3600 token + prompt模板500 token = ~4100 token |
| 输出 | 4人评分+总结 ≈ 600 token |
| 总计 | ~4700 token/场 |
| 每日预算 | 假设20场辩论/日 = 94k token，在Groq免费额度内 |

### 6.3 AI空位填充

```python
FILL_PROMPT = """你是一个辩论赛辩手。当前轮次轮到{side}{seat_number}号位发言。

## 辩题
{topic}

## 你的立场
{side}

## 前面的发言
{previous_messages}

请给出你的发言（150字以内），要求：
- 紧扣辩题
- 回应对方观点
- 有逻辑、有论据
直接输出发言内容，不要加任何前缀。"""
```

每次AI填充约500 token输入+150 token输出=650 token。一场辩论最多4个空位x4轮=16次填充=10400 token。

## 八、异常场景应对

### 8.1 中途断线

| 场景 | 处理 |
|------|------|
| 真人辩手断线 | 该轮视为放弃发言（空位），下一轮如果恢复则继续；如果连续2轮未发言，后续轮次由AI接管该座位 |
| 观众断线 | 无影响，重连后恢复弹幕 |

### 8.2 房主断线

| 场景 | 处理 |
|------|------|
| WAITING阶段房主断线 | 房间保持30分钟，超时自动关闭。其他用户可接管房主权限（MVP简化：第一个加入的真人自动成为房主） |
| DEBATING阶段房主断线 | 不影响辩论流程（流程自动推进），房主作为普通辩手继续 |

### 8.3 LLM评分失败

| 场景 | 处理 |
|------|------|
| Groq API超时 | 重试1次，仍失败则提示"评分服务暂时不可用，结果稍后公布" |
| 返回格式错误 | 后端catch，返回默认结果：平局，无MVP，summary="评分生成失败" |
| 评分服务完全不可用 | 辩论记录保留，评分标记为"待评分"，管理员可手动触发重新评分 |

### 8.4 LLM填充发言失败

| 场景 | 处理 |
|------|------|
| AI填充超时 | 该空位本轮跳过（视为未发言），不阻塞流程 |
| AI填充内容不当 | MVP阶段无法自动检测，依赖观众弹幕反馈。极端情况管理员可删除发言 |

### 8.5 并发冲突

| 场景 | 处理 |
|------|------|
| 多人同时发弹幕 | 弹幕是追加操作，无冲突 |
| 多人同时加入房间 | 后端用数据库事务保证座位不重复分配 |
| 房主开始辩论与用户加入同时 | 后端状态检查：DEBATING状态不允许加入 |
