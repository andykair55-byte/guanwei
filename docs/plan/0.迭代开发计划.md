# 观微项目迭代开发计划

> 版本：v1.0
> 制定日期：2026-07-09
> 原则：前端→AI→后端，先稳定界面再稳定引擎最后稳定基础设施
> Agent设计求稳，和企业方向一致——坑点多，不冒进

---

## 总体节奏

```
Phase 1: 前端界面重构（Sprint 1-2，10个工作日）
  目标：所有P0模块UI到位，每个页面都能走通mock流程
  方法：参照策划案逐个模块改造现有页面，新增AI创作助手页面
  不碰：后端API、LLM调用逻辑

Phase 2: AI能力集成（Sprint 3，5个工作日）
  目标：LLM统一入口重构完成，AI创作助手核心功能可用
  方法：重构4个hardcode Groq的service → callLLM()，接入创作助手AI功能
  依赖：Phase 1的UI组件已就位

Phase 3: 后端稳定化（Sprint 4-5，10个工作日）
  目标：后端API逐模块对接，Agent容错机制验证
  方法：围绕AI稳定性，逐个API端点从mock切换到真实后端
  依赖：Phase 2的AI调用链路已验证
```

## 风险控制

| 风险 | 应对 |
|------|------|
| Phase 1工作量超预期 | AI创作助手可分阶段：编辑器+骨架必须，风格档案+核查可延后到Phase 2 |
| LLM统一重构引入bug | 重构后逐个验证4个service的mock fallback，保留原代码git分支可回退 |
| 后端Agent不稳定 | MVP用最简单方案（单次LLM调用+mock fallback），不上Mirror Commander多Agent |
| 积分经济不平衡 | Phase 3上线前跑一次数值模拟（参考用户体系策划案5.3节） |
| 种子内容质量差 | 人工审核每条种子瓜，不用LLM直出 |

---

> 核心原则：前端先把界面和交互跑通（用mock），再让AI接管内容生成，最后让后端接管数据持久化。
> 每一步都保留mock fallback，任何一层挂了不影响用户体验。
