# 观微「AI 创作助手」策划案

> 版本：v1.0
> 状态：产品策划 + 技术规格
> 优先级：P0（产品活路）
> 最后更新：2026-07

---

## 一、模块定位

**AI创作助手是观微的内容供给侧引擎——不是又一个聊天式AI写作工具，而是嵌入创作流程的结构化辅助系统。它学习用户的写作风格，在写的过程中提供结构建议、事实核查、论据补充，帮用户用自己的声音写出高质量内容。**

解决的问题：
1. **起步难**——面对空白页面不知道怎么写，AI不是给成品，而是给骨架让用户填肉
2. **质量低**——写出来的东西缺乏论据支撑，AI在写作过程中实时调用求证工具验证关键声明
3. **门槛高**——有经验的人没时间，有时间的人没经验，AI降低从"想法"到"可发布内容"的摩擦
4. **风格丢失**——现有AI工具产出千人一面的内容，创作助手学习每个用户的写作风格，辅助而非替代

给谁用：
- **核心用户**：想在瓜田/社区发高质量内容的大学生（有想法但缺写作经验）
- **延伸用户**：校园自媒体/博主（有内容产出需求但缺时间做调研和核查）
- **不是给谁用的**：只想看不想写的纯消费者（他们消费AI辅助产出的内容，但不使用创作助手本身）

**与产品核心叙事的关系**：
"提升全民信息素养"不是教人辨别真假（没人喜欢被教育），而是帮人写出有信息密度的内容（人人都想表达得更好）。AI创作助手是这个理念的产品化落地。

## 二、MVP 范围定义

### 2.1 做什么

| 功能 | 状态 | 说明 |
|------|------|------|
| 结构化创作编辑器 | **新增** | 不是空白textarea，是带AI骨架引导的结构化编辑器 |
| 写作风格档案 | **新增** | 基于用户历史帖子分析写作风格，生成style profile |
| AI骨架生成 | **新增** | 用户输入主题/观点 → AI生成文章骨架（大纲+关键论点+需核查的声明） |
| 写作中事实核查 | **新增** | 写作过程中，AI标记需要证据支撑的声明，一键调用求证工具验证 |
| 风格适配建议 | **新增** | AI基于用户style profile，对草稿给出风格一致性建议 |
| 一键发布到瓜田/社区 | **新增** | 完成的内容可直接发布为瓜的佐证/社区帖子 |
| 内容质量评分 | **新增** | 发布前给出质量预览（论据密度、逻辑连贯性、信息增量） |

### 2.2 砍掉清单

| 砍掉项 | 原因 |
|--------|------|
| 聊天式交互 | 市面上已有太多ChatGPT套壳，不做对话，做编辑器内嵌辅助 |
| 全自动生成 | 用户必须参与思考过程，AI给骨架不给成品 |
| 完整研究Agent | MVP不需要自动搜索+整理资料，用户手动引入证据 |
| 协作编辑 | 过度设计，MVP只做单人创作 |
| 模板市场 | 无用户基础，MVP用预设结构替代 |
| 付费功能 | 先跑通免费流程，商业化v2 |
| 多语言支持 | 目标用户是中文大学生 |
| 图片/视频生成 | 只做文字内容 |

### 2.3 与现有AI写作工具的核心差异

| 维度 | 现有工具（ChatGPT/文心等） | 观微AI创作助手 |
|------|--------------------------|---------------|
| 交互模式 | 聊天式：prompt → 全文输出 | 编辑器内嵌：AI给骨架，用户填内容 |
| 产出质量 | 千人一面，AI味重 | 学习用户风格，产出像用户自己写的 |
| 事实核查 | 无，AI可能编造 | 写作中实时调用求证工具验证声明 |
| 内容去向 | 复制粘贴到各平台 | 一键发布到观微瓜田/社区 |
| 定位 | 通用写作工具 | 垂直于信息密度型内容的创作辅助 |

## 三、技术架构

### 3.1 前端文件结构

```
src/
  pages/
    CreatePage.tsx                    -- 【新增】创作主页面：编辑器+AI辅助面板
  components/
    create/
      StructuredEditor.tsx           -- 【新增】结构化编辑器（非块编辑器，是带引导的富文本）
      AISkeletonPanel.tsx            -- 【新增】AI骨架生成面板（侧栏）
      StyleProfileCard.tsx           -- 【新增】写作风格档案卡片
      ClaimVerifier.tsx              -- 【新增】声明核查组件（内联在编辑器中）
      QualityPreview.tsx             -- 【新增】发布前质量预览
      PublishTargetSelector.tsx      -- 【新增】发布目标选择（瓜田/社区/笔记）
  services/
    creationService.ts              -- 【新增】创作辅助AI服务（统一LLM调用入口）
    styleProfileService.ts          -- 【新增】写作风格分析服务
  stores/
    creationStore.ts                -- 【新增】创作状态zustand store
```

### 3.2 后端API路径

```
POST   /api/creation/skeleton         -- 生成文章骨架
POST   /api/creation/verify-claim     -- 核查单个声明
POST   /api/creation/style-profile    -- 生成/更新写作风格档案
POST   /api/creation/quality-score    -- 内容质量评分
POST   /api/creation/publish          -- 发布到瓜田/社区/笔记
```

### 3.3 与Mirror引擎的集成路径

**MVP阶段（前端直调LLM）**：

创作助手的AI调用走主应用的 `llmStore.callLLM()` 统一入口，使用用户配置的provider（Groq/OpenAI/DeepSeek）。这与现有4个hardcode Groq的service不同——创作助手是第一个真正使用统一LLM入口的模块。

```
用户输入主题
    ↓
creationService.ts → llmStore.callLLM() → Groq/用户配置的provider
    ↓
返回骨架/建议/评分
```

**v2阶段（接入Mirror Commander）**：

当Mirror引擎部署后，创作助手切换为通过Mirror的Commander进行多Agent协作：

```
用户输入主题
    ↓
Mirror Commander（意图分析 → 任务分解）
    ├── Researcher Agent（搜索相关资料，调用abilities工具集）
    ├── Outliner Agent（基于研究结果生成骨架）
    └── Style Agent（基于用户style profile调整表达）
    ↓
热替换系统兜底：
  - 某个Agent超时 → AgentPool热替换，checkpoint交接
  - LLM provider不可用 → 降级到备用provider
  - 输出质量不达标 → OutputValidator触发重试或换Agent
```

**为什么这个路径可行**：
- Mirror已有6种persona预设（creative/critical/efficient/meticulous/patient/warm-teacher），可直接映射为不同的写作辅助风格
- Mirror的AgentPool热替换+ConvergenceMonitor确保创作过程不会因单点故障中断
- Mirror的认知记忆系统（MyDesignBackend + TemporalStore）可存储用户的写作风格档案，跨会话保持
- Mirror的hot-replacement通过WebSocket（WorldBridge）实时推送状态变化到前端，用户可以看到"AI正在搜索相关资料..."等实时状态

### 3.4 统一LLM调用入口重构

当前主应用有4个service各自hardcode Groq调用（aiJudgeService/aiSummaryService/aiTranslatorService/emotionAnalysis），创作助手是统一这些调用的起点：

```
重构前：
  aiJudgeService.ts      → fetch(groq.com, ...)     // 独立调用
  aiSummaryService.ts    → fetch(groq.com, ...)     // 独立调用
  aiTranslatorService.ts → fetch(groq.com, ...)     // 独立调用
  emotionAnalysis.ts     → fetch(groq.com, ...)     // 独立调用
  creationService.ts     → fetch(groq.com, ...)     // 又一个独立调用

重构后：
  llmStore.callLLM()    → 统一入口，读取用户配置
    ├── aiJudgeService      → callLLM()
    ├── aiSummaryService    → callLLM()
    ├── aiTranslatorService → callLLM()
    ├── emotionAnalysis     → callLLM()
    ├── creationService     → callLLM()    ← 新模块
    └── styleProfileService → callLLM()    ← 新模块
```

这样用户的LLM配置（在LLMSettingsPage设置的provider/key/model）才能真正全局生效，而不是只有验证功能用到。

## 四、核心功能详细设计

### 4.1 结构化创作编辑器

**不是**Notion式块编辑器（开发量巨大），**而是**带AI引导的结构化表单+富文本区域。

**编辑流程**：

```
Step 1: 选题与立场
  用户输入：
    - 主题（如"南开大学学生被骗220万"）
    - 立场/观点（如"大学反诈教育严重缺失"）
    - 目标读者（如"大学生"）
  AI辅助：
    - 检查瓜田是否已有同主题瓜（避免重复）
    - 建议切入角度（基于已有佐证的时间线空白点）

Step 2: AI生成骨架
  输入：主题+立场+目标读者
  AI输出：
    - 标题建议（3个备选）
    - 文章骨架：
      ├── 开头：事件概述（需填入关键事实）
      ├── 论点1：[AI建议的论点方向]（需填入论据）
      ├── 论点2：[AI建议的论点方向]（需填入论据）
      ├── 反驳预判：读者可能的质疑（需填入回应）
      └── 结尾：总结观点+呼吁行动
    - 标记需要核查的声明（用 ⚠️ 标记）

  用户操作：
    - 可接受/修改/删除每个骨架节点
    - 可调整顺序（拖拽或上下箭头）
    - 确认后进入写作阶段

Step 3: 填充写作
  用户在骨架引导下逐段写作
  AI辅助（实时/按需）：
    - 检测到声明性语句 → 自动标记为"需核查"
    - 点击 ⚠️ 标记 → 一键调用求证工具验证
    - 段落过长/过短 → 提示调整
    - 检测情绪化表达 → 提示"这段可能影响可信度"

Step 4: 质量预览与发布
  发布前AI给出质量评分：
    - 论据密度：每100字有几个可验证的论据
    - 逻辑连贯性：论点之间的逻辑链是否完整
    - 信息增量：相比已有内容，新增了什么信息
    - 风格一致性：与用户历史写作风格的匹配度
  用户确认后选择发布目标：
    - 发布为瓜田佐证
    - 发布为社区帖子
    - 保存为笔记（私密）
```

### 4.2 写作风格档案（Style Profile）

**核心差异化功能**——让AI产出"像用户自己写的"内容，而不是"像AI写的"。

**构建方式**：

```
输入：用户最近10篇帖子/佐证/笔记（如果不足10篇，用已有的所有）
分析维度：
  - 句式偏好：长句vs短句、主动vs被动
  - 用词特征：高频词汇、口头禅、专业术语使用频率
  - 结构偏好：总分总vs递进、是否有总结段
  - 情绪基调：理性分析vs感性表达vs讽刺幽默
  - 论据风格：数据型vs案例型vs类比型
输出：StyleProfile对象
  {
    sentenceLength: 'medium',        // short/medium/long
    tone: 'analytical',              // analytical/emotional/satirical/balanced
    structure: 'thesis-first',       // thesis-first/narrative/progressive
    evidenceStyle: 'data-heavy',     // data-heavy/case-heavy/analogy-heavy
    vocabulary: ['说白了', '本质上', '从数据看'],  // 高频特征词
    avoidPatterns: ['综上所述', '不难发现'],        // 用户不用的表达
    samplePassages: [...]            // 代表性段落（用于few-shot）
  }
```

**更新机制**：
- 用户每发布一篇内容，自动增量更新style profile
- 用户可在StyleProfileCard中手动调整（如"我希望更正式一点"）
- Style profile存储在用户store中，MVP阶段前端localStorage持久化，v2迁移到后端

**在创作中的使用**：
- AI生成骨架时，注入style profile作为system prompt的一部分
- 风格适配建议基于style profile对比当前草稿
- 确保AI辅助产出的内容读起来像用户自己写的

### 4.3 写作中事实核查（Claim Verifier）

**这是观微独有的功能**——其他AI写作工具没有求证能力。

**工作流程**：

```
用户在编辑器中写下一句话：
  "据统计，2025年大学生遭遇电信诈骗的比例上升了37%"

ClaimVerifier检测到这是一个需要证据支撑的声明
  ↓
编辑器内联显示：
  ⚠️ "据统计...上升了37%" — 需要来源支撑
  [一键核查] [添加来源] [忽略]
  ↓
用户点击 [一键核查]：
  → 调用求证工具集的LLM文字求证功能
  → 搜索相关数据源
  → 返回核查结果：
    ✅ 找到支撑：教育部2025年报告确实显示上升37%（来源URL）
    或
    ❌ 未找到支撑：未找到权威来源确认此数据
    或
    ⚠️ 数据有出入：教育部报告显示上升27%，非37%
  ↓
用户根据核查结果修改内容
```

**技术实现**：
- 声明检测：用LLM从当前段落中提取需要核查的声明（一次性调用，~500 token）
- 核查：复用求证工具集的 `verificationStore.verify()` 流程
- 结果展示：内联在编辑器中，不跳转页面

### 4.4 内容质量评分

**发布前的最后一道关**——不是阻止发布，而是给用户一个预期。

```
评分维度（各25分，总分100）：

1. 论据密度（25分）
   - 每100字有1个可验证论据 = 15分
   - 每100字有2个+ = 25分
   - 无可验证论据 = 5分

2. 逻辑连贯性（25分）
   - AI评估论点之间的逻辑链
   - 有因果关系、有过渡 = 高分
   - 论点跳跃、自相矛盾 = 低分

3. 信息增量（25分）
   - 对比瓜田/社区已有内容
   - 提供了新信息/新角度 = 高分
   - 重复已有内容 = 低分

4. 风格一致性（25分）
   - 对比用户style profile
   - 读起来像用户自己写的 = 高分
   - AI味重 = 低分

展示方式：
  总分 + 4维度雷达图 + 具体改进建议
  60分以下：建议修改后再发布
  60-80分：可以发布，标注改进空间
  80分以上：高质量内容，鼓励发布
```

## 六、异常场景应对

### 6.1 LLM服务不可用

| 场景 | 降级方案 |
|------|---------|
| 骨架生成时LLM不可用 | 提供3个预设模板骨架（事件分析/观点论述/对比评测），用户手动填充 |
| 事实核查时LLM不可用 | 标记声明为"待核查"，提示用户手动搜索验证 |
| 质量评分时LLM不可用 | 降级为纯规则评分（字数、段落数、链接数等可统计指标） |
| 风格分析时LLM不可用 | 使用默认风格（理性分析型），用户可手动调整 |

### 6.2 用户无历史内容

新用户没有帖子供风格分析：
- 提供6种预设风格供选择（对应Mirror的6种soul预设：creative/critical/efficient/meticulous/patient/warm-teacher）
- 用户选一种作为初始风格
- 发布第一篇内容后自动切换到基于实际内容的风格档案

### 6.3 内容质量过低

质量评分<40分时的处理：
- 不阻止发布（用户有发布自由）
- 显示具体改进建议（"建议增加至少2个可验证论据"）
- 发布后自动标注"AI辅助创作"标签（透明性）

### 6.4 并发与性能

| 场景 | 处理 |
|------|------|
| 骨架生成耗时（3-8秒） | 显示生成进度，不阻塞编辑器其他操作 |
| 同时多个用户在创作 | 每次骨架生成独立调用，互不影响 |
| 风格档案更新 | 增量更新，不需要重新分析所有历史帖子 |

### 6.5 Mirror引擎集成异常

当创作助手切换到Mirror Commander时的异常处理：
- Mirror服务不可达 → 自动降级为前端直调LLM（Phase 1-4的实现方式）
- Commander任务分解失败 → 跳过任务分解，直接单次LLM调用生成骨架
- Agent热替换触发 → 通过WebSocket在前端显示"AI正在切换思路..."
- Agent输出质量不达标（OutputValidator拦截） → 自动重试1次，仍失败则降级为前端直调

### 7.1 冷启动

- 运营团队准备10篇"AI辅助创作"的示范帖子，展示创作助手的实际效果
- 在瓜田详情页增加"写分析"按钮（入口），引导用户尝试创作
- 新用户首次进入创作页时，提供guided tour（3步引导：选题→生成骨架→填充发布）

### 7.2 内容质量飞轮

```
AI辅助降低创作门槛
    → 更多用户写内容
    → 内容池变大
    → 消费者有更多内容可看
    → 创作者获得反馈（点赞/评论/佐证支持）
    → 激励创作者写更多
    → 风格档案更准确
    → AI辅助质量更高
    → 更多人愿意写
    （正循环）
```

## 九、与现有模块的联动

```
AI创作助手 → 瓜田系统
  创作的内容一键发布为瓜的佐证
  瓜详情页增加"写分析"入口 → 跳转创作助手并预填瓜信息

AI创作助手 → 社区系统
  创作的内容一键发布为社区帖子
  社区帖子编辑器增加"AI辅助"按钮

AI创作助手 → 求证工具集
  写作中的事实核查调用求证工具的LLM文字求证功能
  核查结果内联在编辑器中

AI创作助手 → 用户体系
  创作并发布内容 → 积分奖励（+5/篇，高于佐证+3，激励高质量内容）
  质量评分>80 → 额外积分奖励（+3）
  创作统计纳入个人主页（"发布了X篇分析"）

AI创作助手 → AI竞技场
  创作助手的LLM调用复用AI竞技场的provider配置
  未来：AI竞技场中的精彩辩论 → 自动转化为"辩论分析"模板供创作助手使用

AI创作助手 → Mirror引擎（v2）
  创作助手的AI调用从前端直调切换为Mirror Commander
  利用Mirror的热替换确保创作过程不中断
  利用Mirror的认知记忆存储用户风格档案（跨会话持久化）
  利用Mirror的soul库提供不同写作辅助人格（如"严谨学术风"/"犀利评论风"）
```

---

> AI创作助手是观微从"猜真假社区"进化为"AI辅助内容质量社区"的核心引擎。
> 它不解决"用户不想写"的问题（那是需求不存在），而是解决"用户想写但写不好"的问题（这是真实痛点）。
> 小而精，不做聊天机器人，做编辑器里的AI副驾驶。
