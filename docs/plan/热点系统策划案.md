# 见微「热点系统」策划案

> 版本：v3.0
> 状态：可落地执行
> 最后更新：2025-05

---

## 一、模块定位

**热点系统是观微的流量入口与事件时间线载体——管理员录入热点事件，LLM辅助生成时间线，用户通过参与瓜田和辩论贡献热度，所有热点按参与人数加时间衰减排序。**

---

## 二、MVP范围定义

### 做什么

| 功能 | 说明 |
|------|------|
| 热点列表页 | 按热度排序展示事件卡片，点击进入详情 |
| 事件详情页 | 展示时间线 + 关联瓜 + 关联辩论 |
| 事件时间线 | 管理员创建事件时LLM辅助生成时间线节点，支持手动编辑 |
| 热度排序 | 参与人数 + 时间衰减，简单透明 |
| 热点-瓜田联动 | 热点自动关联同话题瓜，瓜的参与度计入热度 |

### 砍掉清单

| 砍掉项 | 原因 |
|--------|------|
| "地区"Tab | 无数据来源，大学用户群体无地区区分需求 |
| 协同过滤推荐 | 用户量不足以支撑，MVP无意义 |
| 自动爬取热点 | 无稳定数据源，法律风险，MVP用管理员录入 |
| 热度趋势曲线 | 需要历史数据积累，MVP阶段无数据 |
| "上升"/"新"标记 | 需要历史对比数据，MVP不做 |
| 阅读量统计 | 无法准确统计且无实际意义 |

## 三、技术架构

### 3.1 前端

```
src/
  pages/
    HotListPage.tsx              # 热点列表页
    HotEventDetailPage.tsx       # 事件详情页
  components/
    hot/
      HotEventCard.tsx           # 热点事件卡片
      TimelineView.tsx           # 时间线展示组件
      RelatedMelonList.tsx       # 关联瓜列表
      RelatedDebateList.tsx      # 关联辩论列表
  stores/
    hotStore.ts                  # Zustand store：热点列表、当前事件
  services/
    hotService.ts                # API调用封装
```

### 3.2 后端

```
backend/
  app/
    routers/
      hot.py                     # 热点CRUD + 列表 + 详情
    services/
      hot_service.py             # 热度计算、关联查询
      timeline_service.py        # LLM调用生成时间线节点
    models/
      hot_event.py               # 热点事件ORM
      timeline_node.py           # 时间线节点ORM
```

### 3.3 LLM调用（时间线生成）

- 调用方：后端 `timeline_service.py`
- 模型：Groq `llama-3.3-70b`（速度快，免费额度充足）
- 触发时机：管理员创建热点事件时，传入事件标题+描述，LLM返回时间线节点数组
- Prompt模板：

```
你是一个事件时间线整理助手。根据以下事件描述，按时间顺序列出关键节点。
每个节点包含：日期（YYYY-MM-DD格式，不确定的用"约"前缀）、事件摘要（30字以内）。
输出JSON数组格式。

事件标题：{title}
事件描述：{description}
```

- 返回格式：`[{"date": "2025-05-10", "summary": "微博首次爆料"}, ...]`
- 管理员可在前端编辑/删除/新增节点后保存

## 五、状态流转

### 5.1 热点事件状态机

```
DRAFT（草稿）→ PUBLISHED（已发布）→ EXPIRED（已过期）
                                          ↑
                              管理员手动下架 → REMOVED
```

| 状态 | 含义 | 用户可见 |
|------|------|----------|
| DRAFT | 管理员创建中，时间线可能还在编辑 | 不可见 |
| PUBLISHED | 热点上线，用户可参与 | 可见 |
| EXPIRED | 超过有效期或管理员手动过期 | 沉底展示，不可新增参与 |
| REMOVED | 管理员删除 | 不可见 |

### 5.2 热度计算规则

```python
def calculate_hot_score(event):
    """
    热度 = 参与人数 x 时间衰减

    参与人数 = 关联瓜的猜的人数 + 佐证数 + 评论数 + 关联辩论的参与人数
    时间衰减 = max(0.1, 1 - hours_alive / 168)  # 168小时=7天线性衰减到0.1
    """
    participant_count = (
        sum(melon.guess_count + melon.evidence_count + melon.comment_count
            for melon in event.related_melons)
        + sum(debate.participant_count for debate in event.related_debates)
    )
    hours_alive = (now() - event.created_at).total_seconds() / 3600
    time_decay = max(0.1, 1 - hours_alive / 168)
    return round(participant_count * time_decay)
```

权重说明：不做权重分配。一个参与行为就是一个参与行为，不区分类型。原因：MVP阶段没有数据支撑权重调优，拍脑袋的权重只会引入bug。

## 六、异常场景应对

### 6.1 LLM不可用（时间线生成失败）

| 场景 | 处理 |
|------|------|
| Groq API超时（>10s） | 前端提示"AI生成超时，请手动添加时间线节点" |
| Groq API返回错误 | 同上，降级为纯手动录入 |
| 返回格式解析失败 | 后端catch异常，返回空时间线，管理员手动编辑 |

### 6.2 热点过期处理

| 规则 | 说明 |
|------|------|
| 自动过期 | 发布后7天自动标记EXPIRED（后端定时任务每小时检查） |
| 手动延期 | 管理员可将过期时间延长至14天 |
| 过期后展示 | 沉入列表底部，标记"已结束"，不可新增猜/佐证/辩论 |
| 过期后数据 | 已产生的瓜和辩论内容保留，不影响历史数据 |

### 6.3 关联瓜被删除后的级联

| 场景 | 处理 |
|------|------|
| 瓜被管理员删除 | 从热点关联列表移除，重新计算热度 |
| 瓜被用户自己删除 | 同上 |
| 热点下所有瓜被删除 | 热度降为0，但热点本身不自动下架（时间线仍有价值） |

### 6.4 并发冲突

| 场景 | 处理 |
|------|------|
| 管理员同时编辑同一热点 | 后写入者覆盖（MVP阶段管理员只有1-2人，冲突概率极低） |
| 热度计算与用户参与并发 | 热度非实时计算，每次请求时重新算或5分钟缓存，无冲突 |

### 6.5 恶意刷量

| 场景 | 处理 |
|------|------|
| 单用户重复猜同一瓜刷热度 | 同一用户对同一瓜只计1次参与 |
| 管理员滥用热点 | 操作日志记录，hackathon阶段人工监督 |

## 七、上线后运营

| 运营动作 | 频率 | 说明 |
|----------|------|------|
| 录入新热点 | 每日1-2个 | 管理员从微博/知乎/校园群选取话题 |
| 过期热点清理 | 每周1次 | 将过期热点归档，保持列表简洁 |
| 时间线质量检查 | 每次创建时 | 管理员检查LLM生成的时间线，修正不准确节点 |
| 关联瓜检查 | 每次创建时 | 确认关联瓜的话题标签匹配正确 |

热点数量控制：同时在线热点不超过10个。宁缺毋滥，每个热点都要有2个以上关联瓜才值得上线。
