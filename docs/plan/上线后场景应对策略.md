# 观微上线后场景应对策略

> 版本：v1.0
> 适用阶段：hackathon上线后 0-30天运营期
> 目标：确保系统在真实用户环境下稳定运行，快速响应异常

---

## 一、系统异常应对

### 1.1 LLM服务中断

**场景**：Groq或书生模型API不可用（网络故障/服务升级/额度耗尽）。

**降级方案**：

| 模块 | 降级行为 | 用户看到什么 |
|------|---------|------------|
| **AI创作助手-骨架生成** | **降级为3个预设模板骨架（事件分析/观点论述/对比评测），用户手动填充** | **"AI暂时无法生成骨架，已为你准备模板"** |
| **AI创作助手-事实核查** | **标记声明为"待核查"，提示用户手动搜索验证** | **声明旁显示"待手动核查"标签** |
| **AI创作助手-质量评分** | **降级为纯规则评分（字数、段落数、链接数等可统计指标）** | **评分标注"简易评分模式"** |
| **AI创作助手-风格分析** | **使用默认风格（理性分析型），用户可手动选择其他预设** | **"已使用默认风格，可手动切换"** |
| AI竞技场 | 暂停新对战，进行中的对战用预设台词完成 | "AI正在休息中，请稍后再试" |
| 求证工具-一键求证 | 返回缓存的相似查询结果（如果有），否则提示不可用 | "AI分析暂时不可用" |
| 求证工具-截图鉴定 | 正常（不依赖LLM，纯前端exifr解析） | 无影响 |
| 求证工具-情绪检测 | 降级为简单关键词匹配（正面/负面词库） | 结果标注"简易模式" |
| 内容审核 | 切换为纯关键词过滤模式 | 无感知 |
| 种子帖子生成 | 暂停生成任务 | 无感知 |

**缓存策略**：
```
- AI竞技场：相同角色组合+相似话题的辩论结果缓存24h
- 求证工具：相同输入文本的求证结果缓存7天（LRU，最多1000条）
- 裁判评分：缓存评分prompt的最近100次调用结果
```

**额度监控**：
```
Groq免费额度（主力）：~500K token/分钟
  - AI创作助手日均~1.1M token → Groq完全可覆盖
  - 达到RPM限制 → 创作助手排队等待（显示"AI正在思考..."）

书生模型限额（辅助，仅用于高质量场景）：9k token/月
- 达到70%（6.3k token）→ 管理员告警
- 达到90%（8.1k token）→ 自动限制非核心调用（种子生成、非紧急审核、风格分析）
- 达到100% → 仅保留猜瓜核心流程+AI竞技场，创作助手降级为预设模板

LLM调用优先级（从高到低）：
1. AI竞技场对战（核心体验，不可中断）
2. 瓜田开奖实锤报告（影响积分结算）
3. AI创作助手-骨架生成（用户正在创作，不可打断）
4. AI创作助手-事实核查（可延迟，标记"待核查"即可）
5. AI创作助手-质量评分（可降级为规则评分）
6. 求证工具-一键求证（可缓存）
7. 种子内容生成（后台任务，可随时暂停）
```

**实现方式**：`llmStore` 中增加 `fallbackMode` 开关，各模块service层调用LLM前检查此开关。

### 1.2 后端API超时/宕机

**前端回退策略**：

| API | 超时阈值 | 回退方案 |
|-----|---------|---------|
| GET /melons | 3s | 展示localStorage缓存的上次瓜列表，标注"离线模式" |
| GET /melon/:id | 3s | 展示缓存的瓜详情（如果有），否则显示加载失败 |
| POST /guess | 5s | 乐观更新（先显示已猜，后台重试3次），失败后提示"网络异常，请重试" |
| POST /evidence | 5s | 本地保存草稿（localStorage），提示"已保存草稿，恢复后自动提交" |
| GET /community/posts | 3s | 展示缓存的帖子列表 |
| AI辩论相关 | 10s | SSE流式响应，超时后显示"AI思考中..." |

**重试机制**：
```typescript
// api.ts 统一封装
const fetchWithRetry = async (url: string, options: RequestInit, retries = 3) => {
  for (let i = 0; i < retries; i++) {
    try {
      const res = await fetch(url, { ...options, signal: AbortSignal.timeout(5000) })
      if (res.ok) return res
    } catch (e) {
      if (i === retries - 1) throw e
      await new Promise(r => setTimeout(r, 1000 * (i + 1))) // 1s, 2s, 3s递增
    }
  }
}
```

### 1.3 数据库异常

**SQLite→PostgreSQL迁移期间**：

| 阶段 | 策略 |
|------|------|
| 迁移前 | SQLite只读备份，确保数据完整 |
| 迁移中 | 停机维护，展示维护页面（预计15-30分钟） |
| 迁移后 | 双写7天（SQLite+PostgreSQL），验证数据一致性后停止SQLite写入 |
| 回滚 | 保留SQLite备份30天，可随时回滚 |

**数据一致性校验脚本**：
```bash
# 每日运行，对比SQLite和PostgreSQL的关键数据
python scripts/verify_data_consistency.py
# 对比项：用户数、瓜数、佐证数、帖子数、积分记录数
# 差异>0 → 告警
```

### 1.4 前端构建/部署失败

| 场景 | 应对 |
|------|------|
| Vite构建失败 | CI/CD流水线检测构建状态，失败时不部署，保留上一版本 |
| CDN故障 | 配置备用CDN（阿里云CDN主 + 腾讯云CDN备），DNS切换 |
| 新版本有严重Bug | 保留最近3个版本的构建产物，一键回滚到上一版本 |
| 静态资源404 | index.html中设置版本号路径（`/v1.2.3/assets/`），回滚时路径同步变化 |

## 二、用户反馈处理

### 2.1 反馈收集机制

| 渠道 | 方式 | 覆盖场景 |
|------|------|---------|
| 应用内反馈按钮 | 每个页面底部"反馈"入口 | Bug报告、功能建议 |
| 社区"反馈建议"社区 | 10个官方社区之一 | 用户公开讨论 |
| 问卷星/腾讯问卷 | 上线第3天推送给所有用户 | 结构化满意度调查 |
| 微信用户群 | 首批500用户邀请入群 | 实时反馈、深度交流 |

### 2.2 反馈分类与响应

| 分类 | 定义 | 示例 | 响应SLA |
|------|------|------|---------|
| P0 Bug | 核心功能不可用 | 猜瓜提交失败、AI竞技场崩溃、登录失败 | 24小时内修复 |
| P1 Bug | 功能异常但有绕行方案 | 佐证投票不更新、段位显示错误 | 72小时内修复 |
| P2 Bug | UI/体验问题 | 布局错位、动画卡顿、文案错误 | 下一版本修复 |
| 功能请求 | 用户想要的新功能 | "希望支持图片佐证" | 记录到需求池，每周评审 |
| 体验优化 | 现有功能的改进建议 | "瓜卡片信息太少" | 记录到需求池 |
| 内容质量 | 瓜/佐证/帖子质量问题 | "这个瓜信息不实" | 24小时内核实处理 |

### 2.3 反馈处理流程

```
用户提交反馈
  ↓
自动分类（根据关键词+反馈类型选择）
  ↓
进入对应队列：
  - Bug → GitHub Issues（带优先级标签）
  - 功能请求 → 需求池文档
  - 内容问题 → 管理后台审核队列
  ↓
处理完成后：
  - 应用内通知反馈者"您反馈的问题已修复"
  - 微信群同步修复进展
```

### 2.4 用户投诉处理

| 投诉类型 | 处理方式 |
|---------|---------|
| 账号被封 | 管理员核实→误封则解封+道歉→确认违规则说明原因 |
| 内容被删 | 管理员核实→误删则恢复→确认违规则说明规则 |
| 积分异常 | 查询积分流水→纠正错误→补偿积分 |
| 对其他用户的投诉 | 核实证据→按社区规则处理→告知投诉者结果 |

## 三、性能瓶颈解决方案

### 3.1 前端首屏加载 > 2s

**优化路径（按优先级排序）**：

| 优化项 | 预期收益 | 实施难度 |
|--------|---------|---------|
| 路由级代码分割（React.lazy） | 首屏JS减少40% | 低 |
| 图片懒加载（loading="lazy"） | 首屏请求减少50% | 低 |
| 骨架屏替代loading | 感知速度提升（非实际速度） | 低 |
| 预连接关键域名（preconnect） | DNS+TLS节省200-500ms | 极低 |
| 关键CSS内联 | 减少render-blocking | 中 |
| Service Worker缓存静态资源 | 二次访问秒开 | 中 |
| CDN部署 | 根据用户地理位置加速 | 中 |

**执行顺序**：先做前4项（1天内完成），如果仍>2s再做后3项。

### 3.2 后端并发 > 100

**扩容方案**：

| 阶段 | 并发量 | 方案 |
|------|--------|------|
| 阶段1 | < 100 | FastAPI单进程+uvicorn，SQLite |
| 阶段2 | 100-500 | uvicorn workers=4 + PostgreSQL + Redis缓存热点数据 |
| 阶段3 | 500-2000 | 2台服务器+负载均衡（Nginx）+ PostgreSQL + Redis |
| 阶段4 | > 2000 | 容器化部署（Docker Compose）+ 水平扩展 |

**关键优化点**：
- 瓜列表：Redis缓存，TTL 60秒（瓜数据变化频率低）
- 用户信息：Redis缓存，TTL 300秒
- 佐证列表：分页加载，每页20条，避免全量查询
- LLM调用：异步队列，不阻塞主线程

### 3.3 LLM日均调用 > 1000次

**成本控制**：

| 策略 | 具体做法 | 预期节省 |
|------|---------|---------|
| 结果缓存 | 相同输入7天内返回缓存结果 | 30-50% |
| 调用限频 | 单用户每日最多50次LLM调用 | 防止滥用 |
| 模型降级 | 非核心场景用更便宜的模型（书生模型→Groq） | 20%成本 |
| 批量处理 | 种子内容生成等离线任务合并批量调用 | 10-20% |
| Prompt优化 | 缩短system prompt，减少输入token | 15% |

**预算上限**：
```
书生模型：9k token/月（免费额度）
Groq：按量付费，设置月预算上限$10
超出预算 → 自动降级为缓存模式，停止实时LLM调用
```

### 3.4 数据库数据量 > 10万条

**查询优化**：

| 数据表 | 优化措施 |
|--------|---------|
| melons | 添加索引：category, status, created_at；列表查询分页（每页20条） |
| evidences | 添加索引：melon_id, direction, vote_count；按瓜ID查询时走索引 |
| community_posts | 添加索引：community_id, sort_type；热门排序预计算score |
| comments | 添加索引：post_id, created_at；按帖子ID查询 |
| points_records | 添加索引：user_id, created_at；按月查询用户积分流水 |
| users | 添加索引：rank, points；排行榜查询走索引 |

**归档策略**：
- 已开奖且超过90天的瓜 → 归档表（melons_archive）
- 超过180天的积分记录 → 归档表（points_records_archive）
- 归档数据仍可查询，但不参与默认排序

## 四、内容安全

### 4.1 UGC内容审核流程

```
用户提交内容
  ↓
前端预检：
  - 敏感词库匹配（维护200+敏感词列表）
  - 命中 → 拦截，提示"内容包含不当用语"
  ↓
后端AI预审（LLM可用时）：
  - Prompt: "判断以下内容是否违规（政治敏感/宗教极端/色情/暴力/广告）"
  - 置信度>90%正常 → 自动通过发布
  - 置信度<90% 或 疑似违规 → 进入人工审核队列
  ↓
人工审核（管理员在AdminPage操作）：
  - 通过 → 内容发布
  - 拒绝 → 通知用户，说明原因
  - 标记观察 → 内容发布但标记，后续关注
```

### 4.2 敏感话题处理

| 话题类型 | 处理策略 |
|---------|---------|
| 政治敏感 | 关键词拦截 + AI预审 + 人工审核三重过滤。涉及国家领导人、政治事件、领土争议等关键词的内容一律进入人工审核 |
| 宗教极端 | 正常宗教讨论允许，极端主义/邪教内容拦截。关键词：具体邪教名称+传教话术 |
| 色情 | 文本：关键词过滤。图片：MVP阶段不支持图片上传，暂不处理 |
| 暴力/血腥 | 关键词过滤 + AI预审 |
| 广告/引流 | AI预审识别广告话术，自动标记 |

### 4.3 举报处理

| 指标 | 目标 |
|------|------|
| 举报入口 | 每条内容右上角"举报"按钮 |
| 举报类型 | 虚假信息/色情低俗/人身攻击/广告引流/其他 |
| 处理SLA | 被举报内容24小时内处理 |
| 自动隐藏 | 同一内容被举报≥5次 → 自动隐藏，等待人工审核 |
| 处理结果通知 | 举报者收到处理结果通知 |

### 4.4 紧急下线机制

**触发条件**：某个瓜/话题引发舆情风险（被大量举报、被媒体关注、涉及敏感事件）。

**操作流程**：
```
1. 管理员在AdminPage找到目标内容
2. 点击"紧急下线"，填写原因
3. 内容立即从所有列表/详情页消失
4. 关联内容（佐证、评论、社区帖子中的引用）同步隐藏
5. 访问原URL → 返回"该内容已下线"
6. 记录操作日志
7. 如需恢复 → 取消下线 → 内容恢复
```

**时效要求**：从决定下线到内容不可见，< 1分钟。

## 五、增长与留存

### 5.1 冷启动期（0-1000用户）增长策略

| 阶段 | 用户量 | 策略 | 具体动作 |
|------|--------|------|---------|
| 种子期 | 0-50 | 团队+朋友 | 邀请身边大学生注册，收集反馈 |
| 扩散期 | 50-200 | 校园推广 | 在3-5个大学微信群/QQ群分享；在知乎/小红书发"信息求证工具"相关内容 |
| 增长期 | 200-1000 | 内容驱动 | 每日产出1-2个高质量瓜（蹭热点）；AI斗蛐蛐精彩片段截图分享到社交媒体 |

**关键指标**：
- 种子期：日活>30%，每周反馈≥5条
- 扩散期：日活>20%，邀请转化率>10%
- 增长期：日活>15%，自然增长占比>30%

### 5.2 首日留存目标及达成路径

**目标**：首日留存率 > 30%（即100个新用户中30个第二天回来）。

**达成路径**：

| 时间点 | 动作 | 目的 |
|--------|------|------|
| 注册后0分钟 | 新手引导：3步完成第一次猜瓜 | 让用户体验核心玩法 |
| 注册后5分钟 | 推荐一场正在进行的AI斗蛐蛐 | 展示产品特色 |
| 注册后10分钟 | 引导选择兴趣领域（3个以上） | 为后续推送做准备 |
| 首次离开前 | 显示"明日回归奖励：签到得20积分" | 给回来的理由 |
| 次日 | 推送通知（如果授权）："你关注的领域有新瓜" | 拉回用户 |

### 5.3 内容枯竭时的补充策略

| 策略 | 具体做法 | 频率 |
|------|---------|------|
| LLM生成新瓜 | 基于热点话题自动生成瓜（管理员审核后发布） | 每日3-5条 |
| 热点转瓜 | 热点系统中的热点事件一键转为瓜 | 有热点时立即操作 |
| 用户投稿激励 | 投稿瓜被采用 → 奖励50积分 | 长期 |
| 经典瓜回顾 | 将过去的经典瓜重新推荐（"一年前的今天"） | 数据量足够后 |
| 社区UGC | 社区帖子中的高质量讨论可转为瓜 | 每周1-2条 |

### 5.4 积分通胀/通缩调控

**监控指标**：
```
积分健康度 = 全服总积分消耗 / 全服总积分获取
- > 1.0：通缩（消耗>获取），用户积分不够用
- 0.7-1.0：健康
- < 0.7：通胀（获取>消耗），积分贬值
```

**调控手段**：

| 情况 | 手段 |
|------|------|
| 通胀（积分过多） | 增加积分消费场景：下注额度提升、段位晋升费用、虚拟道具（头像框/主题色） |
| 通缩（积分过少） | 增加获取渠道：每日签到、连续登录奖励、邀请好友奖励；降低消费门槛 |
| 严重通胀 | 一次性重置（谨慎使用，需提前通知用户） |
| 严重通缩 | 新用户注册奖励提升、限时双倍积分活动 |

**初始数值**：
```
获取：猜瓜+10，佐证+10，被赞+1，签到+5，邀请+50
消费：下注-10~-50，段位晋升-100~-1000
初始预估：活跃用户日均获取≈50，日均消费≈30（轻微通胀，可接受）
```

## 六、值班与应急响应

### 6.1 上线首周值班安排

| 时段 | 值班内容 |
|------|---------|
| 08:00-24:00 | 至少1人在线，监控AdminPage告警条+用户反馈 |
| 00:00-08:00 | 不值班，但手机保持可联系 |
| 周末 | 轮班制，每天至少1人 |

### 6.2 应急响应流程

```
发现问题（告警/用户反馈/自己发现）
  ↓
判断严重程度：
  P0（核心功能不可用）→ 立即处理，15分钟内响应
  P1（功能异常有绕行）→ 1小时内响应
  P2（体验问题）→ 记录到Issues，下一工作日处理
  ↓
处理：
  - 能hotfix → 立即修复部署
  - 需要回滚 → 回滚到上一版本
  - 需要停机 → 展示维护页面，修复后恢复
```
