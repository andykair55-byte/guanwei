## 一、优先级重新定义

### 1.1 优先级判定标准

| 等级 | 含义 | 判定标准 |
|------|------|---------|
| **P0-必须完成** | hackathon演示的核心闭环，缺了产品不成立 | 用户进来看不到东西 / 无法完成核心交互 |
| **P1-尽量完成** | 提升演示质量，缺了也能demo | 功能可用但体验粗糙 |
| **P2-可以demo** | 有页面能点就行，不要求功能完整 | 展示产品广度，不要求深度 |
| **P3-砍掉** | hackathon前不开发 | 开发量大、收益低 |

### 1.2 模块优先级表

| 模块 | 优先级 | 判定理由 |
|------|--------|---------|
| **工作间**（原AI创作助手合并工具） | **P0-产品活路** | 供给侧引擎，解决"内容从哪来"的根本问题。AI创作为主、工具为辅，进入即见创作界面 |
| **瓜田系统** | P0-必须完成 | 消费侧核心，用户进来的第一件事就是刷瓜。工作间产出的内容在此消费 |
| **AI竞技场**（归入娱乐大厅） | P0-必须完成 | 病毒传播引擎，零参与门槛，不依赖内容供给 |
| **娱乐大厅** | P0-必须完成 | 娱乐模式组合入口，聚合人机对决/多人辩论/AI斗蛐蛐，是消费侧第二引擎 |
| **用户体系** | P0-必须完成 | 段位+积分是留存基础设施，但只做简化版 |
| **佐证系统** | P0-必须完成 | 工作间的下游产出目标之一（创作→发布为佐证） |
| **trae宝**（全局聊天agent） | P0-必须完成 | 右下角全局agent，陪伴为主，提供总结/提取事实/情绪检测等能力，是内容消费侧的AI伴侣 |
| **管理后台** | P0-必须完成 | 已有基础实现，补充运营功能即可 |
| **通知系统** | P0-必须完成 | 赞/收藏/评论/关注的通知中枢，是社区活跃度的基础设施 |
| **私信系统**（Web端先做） | P1-尽量完成 | 两栏布局会话+聊天，移动端暂缓 |
| **热点系统** | P1-尽量完成 | 流量入口，但手动录入即可。时间线构建工具关联此模块 |
| **求证工具集**（精简） | P1-尽量完成 | 仅保留EXIF分析+时间线构建为独立页，LLM求证已嵌入工作间，情绪检测已归入trae宝 |
| **社区系统** | P2-可以demo | 工作间的另一个下游产出目标 |
| **笔记系统** | P2-可以demo | 简单文本编辑器+发布为佐证 |
| **侧边栏自定义** | P1-尽量完成 | 用户在设置页增删导航项，localStorage持久化，提升个性化体验 |
| **LLM配置** | P2-可以demo | 页面已有，保持现状。工作间是其第一个真正全局联动的消费者 |
| **平台架构** | P0-基础设施 | 已稳定，维护即可 |

## 二、MVP范围定义

### 2.1 hackathon演示的核心闭环

```
                    ┌── 消费侧 ──┐
                    │              │
用户进入 → 看到瓜田（有内容） → 猜真假 → 看开奖
                ↑                    ↓
         ┌──────┘              写高质量佐证
         │                      （AI辅助）
    工作间                          │
    （供给侧引擎）                    ↓
    AI创作为主+工具为辅        佐证/帖子沉淀在瓜田
    LLM求证嵌入创作流程              │
    降级为预设模板                  ↓
                            进娱乐大厅 → 看AI斗蛐蛐/人机对决 → 下注
                                         ↓
              trae宝全程陪伴（总结/提取事实/情绪检测）  段位+积分（成就感）
```

**核心叙事**：工作间AI帮你写出有信息密度的内容 → 内容留在社区被消费 → trae宝陪伴消费过程 → 社区有留存 → 更多内容 → 正循环

## 三、跨模块依赖（具体实现方式）

不是抽象的箭头图，而是具体的代码级依赖：

### 3.1 瓜田 → 佐证

```
verificationStore.ts:
  - getMelonDetail(melonId) 返回的 melon.evidences 来自 evidenceStore
  - 实际调用: evidenceStore.getEvidencesByMelonId(melonId)
  - 数据流: evidenceStore.evidences[] → 按 melonId 过滤 → 按 direction 分组
```

### 3.2 瓜田 → 用户

```
verificationStore.ts → userStore.ts:
  - 猜瓜成功后: userStore.addPoints(amount) // 积分+10
  - 猜瓜后更新准确率: userStore.updateAccuracy(correct, total)
  - 段位自动计算: userStore.recalculateRank() // 基于accuracy和totalGuesses
```

### 3.3 瓜田 → 社区

```
MelonDetailPage.tsx:
  - 瓜数据中有 communityId 字段
  - "去社区讨论"按钮 → navigate(`/community/${melon.communityId}`)
  - communityStore.getCommunity(communityId) 获取社区名称和帖子数
```

### 3.4 AI竞技场 → LLM

```
aiBattleService.ts → llmService:
  - 每轮辩论发言: llmService.chat(rolePrompt, context) 
  - 裁判评分: llmService.chat(judgePrompt, debateHistory)
  - 降级: llmService.isAvailable() === false → 显示预设台词
```

### 3.5 笔记 → 佐证

```
NoteDetailPage.tsx → evidenceStore:
  - 发布为佐证: evidenceStore.submitEvidence({ melonId, content: note.content, direction })
  - 佐证来源标记: evidence.source = 'note', evidence.sourceNoteId = note.id
```

### 3.6 管理后台 → 全部模块

```
AdminPage.tsx:
  - 用户管理 → userStore.getUsers() / userStore.banUser(id)
  - 瓜管理 → verificationStore.getMelons() / verificationStore.setMelonResult(id, result)
  - 审核 → contentReviewService.review(contentId, action)
  - 数据看板 → analyticsService.getDashboard()
```

### 3.7 工作间 → 多模块（供给侧引擎）

```
creationService.ts → llmStore.callLLM():
  - 骨架生成: callLLM(skeletonPrompt, { topic, stance, readerType })
  - 质量评分: callLLM(scoringPrompt, { draft, styleProfile })
  - 风格分析: callLLM(stylePrompt, { recentPosts })

creationService.ts → verificationStore:
  - LLM求证嵌入创作流程: verificationStore.verify(claim) → 复用求证工具的LLM文字求证
  - 核查结果内联在编辑器中

creationStore.ts → evidenceStore:
  - 发布为佐证: evidenceStore.submitEvidence({ melonId, content, direction, source: 'ai-assisted' })
  - 佐证来源标记: evidence.aiAssisted = true

creationStore.ts → communityStore (v2):
  - 发布为帖子: communityStore.createPost({ communityId, content, source: 'ai-assisted' })

creationService.ts → userStore:
  - 发布奖励: userStore.addPoints(5) // 创作发布+5（高于普通佐证+3）
  - 高质量奖励: 质量评分>80 → userStore.addPoints(3)

llmStore.callLLM() 重构:
  - aiJudgeService → callLLM()       // 原hardcode Groq
  - aiSummaryService → callLLM()     // 原hardcode Groq
  - aiTranslatorService → callLLM()  // 原hardcode Groq
  - emotionAnalysis → callLLM()      // 原hardcode Groq（情绪检测现已归入trae宝）
  - creationService → callLLM()      // 工作间，直接使用统一入口
  - traeBotService → callLLM()       // trae宝，新增
  - styleProfileService → callLLM()  // 新模块
```

### 3.8 trae宝 → 多模块（全程陪伴agent）

```
traeBotService.ts → llmStore.callLLM():
  - 聊天: callLLM(chatPrompt, { messages, pageContext })
  - 总结: callLLM(summaryPrompt, { pageContent }) → 3-5句话摘要
  - 提取事实: callLLM(factsPrompt, { pageContent }) → 事实列表
  - 情绪检测: callLLM(emotionPrompt, { selectedText }) → 情绪结果
  - 超时降级: 12秒AbortSignal，超时返回"AI暂时走神了，请稍后再试"

usePageContext() → 页面注入:
  - MelonDetailPage 注入瓜详情+佐证上下文
  - CommunityDetailPage 注入帖子内容+评论
  - HotEventDetailPage 注入热点事件+时间线

traeBotStore → localStorage:
  - 会话跨页面保留: messages 数组持久化
  - 切页面不丢失上下文

trae宝 → 工作间:
  - 分析结果纯展示，用户点击"去工作间深入佐证"→ navigate('/agent-world')
  - 工作间和娱乐大厅不显示trae宝
```

### 3.9 娱乐大厅 → AI竞技场+用户体系

```
娱乐大厅路由 /entertainment:
  - 人机对决: 用户 vs AI辩手 → callLLM()
  - AI斗蛐蛐: 复用现有AIBattle/CricketForge → callLLM()
  - 多人辩论: 标记"即将开放"，需WebSocket

娱乐大厅 → userStore:
  - 下注扣积分 → 辩论结束结算
  - 获胜+15积分（每日5次上限）
```

### 3.10 工作间 → LLM 调用链路（v2路径）

```
Phase 1（MVP）: 前端直调LLM
  creationService.ts → llmStore.callLLM() → Groq API

Phase 2（统一入口重构）:
  creationService.ts → llmStore.callLLM()
    → 统一读取用户配置（LLMSettingsPage 的 provider/key）
    → 12秒超时降级 + mock fallback
    → 草稿持久化走 localStorage（跨会话保留）
```

> 注：原 "Phase 2 计划接入 Mirror 引擎" 的路径已废弃，engine/Mirror 目录已于 2026-07-16 删除。工作间统一走 callLLM() 入口，不再依赖独立 Agent 后端基座。
