## 二、内容审核SOP

### 2.1 审核流程

```
UGC内容提交（瓜/佐证/帖子/评论/笔记）
  ↓
第一层：前端关键词预检（敏感词库匹配）
  → 命中 → 拦截提交，提示"内容包含敏感词"
  → 未命中 → 进入第二层
  ↓
第二层：后端AI预审（LLM判断是否违规）
  → 置信度>90%正常 → 自动通过
  → 置信度<90%或命中疑似违规 → 进入人工审核队列
  ↓
第三层：人工审核（管理员在AdminPage操作）
  → 通过 / 拒绝（填写原因） / 标记待观察
```

### 2.2 审核队列设计

在AdminPage中新增"审核中心"Tab：

| 字段 | 说明 |
|------|------|
| 内容类型 | 瓜/佐证/帖子/评论 |
| 提交人 | 用户昵称+ID |
| 提交时间 | ISO时间 |
| 内容摘要 | 前100字 |
| AI预审结果 | 通过/疑似违规+原因 |
| 操作 | 通过/拒绝/标记待观察 |

### 2.3 审核时效

| 优先级 | 场景 | 处理时限 |
|--------|------|---------|
| 紧急 | 被举报的内容 | 2小时内 |
| 高 | 涉及政治/宗教/色情的疑似违规 | 4小时内 |
| 普通 | 新提交的UGC内容 | 24小时内 |

## 三、异常告警机制

### 3.1 告警指标

| 指标 | 阈值 | 告警方式 |
|------|------|---------|
| API响应时间 | P95 > 3s | 控制台日志+管理员通知 |
| API错误率 | 5xx > 5% | 控制台日志+管理员通知 |
| LLM调用失败率 | > 10% | 控制台日志+管理员通知 |
| LLM调用量 | 日调用 > 800次（限额900） | 管理员通知 |
| 数据库连接失败 | 任意一次 | 控制台日志+管理员通知 |
| 前端JS错误率 | > 2% | Sentry告警 |

### 3.3 管理员通知

## 四、数据备份策略

### 4.1 SQLite阶段（当前）

```bash
# 每日凌晨自动备份
# 脚本: scripts/backup.sh
cp data/jianwei.db data/backups/jianwei_$(date +%Y%m%d).db

# 保留最近30天的备份
find data/backups/ -name "*.db" -mtime +30 -delete
```

### 4.2 PostgreSQL阶段（迁移后）

- 开启WAL归档（持续备份）
- 每日全量备份（pg_dump）
- 保留7天全量备份 + 30天WAL日志
- 备份存储：本地 + 对象存储（阿里云OSS/腾讯云COS）

### 4.3 恢复流程

```
1. 确认需要恢复的时间点
2. 停止服务
3. 恢复最近的全量备份
4. 应用WAL日志到目标时间点
5. 验证数据完整性
6. 重启服务
```

## 六、紧急预案

### 6.1 LLM服务中断降级

```typescript
// 全局降级开关（AdminPage → 系统设置）
const llmFallbackConfig = {
  // 开关：是否启用降级模式
  fallbackMode: false,

  // 降级策略
  strategies: {
    // AI竞技场：停止新对战，已有对战显示"AI思考中..."
    aiArena: 'pause_with_message',
    // 求证工具：显示"AI分析暂时不可用，请稍后重试"
    verifyTools: 'show_unavailable',
    // 内容审核：切换到纯关键词匹配模式
    contentReview: 'keyword_only',
    // 种子帖子生成：暂停生成
    seedGeneration: 'pause',
  },

  // 恢复策略
  recovery: {
    // 每5分钟检测LLM服务可用性
    checkInterval: 300_000,
    // 连续3次检测成功 → 自动恢复
    autoRecoverAfter: 3,
  }
}
```

### 6.2 内容审核紧急模式

当审核人员不在线、但出现大量举报时：

```
1. 管理员在AdminPage开启"紧急审核模式"
2. 所有新UGC内容自动进入"待审核"状态（不对外展示）
3. 已发布的内容不受影响（除非被举报）
4. 被举报内容自动隐藏，等待人工审核
5. 关闭紧急模式后，待审核内容批量释放
```

### 6.3 紧急下线机制

当某个瓜引发舆情风险时：

```
1. 管理员在AdminPage → 瓜管理 → 找到目标瓜
2. 点击"紧急下线"
3. 瓜详情页返回404，提示"该内容已下线"
4. 关联的佐证、评论、社区帖子中的引用同步隐藏
5. 记录操作日志（操作人、时间、原因）
6. 如需恢复：取消下线 → 内容恢复
```

### 7.2 需要关注的技术债

| 问题 | 影响 | 处理建议 |
|------|------|---------|
| mockData.ts 数据硬编码 | 后端就绪后需要全部替换 | 保持service层抽象，替换时只改service实现 |
| 多个Store用内存状态 | 刷新丢失数据 | 后端就绪前可接受，后端就绪后切换为API驱动 |
| Tailwind v4 @theme 自定义 | 与v3语法不完全兼容 | 不升级不改动，保持现状 |
| 前端无错误监控 | 线上JS错误不可见 | hackathon后用Sentry接入 |
