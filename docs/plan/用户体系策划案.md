## 一、模块定位

**用户体系是观微的身份与积分中枢——用7级段位证明判断力、用积分驱动行为（猜瓜/下注/佐证）、用基础个人主页展示战绩。**

解决的问题：
1. 用户"是谁"——段位代表信息判断力的可信度
2. 行为激励——积分获取和消费形成闭环
3. 竞争驱动——排行榜+段位晋升制造持续参与动力

给谁用：所有观微注册用户。

### 4.2 段位晋升方案：准确率方案（推荐）

**决策：使用准确率方案，不用经验值方案。**

理由：
1. 准确率直接反映"信息判断力"，与产品定位"提升信息素质"一致
2. 经验值方案会导致"刷量升级"——用户为了升级而猜瓜，不在乎对错
3. 准确率方案天然防刷——猜错不涨猜对数，也不会提高准确率
4. 对测试用户来说，准确率排名更有说服力（"我准确率72%"比"我2000经验"更有意义）

晋升规则：
```
同时满足以下三个条件即晋升：
1. 猜对次数 >= 下一段位minCorrect
2. 总猜瓜次数 >= 下一段位minTotal
3. 准确率(猜对/总次数) >= 下一段位minAccuracy

段位只升不降（MVP不做保级）
```

### 4.3 段位计算实现

```typescript
// stores/userStore.ts 中的段位计算逻辑（已实现，确认正确）
function calculateRank(correctGuesses: number, totalGuesses: number): Rank {
  if (totalGuesses === 0) return '吃瓜群众'
  const accuracy = correctGuesses / totalGuesses

  // 从高到低遍历，返回第一个满足条件的段位
  for (const rank of [...RANK_ORDER].reverse()) {
    const config = RANK_CONFIG[rank]
    if (
      correctGuesses >= config.minCorrect &&
      totalGuesses >= config.minTotal &&
      accuracy >= config.minAccuracy / 100  // 配置中是百分比
    ) {
      return rank
    }
  }
  return '吃瓜群众'
}
```

| 邀请新用户 | +20 | 3次/天 | 被邀请人完成首次猜瓜后发放 |

### 5.2 积分消费

| 消费项 | 积分 | 说明 |
|--------|------|------|
| AI竞技场下注 | 10-100 | 与AI竞技场联动 |
| 修改猜瓜 | -1 | 已猜的瓜可改答案（每日限3次） |

MVP阶段积分消费场景只有2个，足够形成闭环：
- 赚积分 → 猜瓜/佐证 → 更多积分
- 赚积分 → 下注AI竞技场 → 赢取更多积分 or 输掉

### 5.3 积分数值模拟

#### 假设条件

```
测试用户画像：大学生，每天使用观微 20-30 分钟
日均行为：
- 猜瓜：8次/天（其中猜对5次，猜错3次，准确率62.5%）
- 提交佐证：2条/天
- 看1场AI对战 + 下注1次
- 每天登录1次
```

#### 单用户月度积分收支模拟

```
=== 月度收入 ===
每日登录：5 × 30 = 150
猜瓜参与：2 × 8 × 30 = 480（每天8次，每次2分）
猜对奖励：5 × 5 × 30 = 750（每天对5次，每次5分）
提交佐证：3 × 2 × 30 = 180（每天2条，每次3分）
佐证被选最佳：30 × 2 = 60（每月约2次被选为最佳）
AI竞技场获胜：15 × 3 = 45（每月约3次下注获胜）
邀请新用户：20 × 1 = 20（每月约邀请1人）
---
月收入合计：2,685 积分

=== 月度支出 ===
AI竞技场下注：30 × 20 = 600（每月约20次下注，每次平均30积分）
修改猜瓜：1 × 5 = 5（每月约5次修改）
---
月支出合计：605 积分

=== 月度净收入 ===
净收入 = 2,685 - 605 = 2,080 积分
```

#### 3个月积分余额预测

```
初始余额：0（新用户注册）

第1个月末：
  余额 = 0 + 2,080 = 2,080
  段位：吃瓜群众 → 瓜田新手（猜对150次，准确率62.5%）

第2个月末：
  余额 = 2,080 + 2,080 = 4,160
  段位：瓜田新手 → 鉴瓜学徒（猜对300次，准确率62.5%）

第3个月末：
  余额 = 4,160 + 2,080 = 6,240
  段位：鉴瓜学徒 → 瓜田侦探（猜对450次，准确率62.5%）
```

#### 积分通胀风险评估

```
关键指标：积分获取/消费比 = 2,685 / 605 = 4.44

评估：
- 积分净增速 2,080/月，3个月后余额 6,240
- 如果开放更多消费场景（头像框200、主题色150），
  可以消耗掉多余积分
- 当前通胀风险：中等偏高（赚多花少）

对策（v2）：
1. 增加积分消费场景（头像框、主题色、改名卡等）
2. 引入"赛季"机制，每赛季积分清零重算
3. 高段位用户积分获取倍率降低（如L5以上打8折）
4. 下注赔率动态调整，增加不确定性

MVP阶段不处理通胀，因为：
- 用户基数小（500-1000人），积分无真实价值
- 积分消费场景少是有意为之，避免过度游戏化
- 先观察数据，再决定v2的调控手段
```

#### 1000人规模下的系统积分估算

```
假设1000人测试，30%日活（300 DAU），运营3个月：

总积分发行量：300人 × 2,685积分/月 × 3月 = 2,416,500
总积分消耗量：300人 × 605积分/月 × 3月 = 544,500
净积分存量：1,872,000

这个量级下积分没有任何经济压力（不是真实货币），
无需担心"破产"。积分只是数字，核心是让用户有成就感。
```

### 6.2 每日积分上限实现

// stores/userStore.ts 中增加每日上限检查

const DAILY_POINTS_CAP = 150  // 每日积分获取上限

function canEarnPoints(userId: string, amount: number): { allowed: boolean; adjusted: number } {
  const todayEarned = getTodayEarned(userId)
  const remaining = DAILY_POINTS_CAP - todayEarned

  if (remaining <= 0) {
    return { allowed: false, adjusted: 0 }
  }

  if (todayEarned + amount > DAILY_POINTS_CAP) {
    return { allowed: true, adjusted: remaining }  // 只给到上限
  }

  return { allowed: true, adjusted: amount }
}

### 6.3 行为异常检测

```typescript
// services/anti_farm.ts

interface BehaviorLog {
  timestamp: number
  action: 'guess' | 'evidence' | 'bet' | 'login'
  content?: string  // 佐证内容（用于重复检测）
}

class AntiFarmDetector {
  private logs: BehaviorLog[] = []

  // 检测快速连点
  checkRapidAction(userId: string, action: string): boolean {
    const recent = this.logs.filter(l =>
      l.action === action && Date.now() - l.timestamp < 60000
    )
    if (action === 'guess' && recent.length > 5) return true  // 1分钟>5次
    if (action === 'evidence' && recent.length > 3) return true  // 1分钟>3条
    return false
  }

  // 检测机器人行为（间隔过短且规律）
  checkBotPattern(): boolean {
    const recentGuesses = this.logs
      .filter(l => l.action === 'guess')
      .slice(-10)

    if (recentGuesses.length < 10) return false

    const intervals = []
    for (let i = 1; i < recentGuesses.length; i++) {
      intervals.push(recentGuesses[i].timestamp - recentGuesses[i-1].timestamp)
    }

    // 如果所有间隔都<3秒，判定为机器人
    if (intervals.every(i => i < 3000)) return true

    // 如果间隔标准差<500ms（过于规律），判定为脚本
    const avg = intervals.reduce((a, b) => a + b, 0) / intervals.length
    const variance = intervals.reduce((a, b) => a + Math.pow(b - avg, 2), 0) / intervals.length
    if (Math.sqrt(variance) < 500) return true

    return false
  }

  // 检测佐证重复
  checkEvidenceDuplicate(content: string): boolean {
    const recentEvidence = this.logs
      .filter(l => l.action === 'evidence' && l.content)
      .slice(-5)

    for (const e of recentEvidence) {
      const similarity = calculateSimilarity(content, e.content!)
      if (similarity > 0.8) return true  // 80%相似
    }
    return false
  }
}

// 简单的字符串相似度计算（Jaccard similarity on character bigrams）
function calculateSimilarity(a: string, b: string): number {
  const bigramsA = new Set<string>()
  const bigramsB = new Set<string>()

  for (let i = 0; i < a.length - 1; i++) bigramsA.add(a.slice(i, i + 2))
  for (let i = 0; i < b.length - 1; i++) bigramsB.add(b.slice(i, i + 2))

  const intersection = [...bigramsA].filter(b => bigramsB.has(b))
  const union = new Set([...bigramsA, ...bigramsB])

  return intersection.length / union.size
}
```

### 6.4 设备指纹

```typescript
// 纯前端设备指纹（不依赖第三方库）
function generateDeviceFingerprint(): string {
  const components = [
    navigator.userAgent,
    screen.width + 'x' + screen.height,
    Intl.DateTimeFormat().resolvedOptions().timeZone,
    navigator.language,
    navigator.hardwareConcurrency?.toString() || 'unknown',
    // 不使用canvas指纹（太侵入式）
  ]

  // 简单hash
  const raw = components.join('|')
  let hash = 0
  for (let i = 0; i < raw.length; i++) {
    const char = raw.charCodeAt(i)
    hash = ((hash << 5) - hash) + char
    hash = hash & hash  // Convert to 32bit integer
  }
  return 'jw_' + Math.abs(hash).toString(36)
}

// 存储到localStorage
function saveDeviceFingerprint() {
  const fp = generateDeviceFingerprint()
  const existing = localStorage.getItem('jw_device_fp')
  if (!existing) {
    localStorage.setItem('jw_device_fp', fp)
  }
  // 注册时发送device_fingerprint到后端
  return fp
}
```

### 6.5 封禁处理

```
封禁触发条件：
1. 被AntiFarmDetector标记为机器人 → 自动封禁7天
2. 同设备>3个账号 → 封禁多余账号
3. 管理员手动封禁（AdminPage）

封禁效果：
1. 用户无法进行任何积分操作（猜瓜/佐证/下注）
2. 已冻结的积分退还
3. 段位冻结（不降级，但无法晋升）
4. 个人页显示"账号已限制"提示

封禁后的残留会话处理：
1. 已登录的session立即失效（前端检测isBanned后强制登出）
2. 进行中的AI竞技场对战中断，积分退还
3. 已提交但未开奖的猜瓜保留，开奖后积分不发放（捐入系统池）
4. 已发布的佐证保留但标记"该用户已限制"
```

### 9.2 积分变动流程

```
用户操作 → 前端校验 → 后端校验 → 积分变动 → 前端更新

详细流程：
1. 用户触发操作（如猜瓜）
2. 前端校验：
   - 是否已登录
   - 是否被封禁
   - 是否达到每日上限
   - 积分余额是否足够（消费场景）
3. 后端校验（重复前端校验 + 服务端权威校验）：
   - 幂等检查（防止重复提交）
   - 每日上限（服务端为准）
   - 防刷检测
4. 积分变动：
   - 数据库事务：User.points += amount
   - 写入积分流水：PointsRecord
   - 更新每日统计：DailyStats
5. 前端更新：
   - 更新userStore中的points
   - 显示积分变动toast（+5 / -30）
```

## 十、异常场景应对

### 10.1 积分并发操作

```
场景：用户快速连续点击"猜瓜"按钮，导致多次提交
处理：
1. 前端防抖：点击后按钮禁用3秒（disabled状态）
2. 幂等key：每次猜瓜生成唯一guessId = userId + melonId + timestamp
3. 后端去重：guessId唯一约束，重复提交返回已有结果
4. 积分原子操作：数据库层面用 UPDATE ... WHERE 保证原子性
5. 乐观锁：更新时检查版本号，冲突则重试
```

### 10.2 段位数据不一致

```
场景：前端计算的段位与后端不一致
处理：
1. 以服务端为准：每次API请求返回最新段位
2. 前端延迟更新：猜瓜后不立即重新计算段位，等API返回
3. 定期同步：每5分钟调用 GET /api/users/me 同步最新数据
4. 冲突处理：如果前端段位 > 后端段位，以后端为准（防止前端篡改）
5. 展示策略：段位变化时显示"段位更新中..."过渡状态
```

### 10.3 被封禁后的残留会话

```
场景：用户在A设备被封禁，但B设备仍在使用
处理：
1. 心跳检测：前端每60秒调用 GET /api/users/me/status
2. 检测到封禁：立即弹出提示"你的账号已被限制"
3. 强制登出：清除本地token，跳转登录页
4. 进行中的操作：
   - AI竞技场对战中断，积分退还
   - 未提交的表单数据保存到localStorage
   - 已提交但未处理的积分操作回滚
```

### 10.4 每日上限跨时区

```
场景：用户在UTC+8的23:59猜瓜，UTC+0的00:00又猜瓜，是否算新的一天
处理：
1. 统一使用服务器时间（UTC+8）
2. 每日上限按服务器日期的00:00重置
3. 前端显示"距离上限重置还有X小时"
4. 不做客户端时区转换（避免用户改系统时间刷积分）
```

### 10.5 积分余额为负

```
场景：下注扣积分时，并发操作导致余额变为负数
处理：
1. 数据库约束：points字段 CHECK (points >= 0)
2. 扣费前检查：if (points < amount) throw Error('积分不足')
3. 事务隔离：下注操作在事务中执行，先冻结积分再确认
4. 前端同步：扣费失败时回滚UI，提示"积分不足"
```

## 十一、上线后运营

### 11.1 段位分布监控

```
目标分布（500-1000人测试3个月后）：
- L1 吃瓜群众：10-15%（不活跃用户）
- L2 瓜田新手：25-30%（轻度用户）
- L3 鉴瓜学徒：30-35%（中度用户）
- L4 瓜田侦探：15-20%（活跃用户）
- L5 鉴瓜达人：5-10%（核心用户）
- L6 鉴瓜大师：1-3%（超级用户）
- L7 见微先知：<1%（极少数）

如果L1占比>50%：说明晋升太难或引导不够
如果L5+占比>20%：说明阈值太低，需要调高
```

### 11.2 积分健康度指标

```
监控指标：
1. 积分基尼系数：衡量积分分布是否过于集中
   - 目标：0.3-0.5（适度不平等）
   - >0.7说明少数人垄断，需增加新手积分获取

2. 日均积分发行量/消耗量比：
   - 目标：1.5-2.5（略大于1，让用户有积累感）
   - >4说明通胀严重，需增加消费场景
   - <1说明积分不够花，需增加获取渠道

3. 积分中位数：
   - 目标：500-2000（3个月后）
   - 太低说明用户不活跃
   - 太高说明通胀
```

### 11.3 新用户引导

```
新用户首次登录的流程：
1. 注册/登录 → 自动获得"吃瓜群众"段位 + 50积分（新手礼包）
2. 引导猜第一个瓜：
   - 高亮瓜田列表中的第一个瓜
   - "猜一下这个瓜是真是假？猜对可以获得积分哦！"
   - 首次猜瓜额外奖励10积分
3. 引导看一场AI对战：
   - "等开奖无聊？去看看AI斗蛐蛐！"
   - 首次观看奖励5积分
4. 完成引导后：段位仍为L1，但已有65积分
```
