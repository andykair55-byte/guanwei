# 观微「工作间」策划案

> 版本：v2.0
> 状态：产品策划 + 技术规格
> 优先级：P0（产品活路）
> 最后更新：2026-07-11
> v2.0更新：重新定位为"信息组织工作台"，新增碎片收集/信息全景/反馈闭环设计

---

## 一、模块定位

### 1.1 一句话定义

**不是帮用户写文章，而是帮用户完成从"碎片化信息"到"结构化观点"的全过程。**

### 1.2 核心哲学

工作间与市面上所有AI写作工具的本质区别：

| | 通义千问/Notion AI/ChatGPT写作 | 观微工作间 |
|---|---|---|
| 起点 | 一个 prompt | 一堆碎片（链接/截图/引文/随手记/上下文） |
| AI角色 | 替你写 | 帮你整理、帮你发现关联 |
| 用户角色 | 提需求的人 | 真正在思考的人 |
| 过程价值 | 无（用户只看结果） | 高（用户完成了一次信息梳理） |
| 输出形态 | 一篇"文章" | 一个有信息密度的"观点" |

### 1.3 AI 的角色边界

市面上的 AI 产品都在走"省时间"——替用户工作，用完就走。观微的逻辑不是这个。

用户辩论输了赌气去查资料——这不是"省时间"，这是"咽不下这口气"。自发式的，有情感驱动的。

**工作间的设计基调：**

- **不是"省时间"**，而是**"让想做的事变得可行"**——用户本来就想分析这个瓜，只是缺工具。工作间不是替代他分析，是给他工具
- **不是"替用户工作"**，而是**"让用户能做他本来做不了的事"**——结论始终是用户自己得出的
- **入口是情绪驱动的**——"我不服" / "我想搞清楚" / "这个太有意思了" → 打开工作间

### 1.4 解决的问题

1. **起步难**——面对空白页面不知道从哪开始。工作间的起点不是空白页，而是已经收集的碎片和上下文
2. **整理难**——刷了一堆信息，知道有话想说但理不清楚。工作间帮用户识别关联、发现矛盾、找到缺失角度
3. **核查难**——写出来的内容不知道靠不靠谱。核查台一键核查，结果内联展示
4. **反馈断**——发完就没了，不知道有没有人看、有没有人受影响。发布后展示影响力数据

### 1.5 给谁用

- **核心用户**：大学生，在瓜田/社区看到热点，想分析清楚但不知道从哪下手
- **延伸用户**：辩论输了赌气想找证据反击的人；看到一篇文章不同意想反驳的人
- **不是给谁用的**：只想看不想写的纯消费者（他们用trae宝消费内容，不进工作间）

## 二、产品哲学：为什么工作间长这样

### 2.1 自发式 > 任务式

用户打开工作间的动机不是"我要写一篇文章"，而是：

| 情绪 | 场景 |
|------|------|
| **不服** | 辩论输了 / 看到一个观点觉得不对但说不出哪里不对 |
| **好奇** | 看到一个瓜，想知道更多 |
| **分享欲** | 发现一个很有意思的角度，想让更多人看到 |
| **求证欲** | 看到一条消息，怀疑是假的 / 想确认是不是真的 |
| **记录** | 看到一段有用的信息，怕忘了 |

### 2.2 设计原则

1. **永远不要求用户"一次完成"**——写了一半关掉，下次打开还在原地。收集了 10 条碎片一条都没用上，没问题
2. **AI 建议是被动的，不是主动打扰的**——不在用户不注意的时候弹窗。放在侧边栏，用户想用的时候自己点
3. **收集箱本身就有价值**——不一定非要"产出"。存了 50 条碎片一条没发布，这也是一种使用方式
4. **情绪出口用情绪文案，功能按钮用中性文案**——"这个瓜不对劲？来分析一下" 替代 "写佐证"

## 三、MVP 范围定义

### 3.1 先打通一条完整链路

不一口气建完整个工作间。v1 只做一条闭环链路：

```
瓜田详情页 → "来分析一下" → 自动带入瓜上下文
  → 写作板（左：写作区 / 右：参考面板）
    → 引用碎片 → 写分析
      → 发布佐证
        → 看到"你的佐证影响了 X 人的判断"
```

这条链覆盖了：触发 → 写作 → 引用 → 发布 → 反馈。每个环节都能用，用户能走通。

### 3.2 v1 功能清单

| 功能 | 状态 | 说明 |
|------|------|------|
| 情绪化入口 | **新增** | 瓜田详情页"这个瓜有蹊跷？来分析一下" |
| 上下文自动带入 | **新增** | 从瓜田进入时自动携带瓜标题+描述+已有佐证 |
| 写作板（左右分栏） | **改造** | 替代当前单页编辑器。左侧写作区，右侧参考面板 |
| AI骨架生成 | **已有，改造** | 基于主题生成骨架（v1不要求基于碎片） |
| 碎片引用 | **新增** | 从右侧参考面板点击引用，嵌入写作区 |
| 收集箱（基础版） | **新增** | 支持粘贴链接/文本手动收集，但v1靠上下文自动填充为主 |
| 核查台（基础版） | **新增** | 标记待核查声明→一键核查→结果内联展示 |
| 发布佐证 | **已有，改造** | 发布后展示影响力数据 |
| 草稿持久化 | **已有** | localStorage自动保存 |
| LLM降级 | **已有** | 3个预设模板 |

### 3.3 v1 不做

- 信息全景（竞品分析agent）— v2
- 独立的收集箱页面 — v1只作为写作板的右侧面板
- 拖拽交互 — v1用点击引用替代
- 系统分享收集 — v1仅粘贴收集
- 多草稿管理 — v1只保留当前草稿

### 3.4 降级方案

LLM不可用时降级为3个预设模板（保留v1.0设计）：

| 模板名 | 结构 |
|--------|------|
| 事件分析 | 事件概述→关键事实→影响分析→各方观点→总结 |
| 观点论述 | 观点陈述→论据1→论据2→反驳预判→呼吁 |
| 对比评测 | 对比对象A→对比对象B→维度1对比→维度2对比→结论 |

降级触发条件：
- callLLM() 超时（12秒）
- LLM provider 未配置或 API key 无效
- 返回格式错误重试1次仍失败

## 四、工作流设计

### 4.1 完整蓝图（v1+v2 总览）

```
触发（情绪驱动）
  ↓
[收集箱] ← 手动收藏 / 上下文关联 / AI采集 / 外部导入
  ↓
[信息全景] ← 竞品分析agent（v2）
  ↓
[写作板] ← 左侧写作区 + 右侧参考面板
  ↓
[核查台] ← 声明标记 → 一键核查
  ↓
[发布 + 反馈] ← 影响力数据
```

### 4.2 v1 实现的工作流

```
Stage 0：触发
  用户从瓜田详情页点击"这个瓜有蹊跷？来分析一下"
  → 自动携带：瓜标题、瓜描述、已有佐证（3-5条）
  → 进入写作板

Stage 1：写作板
  左侧：写作区（标题 + 分段写作）
  右侧：参考面板
    ├─ 📦 上下文素材（自动带入的瓜信息 + 已有佐证）
    ├─ 📋 手动收集（用户粘贴的链接/文本）
    └─ 🤖 AI 建议（骨架建议 / 核查提醒）

Stage 2：写作
  用户选择是否使用AI骨架生成
  写的过程中：
    - 从参考面板点击碎片 → 自动以引用块形式嵌入
    - AI检测声明性语句 → 标记⚠️待核查
    - 点击核查 → 结果内联展示

Stage 3：发布
  选择发布为瓜田佐证
  → 自动关联到当前瓜
  → 发布成功

Stage 4：反馈
  在个人中心→我的创作 看到：
    - 阅读量
    - 点赞数
    - "因你的佐证改变了判断的人数"
```

### 4.3 碎片的数据结构

```typescript
interface Fragment {
  id: string
  type: 'link' | 'quote' | 'image' | 'note'
  content: string
  source?: string            // 来源 URL
  sourceTitle?: string       // 来源标题
  author?: string            // 原始作者
  entryPoint: 'manual' | 'context' | 'ai_scan' | 'external'
  // ^ 标记这条碎片是怎么进来的
  contextOrigin?: string     // 从其他模块带来的，记录来源
  // 如：'melon:xxx' / 'debate:xxx'
  tags: string[]
  collectedAt: number
}
```

### 5.2 核心数据流（v1）

```
用户从瓜田详情页点击"来分析一下"
  ↓
/agent-world?melonId=xxx
  ↓
AgentWorldPage 读取 melonId → 获取瓜信息
  ↓
自动写入 fragmentStore：瓜标题 + 瓜描述 + 已有佐证作为初始碎片
  ↓
写作板渲染：左侧空白写作区，右侧参考面板展示碎片
  ↓
用户操作流程：
  1. [可选] 点击"生成骨架" → creationService → callLLM → 骨架显示在写作区
  2. 点击碎片 → 以引用块嵌入写作区
  3. 写分析内容
  4. 声明被自动标记 → 点击核查
  5. 确认 → 发布佐证
  6. 跳转到瓜田详情页，展示影响力
```

## 六、发布后的反馈闭环

### 6.1 影响力数据

用户发布佐证后能看到：

```
你的佐证
┌──────────────────────────────────────┐
│  📊 28 人阅读                        │
│  👍 5 人点赞                         │
│  💬 3 条评论                         │
│  🎯 7 人因你的佐证改变了判断          │
│                                      │
│  [查看原文] [分享]                    │
└──────────────────────────────────────┘
```

"改变了判断"是指：用户在同一个瓜上先投了A方向，看了你的佐证后改投B方向。
这个指标是工作间特有的反馈——不是"阅读量"，而是"你真的影响了别人"。

### 6.2 存储在 userStore 中

```typescript
// 在 userStore 中扩展
interface UserCreation {
  id: string
  title: string
  type: 'evidence' | 'post'
  publishedAt: number
  views: number
  likes: number
  impactCount: number    // 改变了多少人的判断
  melonId?: string       // 关联的瓜（如果是佐证）
}
```

## 七、实现路径

### Phase 1：核心链路打通（本次实现）

**目标**：从瓜田进入工作间 → 写作板 → 发布佐证 → 看到反馈。完整闭环可走通。

| 任务 | 交付物 | 前置依赖 |
|------|--------|---------|
| WritingBoard 左右分栏布局 | 写作区 + 参考面板两个区域 | 无 |
| ReferencePanel 展示碎片 | 显示上下文素材，支持点击引用 | fragmentStore |
| fragmentStore 创建 | 碎片收集 store，persist | 无 |
| creationStore 扩展 | 新增 fragments 字段 | 无 |
| AgentWorldPage 改造 | 接入写作板 + 参考面板 | 以上全部 |
| 瓜田入口改造 | "这个瓜有蹊跷？来分析一下" | MelonDetailPage |
| 骨架生成适配写作板 | 骨架展示在写作区 | creationService |
| ClaimVerifier 适配 | 声明标记 + 一键核查 | 无 |
| 发布佐证 + 反馈数据 | 发布后展示影响力 | userStore 扩展 |

### Phase 2：完善体验（v2）

- 信息全景（竞品分析agent）
- 独立的收集箱页面
- 多草稿管理
- 从辩论房/社区进入工作间
- 系统分享收集

## 八、与其他模块的联动

```
工作间 → 瓜田系统
  v1：瓜田详情页"来分析一下" → 自动带入上下文
  v1：发布为瓜田佐证
  v1：佐证的影响力数据在瓜田详情页展示

工作间 → 用户体系
  v1：发布佐证 +5 积分
  v1：佐证被点赞 +2 积分/赞
  v1：连续发布 7 天 → 成就徽章

工作间 → LLM配置
  通过 llmStore.callLLM() 统一入口

工作间 → trae宝
  trae宝分析结果可引导用户"去工作间深入分析"
  工作间内隐藏 trae宝（页面上下文已被工作间占用）
```

---

> 工作间是观微供给侧的核心引擎。v1 不做完美，只打通一条链路：
> **瓜田 → 工作间 → 佐证 → 反馈**。
> 等这条路走通了，再把收集箱、全景agent、多入口逐步加上。
> 心急吃不了热豆腐。
