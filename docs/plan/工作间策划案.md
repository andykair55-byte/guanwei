# 观微「工作间」策划案

> 版本：v1.0
> 状态：产品策划 + 技术规格
> 优先级：P0（产品活路）
> 最后更新：2026-07
> 架构审查后新增：由原 AI创作助手 + 求证工具集 合并重组而成

---

## 一、模块定位

**工作间是观微的供给侧引擎——以AI创作为主、工具为辅的工作台。用户进入工作间直接看到创作界面，AI辅助生成骨架、嵌入LLM求证、选择发布目标。求证工具按归属重组：LLM文字求证嵌入创作流程，EXIF分析和时间线构建保持独立，情绪检测归入trae宝，其余复杂工具砍掉。**

解决的问题：
1. **起步难**——面对空白页面不知道怎么写，AI给骨架让用户填肉
2. **质量低**——写出来的东西缺乏论据支撑，LLM求证嵌入创作流程实时验证关键声明
3. **门槛高**——AI降低从"想法"到"可发布内容"的摩擦
4. **工具散乱**——原求证工具集分散，现按归属重组，创作+求证一体化

给谁用：
- **核心用户**：想在瓜田/社区发高质量内容的大学生（有想法但缺写作经验）
- **延伸用户**：校园自媒体/博主（有内容产出需求但缺时间做调研和核查）
- **不是给谁用的**：只想看不想写的纯消费者（他们用trae宝消费内容，不进工作间）

**路由与布局**：
- 路由：`/agent-world`（原 `/create` 已废弃）
- platform：web
- layout：带侧边栏（从 standalone 改为 layout）
- 顶部"发布"按钮改为搜索栏，发布入口收敛到工作间内部
- 用户进入工作间直接看到创作界面

## 二、MVP 范围定义

### 2.1 核心功能

| 功能 | 状态 | 说明 |
|------|------|------|
| 结构化编辑器 | **新增** | 标题+立场输入→骨架展示→逐段填充，非空白textarea |
| AI骨架生成 | **新增** | 输入主题+立场+目标读者→调用callLLM→生成骨架JSON |
| LLM求证嵌入 | **新增** | 检测声明→标记⚠️→一键核查→结果内联展示 |
| 发布目标选择器 | **新增** | 瓜田佐证 / 社区帖子 |
| 草稿持久化 | **新增** | localStorage，每3秒自动保存 |
| LLM降级 | **新增** | LLM不可用降级为3个预设模板 |

### 2.2 工具归属重组

| 工具 | 归属 | 说明 |
|------|------|------|
| LLM文字求证 | **嵌入工作间** | 声明核查复用此能力，结果内联在编辑器 |
| EXIF分析 | **保持独立页** `/tools/exif` | 图片元数据分析，不嵌入创作流程 |
| 时间线构建 | **保持独立页** `/tools/timeline` | 关联热点系统 |
| 情绪检测 | **归入trae宝** | 选中文本→分析→展示结果，路由移除 |
| 反向搜图 | **已砍掉** | 需图片搜索引擎，MVP不可能实现 |
| 洗稿检测 | **已砍掉** | 需embedding模型，MVP不可能实现 |
| 多源验证 | **已砍掉** | 需爬取+提取+比对，复杂度极高 |

### 2.3 降级方案

LLM不可用时降级为3个预设模板：

| 模板名 | 结构 | 适用场景 |
|--------|------|---------|
| 事件分析 | 事件概述→关键事实→影响分析→各方观点→总结 | 瓜田佐证，分析某个事件真假 |
| 观点论述 | 观点陈述→论据1→论据2→反驳预判→呼吁 | 社区帖子，表达立场 |
| 对比评测 | 对比对象A→对比对象B→维度1对比→维度2对比→结论 | 多个对象横向对比 |

降级触发条件：
- callLLM() 超时（12秒）
- LLM provider 未配置或API key无效
- 返回格式错误重试1次仍失败

## 三、技术架构

### 3.1 前端文件结构

```
src/
  pages/
    AgentWorldPage.tsx               -- 工作间主页面（原CreatePage.tsx，路由 /agent-world）
  components/
    create/
      StructuredEditor.tsx           -- 结构化编辑器
      AISkeletonPanel.tsx            -- AI骨架生成面板（侧栏）
      ClaimVerifier.tsx              -- LLM求证嵌入组件（内联在编辑器中）
      PublishTargetSelector.tsx      -- 发布目标选择器（瓜田佐证/社区帖子）
      DraftAutoSaver.tsx             -- 草稿自动保存组件（每3秒）
      PresetTemplatePicker.tsx       -- 降级模板选择器（3个预设）
  services/
    creationService.ts              -- 创作辅助AI服务（统一LLM调用入口）
  stores/
    creationStore.ts                -- 创作状态zustand store（localStorage持久化）
```

### 3.2 核心数据流

```
用户输入主题+立场+目标读者
    ↓
creationService.ts → llmStore.callLLM()
    ↓
返回骨架JSON（标题建议+大纲+待核查标记）
    ↓
用户逐段填充
    ↓
ClaimVerifier检测声明性语句 → 标记⚠️
    ↓
用户点击[一键核查] → 复用LLM文字求证流程
    ↓
核查结果内联展示（支撑/未支撑/有出入）
    ↓
用户确认 → PublishTargetSelector选择发布目标
    ↓
发布到瓜田佐证 或 社区帖子
```

### 3.3 草稿持久化

```
creationStore.ts:
  - draft: { title, stance, content, skeleton, lastSavedAt }
  - 每3秒检查内容变化，变化则写入localStorage
  - key: 'jw_creation_draft'
  - 进入工作间时检查是否有未发布草稿，提示恢复

保底逻辑：
  - localStorage写入失败（隐私模式）→ 仅内存保存，提示"草稿可能不会保留"
  - 草稿超过5篇 → 提示用户清理旧草稿
```

## 四、核心功能详细设计

### 4.1 结构化编辑器

**编辑流程**：

```
Step 1: 选题与立场
  用户输入：
    - 主题（如"南开大学学生被骗220万"）
    - 立场/观点（如"大学反诈教育严重缺失"）
    - 目标读者（如"大学生"）
  AI辅助：
    - 检查瓜田是否已有同主题瓜（避免重复）
    - 建议切入角度

Step 2: AI生成骨架
  输入：主题+立场+目标读者
  调用：creationService.ts → callLLM()
  AI输出骨架JSON：
    {
      titleSuggestions: ["标题1", "标题2", "标题3"],
      outline: [
        { type: "intro", label: "开头：事件概述", hint: "需填入关键事实", needsVerify: true },
        { type: "argument", label: "论点1", hint: "AI建议的论点方向", needsVerify: true },
        { type: "argument", label: "论点2", hint: "AI建议的论点方向", needsVerify: true },
        { type: "rebuttal", label: "反驳预判", hint: "读者可能的质疑" },
        { type: "conclusion", label: "结尾", hint: "总结观点+呼吁行动" }
      ]
    }
  用户操作：
    - 可接受/修改/删除每个骨架节点
    - 可调整顺序
    - 确认后进入写作阶段

Step 3: 填充写作
  用户在骨架引导下逐段写作
  LLM求证嵌入：
    - 检测到声明性语句 → 自动标记为"需核查"⚠️
    - 点击 ⚠️ 标记 → 一键调用LLM文字求证
    - 核查结果内联展示

Step 4: 发布
  用户确认后选择发布目标：
    - 发布为瓜田佐证
    - 发布为社区帖子
```

### 4.2 AI骨架生成

**调用方式**：
```
creationService.ts → llmStore.callLLM(skeletonPrompt, { topic, stance, readerType })
```

**Prompt要点**：
- 输入：主题+立场+目标读者
- 输出：严格JSON格式（标题建议+大纲+待核查标记）
- 温度：0.7（平衡创造性和稳定性）
- Token预算：输入~500，输出~1500

**降级**：
- callLLM超时（12秒）→ 返回3个预设模板供用户选择
- 返回格式错误 → 重试1次，仍失败用预设模板

### 4.3 LLM求证嵌入（Claim Verifier）

**工作流程**：

```
用户在编辑器中写下一句话：
  "据统计，2025年大学生遭遇电信诈骗的比例上升了37%"

ClaimVerifier检测到这是一个需要证据支撑的声明
  ↓
编辑器内联显示：
  ⚠️ "据统计...上升了37%" — 需要来源支撑
  [一键核查] [添加来源] [忽略]
  ↓
用户点击 [一键核查]：
  → 复用求证工具集的LLM文字求证能力（已嵌入工作间）
  → 调用callLLM分析可信度
  → 返回核查结果：
    ✅ 找到支撑：教育部2025年报告确实显示上升37%（来源URL）
    或
    ❌ 未找到支撑：未找到权威来源确认此数据
    或
    ⚠️ 数据有出入：教育部报告显示上升27%，非37%
  ↓
用户根据核查结果修改内容
```

**技术实现**：
- 声明检测：用LLM从当前段落中提取需要核查的声明（一次性调用，~500 token）
- 核查：复用LLM文字求证流程（已嵌入工作间，不再独立页面）
- 结果展示：内联在编辑器中，不跳转页面

### 4.4 发布目标选择器

```
PublishTargetSelector.tsx:
  - 选项1：发布为瓜田佐证
    → 选择目标瓜（从瓜田列表选择）
    → 选择方向（真/假）
    → evidenceStore.submitEvidence({ melonId, content, direction, source: 'ai-assisted' })
    → 佐证来源标记：evidence.aiAssisted = true

  - 选项2：发布为社区帖子
    → 选择目标社区
    → communityStore.createPost({ communityId, content, source: 'ai-assisted' })

发布奖励：
  - 创作发布 +5 积分（高于普通佐证+3，激励高质量内容）
  - 质量评分>80 → 额外 +3 积分
```

### 4.5 草稿持久化

```
creationStore.ts:
  - 状态字段：
    draft: {
      id: string,
      title: string,
      stance: string,
      readerType: string,
      skeleton: OutlineNode[],
      content: string,
      lastSavedAt: timestamp
    }

  - 自动保存逻辑：
    useEffect 监听 draft 变化
    debounce 3秒 → localStorage.setItem('jw_creation_draft', JSON.stringify(draft))
    保存成功后更新 lastSavedAt

  - 恢复逻辑：
    进入工作间时 → localStorage.getItem('jw_creation_draft')
    有未发布草稿 → 弹窗提示"检测到未完成草稿，是否恢复？"
```

#### 模板1：事件分析

```
标题：[事件名称] 分析

一、事件概述
  [提示：填入事件发生的时间、地点、人物、起因]

二、关键事实
  [提示：列出已知的关键事实，每条标注来源]
  - 事实1：____（来源：____）⚠️需核查
  - 事实2：____（来源：____）⚠️需核查

三、影响分析
  [提示：分析事件对相关群体的影响]

四、各方观点
  [提示：列出不同立场的声音]
  - 观点A：____
  - 观点B：____

五、总结
  [提示：你的判断和依据]
```

#### 模板2：观点论述

```
标题：[观点核心表述]

一、观点陈述
  [提示：清晰表达你的立场]

  [提示：第一个支撑论据，标注来源]⚠️需核查

三、论据2
  [提示：第二个支撑论据，标注来源]⚠️需核查

四、反驳预判
  [提示：读者可能的质疑及你的回应]

五、呼吁
  [提示：你希望读者做什么]
```

#### 模板3：对比评测

```
标题：[对象A] vs [对象B] 对比

一、对比对象介绍
  - 对象A：____
  - 对象B：____

二、维度1对比：[维度名]
  - 对象A：____
  - 对象B：____

三、维度2对比：[维度名]
  - 对象A：____
  - 对象B：____

四、综合结论
  [提示：基于对比给出结论]
```

### 5.2 降级触发与处理

| 场景 | 降级方案 |
|------|---------|
| 骨架生成时LLM不可用 | 展示3个预设模板，用户选择一个手动填充 |
| LLM求证时LLM不可用 | 标记声明为"待手动核查"，提示用户自行搜索验证 |
| callLLM超时（12秒） | AbortSignal终止，返回降级模板选择器 |
| 返回格式错误 | 重试1次，仍失败用预设模板 |

## 六、异常场景应对

### 6.1 LLM服务不可用

```
进入工作间 → callLLM() 检测LLM可用性
  ├─ 可用 → 正常创作流程
  └─ 不可用 → 展示降级提示横幅：
      "AI助手暂时不可用，已切换为预设模板模式"
      展示3个预设模板卡片供选择
```

### 6.2 草稿恢复冲突

```
场景：用户在A设备有草稿，在B设备打开工作间
处理：
1. 以lastSavedAt最新为准
2. 弹窗提示"检测到云端有更新草稿（实际是localStorage）"
3. MVP阶段不处理多设备同步，仅单设备草稿
```

### 6.3 发布失败

```
场景：发布到瓜田佐证时后端不可达
处理：
1. 保留草稿不丢失
2. 提示"发布失败，草稿已保留，请稍后重试"
3. 重试按钮
```

## 七、与其他模块的联动

```
工作间 → 瓜田系统
  创作的内容一键发布为瓜的佐证
  瓜详情页"写分析"入口 → 跳转 /agent-world?melonId=xxx 预填瓜信息

工作间 → 社区系统
  创作的内容一键发布为社区帖子

工作间 → LLM配置
  通过 llmStore.callLLM() 统一入口，跟随用户配置的provider

工作间 → trae宝
  trae宝分析结果纯展示，用户点击"去工作间深入佐证" → navigate('/agent-world')
  工作间不显示trae宝

工作间 → 用户体系
  创作并发布 → 积分奖励（+5/篇，高于普通佐证+3）
  质量评分>80 → 额外+3积分
```
