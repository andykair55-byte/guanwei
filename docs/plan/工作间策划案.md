# 观微「工作间」策划案

> 版本：v3.0 | 更新日期：2026-07-16 | 状态：基于 Demo 现状重写

---

## 一、模块定位

**多平台内容工作台** —— 把一个话题，变成适配观微/知乎/小红书/微博/抖音/贴吧等多个平台的可发布内容。

工作间不再走 v2.0 的"碎片收集 → 信息全景"路线，而是回归到 Mirror 式的工作台架构：左侧工作空间列表、中间多平台编辑器、右侧 Agent 活动流、底部 Commander 输入栏、顶部版本快照栏，由 Commander 调度 5 个 Agent（搜索/研究/核查/写作/适配）完成从话题到多平台内容的全流程。

---

## 二、路由与入口

| 路由 | 页面组件 | 平台 | 说明 |
|------|----------|------|------|
| `/agent-world` | `AgentWorldPage` | 仅 Web | 工作间主页，Mirror 工作台架构 |
| `/ai-creation` | `AICreationPage` | Web + Mobile | AI 创作引导页（多步表单式） |

入口定义在 `src/router/routes.ts` 的 `layoutRoutes` 中。`/agent-world` 仅注册为 `web`，移动端不可达；`/ai-creation` 为 `both`。

首次访问 `/agent-world` 时，若本地无工作空间或检测到脏数据，自动注入 8 个 Demo 工作空间，首个为 `DEMO_TOPIC = 'AI换脸诈骗频发：技术滥用下的信任危机'`，并预填 6 个平台内容、Demo 活动事件、V1–V5 快照，供评委/新用户即开即看。

## 三、工作台架构

`AgentWorldPage` 采用「五件套」布局，配套新手引导：

```
┌─────────────┬───────────────────────────┬─────────────┐
│             │  顶部面包屑 + Assist/Auto   │             │
│             │  平台 Tabs + 发布按钮栏     │             │
│  Workspace  │  状态栏（字数/阅读/Agent）  │  Activity   │
│   Sidebar   │  ─────────────────────────  │   Stream    │
│             │  标题输入 + MarkdownEditor  │             │
│             │  ─────────────────────────  │             │
│             │  VersionBar（版本快照）     │             │
│             │  CommanderInput（指令栏）   │             │
└─────────────┴───────────────────────────┴─────────────┘
                  首次访问叠加 OnboardingGuide
```

### 3.1 WorkspaceSidebar（左侧）
工作空间列表，从 `workspaceStore.workspaces` 读取。支持创建/切换/收藏/归档/状态标记（active/tracking/completed/draft/archived）。首次访问自动注入 8 个 Demo 工作空间。

### 3.2 MarkdownEditor（中间）
- **顶部 Header**：面包屑（工作间 / 当前标题 / 最后保存时间）、Assist/Auto 模式切换、通知按钮
- **平台 Tabs**：通用稿（主稿）+ 已添加的外部平台，每个 Tab 显示平台图标/名称/已生成勾选/移除按钮；末尾「+」按钮打开添加平台弹窗
- **发布确认栏**：所有平台发布按钮 + 「已完成发布 (N)」确认
- **状态栏**：Agent 运行指示、字数、预计阅读时间、已复制/生成中提示
- **编辑区**：标题输入框 + MarkdownEditor 组件，内容按平台独立存储
- **底部 VersionBar**：版本快照栏，支持回退/锁定，3 秒空闲自动保存
- **底部 CommanderInput**：指令输入栏，发送后进入 Commander 状态机

### 3.3 ActivityStream（右侧）
活动流，按 workspace 隔离，事件按 Agent 类型筛选。事件类型包括：`commander_welcome` / `user_action` / `commander_plan` / `commander_question` / `agent_started` / `search_complete` / `research_complete` / `verify_warning` / `writing_complete` / `info` / `error`。每个事件可携带 `actions`（按钮），点击触发 `handleAction`（如确认计划、采纳观点、切换平台、重新核查、适配全部）。

### 3.4 CommanderInput（底部）
指令输入栏，发送后调用 `commanderService.handleUserInput(workspaceId, text, mode)`。Commander 会先做内容过滤（`filterUserInput`），再根据当前 phase（idle/collecting/awaiting_confirmation/executing/done）走不同分支。

### 3.5 VersionBar（底部中间）
版本快照栏，从 `snapshotStore` 读取。编辑器内容变更后 3 秒防抖自动建快照。支持锁定（不被 LRU 回收）、回退（回退前自动保存当前状态作为「回退前的状态」快照）。

### 3.6 OnboardingGuide（叠加层）
首次访问且 `workspaceStore.onboardingCompleted === false` 时显示，完成后写入 `completeOnboarding()`（持久化）。

## 四、多平台适配

### 4.1 platformTemplates 机制

定义在 `src/config/platformTemplates.ts`，6 个平台模板：

| PlatformId | 名称 | 字数上限 | 描述 | 发布 URL |
|------------|------|----------|------|----------|
| `guanwei` | 观微 | 5000 | Markdown 长文 + 佐证引用 | `/publish` |
| `zhihu` | 知乎 | 10000 | 专业长文 + 引用 + 数据 | 知乎专栏写作页 |
| `xiaohongshu` | 小红书 | 2000 | 图文 + 标题党 + emoji | 小红书发布页 |
| `weibo` | 微博 | 2000 | 短文 + 话题标签 | 微博发布页 |
| `douyin` | 抖音 | 500 | 短视频脚本（口播+画面） | 抖音创作者中心 |
| `tieba` | 贴吧 | 3000 | 社区帖子，随性风格 | 贴吧首页 |

每个模板包含 `promptTemplate`（LLM 适配 prompt）。`guanwei` 为主稿平台，不可移除；其余 5 个为外部平台，可由用户按需添加/移除。

### 4.2 平台适配流程

1. **主稿生成**：Commander 管线的 `writing` 步骤生成 `guanwei` 主稿，写入 `workspace.platformContents.guanwei`
2. **平台适配**：`adapt` 步骤调用 `adaptToAllPlatforms(canonicalDraft)`，基于 Canonical Draft 一次性适配所有已添加平台；或单个平台调用 `adaptToPlatform(draft, platform)`
3. **手动覆盖**：用户在编辑器中修改某平台内容后，该平台 `overridden=true`，`overriddenAt` 记录覆盖时间
4. **发布**：点击发布按钮 → 复制内容到剪贴板 → 打开平台 `publishUrl` → 用户手动粘贴发布 → 点「已完成发布」将 workspace 状态置为 `published`

### 4.3 平台状态字段

```typescript
interface PlatformContent {
  title: string
  content: string
  generated: boolean       // 是否已由 Agent 生成
  overridden: boolean      // 是否被用户手动修改
  overriddenAt?: string
  generatedAt?: string
}
```

## 五、AI 创作引导页

`AICreationPage` 是与工作间平行的轻量引导页，面向移动端用户，采用多步表单式交互。代码中 `STEPS` 数组共 6 步，前 5 步为创作引导，第 6 步为发布确认：

| 步骤 | 标题 | 内容 |
|------|------|------|
| 1 | 选择平台 | 小红书 / 抖音 / B站 / 微博（4 选 1，与工作间的 6 平台不完全一致） |
| 2 | 输入主题 | 文本域，500 字上限 |
| 3 | 选择类型 | 图文 / 视频 |
| 4 | 选择风格 | 专业严谨 / 轻松日常 / 幽默风趣 / 知识教育 |
| 5 | 生成预览 | 点击「开始生成」→ 展示预览卡片，可重新生成或确认发布 |
| 6 | 发布 | 展示汇总卡片，点击「立即发布」→ `alert` + 跳转 `/melon` |

**与工作间的差异**：
- 引导页是表单驱动、单平台、纯前端 mock（`handleGenerate` 仅置 `generated=true`，未接 LLM）
- 工作间是工作台驱动、多平台、接 Commander + Agent 管线 + Canonical Draft
- 引导页的 4 个平台与工作间的 6 个平台不一致（引导页多了 B站，少了观微/知乎/贴吧）

## 六、数据层

5 个 Zustand store 各司其职：

| Store | 文件 | 持久化 key | 职责 |
|-------|------|-----------|------|
| `workspaceStore` | `stores/workspaceStore.ts` | `guanwei-workspaces` | 工作空间列表、当前选中、平台增删、平台内容、草稿（Canonical Draft 副本）、收藏/归档/状态、新手引导完成标记 |
| `activityStore` | `stores/activityStore.ts` | ❌ 不持久化 | 按 workspace 隔离的活动事件（上限 200 条/空间）、Agent 类型筛选、Demo 事件加载 |
| `commanderStore` | `stores/commanderStore.ts` | ❌ 不持久化 | 管线模式（assist/auto）、管线状态、5 个 Agent 运行时状态（含依赖图）、日志、追问卡片 |
| `canonicalStore` | `stores/canonicalStore.ts` | `guanwei-canonical-draft` | 规范内容草稿：topic / facts / claims / viewpoints / structure / references / metadata |
| `snapshotStore` | `stores/snapshotStore.ts` | `guanwei-snapshots` | 版本快照（上限 50 条，LRU 淘汰未锁定的）、当前快照指针、回退前自动存档、undo |

### 6.1 Agent 依赖图（commanderStore）

```
orchestrator（无依赖）
    └─ search（依赖 orchestrator）
         ├─ research（依赖 search）
         └─ verify（依赖 search）
              └─ writing（依赖 research + verify）
```

`adapt` 步骤不在 5 Agent 依赖图中，由 `commanderService` 在 `writing` 完成后按 `targetPlatform` 决定是否执行（`all` → 适配全部，单平台 → 适配该平台，`guanwei` → 跳过）。

### 6.2 Commander 状态机（commanderService）

```
idle ──handleUserInput──► collecting ──信息齐全──► planning
                              │                        │
                              │                  awaiting_confirmation
                              │                        │
                              │                  confirm / execute
                              ▼                        ▼
                          collecting              executing ──► done
```

- `idle`：解析目标 → `collecting`
- `collecting`：LLM 判断缺失信息（platform/audience/depth），缺失则追问，齐全则 `planning`
- `planning`：构建 PlanStep[]（search/research/verify/writing/adapt）→ `awaiting_confirmation`
- `awaiting_confirmation`：用户确认 → `executing`；否则回到 `parseGoal`
- `executing`：按 plan 顺序运行各 Agent，每步产出写入 activityStore + canonicalStore + workspaceStore
- `done`：下次输入重置后重新 `parseGoal`

### 6.3 Demo 数据注入

首次访问 `/agent-world`（无 title 参数且无有效工作空间）时，`AgentWorldPage` 的 init effect 会：
1. 清空现有工作空间
2. 创建 8 个 Demo 工作空间（覆盖 active/tracking/completed/draft 四种状态）
3. 切换到首个工作空间（AI 换脸诈骗），写入 6 个平台内容（全部 `generated=true`）
4. 注入 9 条 Demo 活动事件（模拟完整管线时间线，含 `agent_started` / `search_complete` / `verify_warning` / `writing_complete`）
5. 创建 V1–V5 快照（V1/V3 锁定）

若 workspace 无历史事件，`loadDemoEvents` 会补注 13 条 Demo 指挥调度事件（welcome → plan → 5 Agent 启动/完成 → 任务完成）。
