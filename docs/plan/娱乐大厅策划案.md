# 观微「娱乐大厅」策划案

> 版本：v1.0
> 状态：产品策划 + 技术规格
> 优先级：P0（消费侧第二引擎）
> 最后更新：2026-07
> 架构审查后新增：原 AI竞技场 归入娱乐大厅作为子模式

---

## 一、模块定位

**娱乐大厅是观微的娱乐模式组合入口——聚合人机对决、AI斗蛐蛐、多人辩论三种模式，是消费侧第二引擎。用户从侧边栏"娱乐"导航项进入 `/entertainment`，看到三个模式卡片，选择后进入对应玩法。**

解决的问题：
1. **娱乐入口分散**——原AI竞技场是独立路由，现归入娱乐大厅统一管理
2. **参与门槛**——AI斗蛐蛐零参与门槛，人机对决低门槛，多人辩论（即将开放）提供深度参与
3. **病毒传播**——辩论内容自带传播属性，是冷启动期的流量引擎

给谁用：所有用户（零门槛参与）。

**与原AI竞技场的关系**：
- 原AI竞技场（独立路由 `/arena`）已归入娱乐大厅
- AI斗蛐蛐作为娱乐大厅的核心子模式，复用现有 AIBattle/CricketForge 代码
- 路由从 `/arena` 调整为 `/entertainment` 大厅入口 + 子路由

## 三、模式一：人机对决

### 3.1 玩法设计

```
用户进入 /entertainment/debate
  ↓
选择辩题（从预设辩题池选择，或随机）
  ↓
选择立场（正方/反方）
  ↓
选择AI对手性格（6种预设：犀利/温和/严谨/激进/幽默/学院派）
  ↓
开始辩论：
  - 用户先发言（或AI先发言，可配置）
  - 双方轮流发言，5-8轮
  - AI发言调用callLLM()
  ↓
辩论结束 → AI裁判评分 → 给出胜负
  ↓
下注结算（如下注）
```

### 3.2 辩题池（预设）

```
1. "大学生应不应该兼职"（正方：应该 / 反方：不应该）
2. "AI是否会取代人类创作"（正方：会 / 反方：不会）
3. "社交媒体利大于弊还是弊大于利"（正方：利大于弊 / 反方：弊大于利）
4. "应试教育是否应该被废除"（正方：应该 / 反方：不应该）
5. "远程办公是否应该成为常态"（正方：应该 / 反方：不应该）
... （预设10-15个辩题）
```

### 3.3 AI辩手性格

| 性格 | Prompt要点 | 辩论风格 |
|------|-----------|---------|
| 犀利 | 直击要害，快速反驳 | 攻击性强 |
| 温和 | 建设性论证，承认对方合理处 | 防守反击 |
| 严谨 | 引用数据，逻辑严密 | 学术风 |
| 激进 | 极端立场，放大矛盾 | 高对抗 |
| 幽默 | 调侃式反驳，段子手 | 娱乐向 |
| 学院派 | 结构化论证，总分总 | 教科书风 |

### 3.4 裁判评分

```
辩论结束后，调用callLLM()进行裁判：
  - 评分维度：逻辑性、论据充分性、表达力、反驳能力
  - 每维度1-10分
  - 总分高者获胜
  - 返回胜负+评分理由

展示：
  ┌──────────────────────────────┐
  │  裁判评分                     │
  │  你：32分  AI：35分           │
  │  AI获胜 🤖                    │
  │                              │
  │  评分理由：                   │
  │  你的逻辑性更强（9分），但     │
  │  AI的论据更充分（9分）...      │
  └──────────────────────────────┘
```

## 四、模式二：AI斗蛐蛐（已有基础）

### 4.1 玩法设计

```
用户进入 /entertainment/cricket
  ↓
选择两个AI角色（从6个预设角色中选择）
  ↓
开始对战：
  - AI双方自动轮流发言，5-8轮
  - 每轮调用callLLM()
  ↓
辩论结束 → AI裁判评分 → 给出胜负
  ↓
弹幕 + 围观下注
```

### 4.2 已有基础

复用现有代码：
- `AIBattle.tsx`（741行）— 对战主页面
- `CricketForge.tsx`（384行）— 蛐蛐锻造（自定义角色）
- `RefereeDice.tsx`（199行）— 裁判骰子动画
- `aiJudgeService.ts`（279行）— 裁判评分服务

### 4.3 功能清单

| 功能 | MVP要求 | 状态 |
|------|---------|------|
| AI角色 | 6个预设角色，各有性格prompt | 已有 |
| 自动辩论 | AI双方自动轮流发言，5-8轮 | 已有 |
| 裁判评分 | 辩论结束后AI裁判给出胜负 | 已有 |
| 弹幕 | 观众发弹幕，5条/分钟限流 | 需完善 |
| 围观下注 | 选边下注，固定赔率1.8 | 需新增 |
| CricketForge持久化 | 自定义蛐蛐存入localStorage | 需修复 |

### 4.4 下注设计

```
BetPanel.tsx:
  - 选边：正方/反方
  - 下注金额：10-100积分（L2以下不可下注）
  - 固定赔率：1.8（下注100赢则获得180，净赚80）
  - 每日下注上限：200积分
  - 结算：辩论结束后自动结算

积分联动：
  - 下注时：userStore.deductPoints(amount)
  - 赢：userStore.addPoints(amount * 1.8)
  - 输：已扣除，不退还
  - 每日获胜上限5次（+15积分额外奖励）
```

## 七、异常场景应对

### 7.1 LLM不可用

| 场景 | 降级方案 |
|------|---------|
| AI发言时LLM超时（12秒） | 返回预设台词"（AI正在思考...）"，继续辩论 |
| 裁判评分时LLM不可用 | 随机判定胜负+mock评分理由 |
| 连续3次超时 | 提示"AI服务暂时不可用，请稍后再试" |

### 7.2 下注异常

| 场景 | 处理 |
|------|------|
| 用户积分不足 | 提示"积分不足，无法下注" |
| 辩论中途异常中断 | 退还下注积分 |
| 并发下注（多tab） | 幂等处理，同一辩论只能下注一次 |

### 7.3 弹幕防刷

```
DanmakuInput.tsx 限流：
  - 5条/分钟上限
  - 3秒最小间隔
  - 60秒重复内容检测
  - 敏感词过滤（基础版）

LLM不可用时：
  - 使用预设弹幕池模拟热闹氛围
  - 随机延迟1-3秒展示一条
```

```
娱乐大厅 → 用户体系
  下注扣积分 → 辩论结束结算
  获胜+15积分（每日5次上限）
  L2以下不可下注

娱乐大厅 → LLM配置
  通过 llmStore.callLLM() 统一入口
  跟随用户配置的provider

娱乐大厅 → trae宝
  娱乐大厅不显示trae宝（娱乐模式不需要陪伴）

娱乐大厅 → 工作间
  精彩辩论内容 → v2可转化为"辩论分析"模板供工作间使用
```
