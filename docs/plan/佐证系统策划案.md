# 见微「佐证系统」策划案

> 版本：v3.0
> 状态：可落地执行
> 最后更新：2025-05

---

## 一、模块定位

**佐证系统是观微的众包事实核查层——用户为瓜提交支持真或支持假的论据，通过社区投票筛选最佳佐证，开奖时自动评选，为瓜的可信度提供依据。**

---

## 二、MVP范围定义

### 做什么

| 功能 | 说明 |
|------|------|
| 佐证提交 | 用户为瓜写佐证，选择方向（支持真/支持假），纯文字+可选外链 |
| **AI辅助创作提交** | **瓜详情页"写分析"按钮 → 跳转AI创作助手（/create?melonId=xxx）→ 预填瓜信息 → AI辅助写分析 → 一键发布为佐证。佐证来源标记为ai-assisted，透明展示** |
| 佐证列表 | 瓜详情页按方向分组展示佐证，最佳佐证置顶 |
| 社区投票 | 点赞/踩，每人每条佐证只能投一次 |
| 字数门槛 | 最低20字，建议50字 |
| 频率限制 | 每用户每日最多5条佐证（手动+AI辅助共享额度） |
| 最佳佐证评选 | 瓜开奖时自动评选，每方向选1条 |
| 佐证激励 | 积分奖励 + 简单佐证榜（AI辅助创作+5积分/篇，高于普通佐证+3） |

### 砍掉清单

| 砍掉项 | 原因 |
|--------|------|
| AI质量评分 | 成本不可控（每条佐证调一次LLM），延迟影响提交体验 |
| AI实时检测 | 同上，且MVP阶段无必要 |
| 同IP降权 | 大学NAT环境下整个宿舍楼共享IP，会误杀所有室友 |
| 引用体系 | 需要跨模块联动，MVP复杂度过高 |
| 视频/数据佐证类型 | MVP只支持纯文字+外链，降低前端复杂度 |
| 佐证协作编辑 | 需要版本控制系统，远超MVP范围 |
| 多维度排序 | MVP只用一种排序：投票得分 |
| 来源可信度分级 | 需要人工标注体系，MVP无法实现 |
| 佐证版本历史 | 复杂度过高 |

### 3.3 佐证与评论的关系

**MVP决策：佐证和评论保持独立，但明确区分定位。**

| 维度 | 佐证 | 评论 |
|------|------|------|
| 定位 | 带方向的论据，影响瓜的可信度 | 自由讨论，不影响可信度 |
| 提交门槛 | 字数>=20，每日限5条 | 无字数限制 |
| 方向 | 必须选择支持真/支持假 | 无方向 |
| 激励 | 有积分奖励，可当选最佳 | 无特殊激励 |
| 展示位置 | 瓜详情页上方独立区域 | 瓜详情页下方 |

如果上线后发现用户不理解两者区别，备选方案：将佐证降级为"带方向标签的特殊评论"，统一用评论组件渲染，通过标签筛选区分。

## 四、实现路径

### 阶段1：佐证CRUD + 列表展示（2天）

| 步骤 | 交付物 | 验证方式 |
|------|--------|----------|
| 后端佐证CRUD接口 | `POST/GET /api/evidence` | curl测试，数据正确入库 |
| 前端佐证提交表单 | `EvidenceForm.tsx` 含方向选择+文字输入+外链输入 | 提交成功，字数校验生效 |
| 前端佐证列表 | `EvidenceList.tsx` 按方向分组 | 支持真/支持假分区展示 |
| 字数门槛 | 前端+后端双重校验>=20字 | 低于20字提交被拒 |

### 阶段2：投票 + 排序（1天）

| 步骤 | 交付物 | 验证方式 |
|------|--------|----------|
| 投票接口 | `POST /api/evidence/{id}/vote` | 点赞/踩成功，不可重复投票 |
| 投票UI | `EvidenceVoteButtons.tsx` | 点击后数字更新，已投票状态正确 |
| 排序逻辑 | 后端按投票得分排序 | 高赞佐证排在前面 |

### 阶段3：最佳佐证评选（1天）

| 步骤 | 交付物 | 验证方式 |
|------|--------|----------|
| 开奖触发评选 | 瓜开奖时调用 `evidence_service.select_best()` | 每方向选出1条最佳 |
| 最佳标记展示 | `BestEvidenceBadge.tsx` 置顶展示 | 瓜详情页可见最佳佐证 |

### 阶段4：激励 + 佐证榜（1天）

| 步骤 | 交付物 | 验证方式 |
|------|--------|----------|
| 积分发放 | 提交佐证+3，最佳佐证+30 | 用户积分正确增加 |
| 佐证榜页面 | 展示累计最佳佐证数TOP10 | 排名正确 |

## 五、状态流转

### 5.1 佐证状态机

```
提交 → 可见（正常展示）→ 被踩折叠（downvotes >= 10）→ 开奖评选 → 最佳/普通
                         ↑
                    管理员删除 → 不可见
```

### 5.2 投票状态

```
未投票 → 点赞（upvote）  → 可切换为踩
未投票 → 踩（downvote）  → 可切换为点赞
已点赞 → 取消 → 未投票
已踩   → 取消 → 未投票
```

### 5.3 最佳佐证评选规则

```python
def select_best_evidence(melon_id, direction):
    """
    瓜开奖时调用，为指定方向选出最佳佐证。

    评选标准（按优先级）：
    1. 投票得分 = upvotes * 3 - downvotes * 1（权重可调但不拍脑袋，3:1是社区投票常见比例）
    2. 字数 >= 50字的优先（同等得分下，长文排前面）
    3. 提交时间更早的优先（鼓励早期参与者）

    前提条件：
    - 佐证字数 >= 20
    - 投票得分 > 0（得分为负的不参与评选）
    """
    evidences = Evidence.query.filter_by(
        melon_id=melon_id, direction=direction
    ).all()

    candidates = [e for e in evidences if e.upvotes * 3 - e.downvotes > 0]
    candidates.sort(key=lambda e: (
        e.upvotes * 3 - e.downvotes,
        len(e.content) >= 50,  # 布尔值False < True
        -e.created_at_timestamp  # 越早越好
    ), reverse=True)

    return candidates[0] if candidates else None
```

## 六、异常场景应对

### 6.1 恶意反向佐证

| 场景 | 处理 |
|------|------|
| 用户故意在"真"方向写支持假的佐证捣乱 | 方向由用户自选，但投票机制会自然淘汰低质内容；如果佐证内容与方向明显矛盾，社区会踩 |
| 用户复制粘贴他人佐证 | 后端做同用户同瓜内容去重（hash比较），完全相同内容拒绝提交 |

### 6.2 串通踩佐证

| 场景 | 处理 |
|------|------|
| 多人串通踩某条佐证 | MVP阶段无法有效检测（无AI、无IP判断），依赖社区自净：如果佐证确实好，其他用户会点赞抵消 |
| 极端情况 | 管理员可手动恢复被恶意踩下的佐证（管理后台操作） |

### 6.3 开奖后佐证处理

| 场景 | 处理 |
|------|------|
| 开奖后佐证仍可投票 | 可以，但投票不再影响评选结果（已评完） |
| 开奖后仍可提交新佐证 | 可以，但不再参与最佳评选 |
| 开奖后发现最佳佐证是低质内容 | 管理员可手动取消最佳标记，重新指定 |

### 6.4 网络中断

| 场景 | 处理 |
|------|------|
| 提交佐证时断网 | 前端提示"提交失败，请重试"，表单内容保留不丢失 |
| 投票时断网 | 前端乐观更新，失败时回滚并提示 |

### 6.5 并发冲突

| 场景 | 处理 |
|------|------|
| 同一用户多端同时投票 | 后端幂等处理：同一用户对同一佐证只保留最后一次投票状态 |
| 开奖与投票并发 | 开奖时锁定该瓜的佐证投票（设置flag），开奖后的投票不计入评选 |

## 七、佐证供给激励

**核心问题：为什么用户要写佐证而不是评论？**

| 策略 | 说明 |
|------|------|
| 积分差异 | 佐证+3积分，评论+1积分。最佳佐证+30积分 |
| 荣誉展示 | 最佳佐证置顶+标记，评论区没有这个机制 |
| 佐证榜 | 累计最佳佐证数排行榜，满足竞争心理 |
| 引导提示 | 瓜详情页顶部提示"你的分析可以成为最佳佐证" |
| 降低门槛 | 不要求外链、不要求图片，纯文字分析即可 |

**关键设计：佐证的价值在于"被评选为最佳"。开奖时刻是产品的高光时刻——谁的分析被社区认可为最佳佐证，这种认可感是核心驱动力。**
