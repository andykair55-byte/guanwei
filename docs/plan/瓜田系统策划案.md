## 一、模块定位

**瓜田是观微的核心内容消费场景——大学生在这里刷"瓜"（社会热点/传闻），判断真假、提交佐证、参与事件时间线收束，最终在开奖时验证自己的判断力。**

解决的问题：用户打开观微后"看什么、做什么"。
给谁用：18-26岁大学生，每天刷手机3h+，对热点敏感但缺乏求证习惯。

### 4.4 运营期更新节奏

| 阶段 | 频率 | 内容来源 |
|------|------|---------|
| 第1周（冷启动） | 3个/天 | 预制的30个种子瓜分批释放 |
| 第2-4周 | 1-2个/天 | 热点跟踪LLM生成+人工审核 |
| 第5周+ | 1个/天 | 热点跟踪+用户话题投票选取 |

### 4.5 第三条供给路径：AI创作助手驱动的UGC

**核心逻辑**：前两条路径（种子瓜+运营生成）解决冷启动，但长期可持续的内容供给必须来自用户。AI创作助手是降低用户创作门槛的核心工具——它不是让用户"提交瓜"（门槛太高），而是让用户"写分析/佐证"（门槛低），高质量的佐证本身就成了瓜田的内容。

```
内容供给三条路径：

路径1: 运营手动录入（冷启动）
  运营团队 → LLM生成种子瓜 → 人工审核 → 发布到瓜田
  频率：30个种子 + 1-3个/天

路径2: 热点自动关联（v2）
  热点系统发现事件 → LLM生成瓜草稿 → 运营审核 → 发布
  频率：取决于热点频率

路径3: AI创作助手驱动的UGC（核心长期路径）
  用户看到瓜 → 想用AI辅助写分析 → 创作助手给骨架+核查 → 用户填充发布
  → 高质量佐证/分析自动沉淀在瓜详情页
  → 其他用户看到这些内容 → 也被激励写分析
  → 正循环

路径3的关键指标：
  - AI创作助手使用率 > 20%活跃用户
  - 创作→发布转化率 > 50%
  - AI辅助产出的佐证质量评分均值 > 65分
```

**与瓜田的具体联动**：

```
瓜详情页变化：
  原有：猜真假 + 佐证列表（纯用户手动写）
  新增：佐证列表顶部增加"写分析"按钮
    → 点击跳转创作助手（/create?melonId=xxx）
    → 自动预填瓜的标题、描述、时间线作为创作素材
    → 用户用AI辅助写分析 → 一键发布为该瓜的佐证
    → 佐证来源标记为 ai-assisted（透明性）

佐证列表展示变化：
  原有：按投票数排序
  新增：AI辅助产出的佐证增加"AI辅助"小标签（不隐藏，透明展示）
  质量评分>80的AI辅助佐证自动置顶（高质量内容优先展示）
```

**这解决了"用户不愿贡献"的根本问题**：
- 不是用户不想表达，是从想法到成品的摩擦太大
- AI创作助手把摩擦从"写一篇完整分析"降低到"填一个有引导的骨架"
- 事实核查功能确保产出内容有信息密度（不是灌水）
- 风格档案确保产出内容像用户自己写的（不是AI味）

## 六、事件时间线（产品方向增强）

### 6.1 设计理念

瓜不再是简单的"真/假"二元判断，而是**事件的时间线收束**：
- 用户看到的不是"这个消息是真是假"，而是"这个事件到底发生了什么"
- 随着时间推移，新信息不断加入时间线，真相逐渐清晰
- 开奖 = 时间线收束完成，形成完整事件脉络

### 6.2 时间线组件（EvidenceTimeline.tsx增强）

```
已有文件：src/components/EvidenceTimeline.tsx

增强内容：
1. 纵向时间线布局（左时间右事件）
2. 每个节点标注：时间、事件、来源、可信度星级
3. 节点状态：
   - confirmed（已证实）：绿色实线圆点
   - disputed（有争议）：黄色虚线圆点
   - unverified（未证实）：灰色空心圆点
4. 开奖后：所有节点状态确定，时间线完整展示
5. 展开/收起：默认显示最近3个节点，点击展开全部
```

### 6.3 时间线与猜瓜的联动

```
用户在瓜详情页看到的流程：

1. 阅读瓜描述 + 浏览时间线（已有信息）
2. 基于当前信息做出判断：选真/选假
3. 提交佐证（可选）：补充时间线节点或证据
4. 等待开奖：期间可能有新节点加入（运营更新）
5. 开奖：时间线完整展示 + 实锤报告
6. 对比：我的判断 vs 真相，积分结算
```

## 七、状态流转

### 7.1 瓜（Melon）状态机

```
                    ┌──────────────┐
                    │   draft      │  管理员/LLM创建
                    │  （草稿）     │  仅管理员可见
                    └──────┬───────┘
                           │ 管理员发布
                           v
                    ┌──────────────┐
                    │   pending    │  用户可见可参与
                    │  （进行中）   │  猜瓜+佐证+评论
                    └──────┬───────┘
                           │ 管理员开奖 / 到达revealTime
                           v
                    ┌──────────────┐
                    │   revealed   │  已开奖
                    │  （已揭晓）   │  展示报告+结算积分
                    └──────┬───────┘
                           │ 30天后
                           v
                    ┌──────────────┐
                    │   archived   │  归档
                    │  （已归档）   │  不再出现在列表
                    └──────────────┘
```

### 7.2 状态转换规则

| 转换 | 触发条件 | 执行动作 |
|------|---------|---------|
| draft -> pending | 管理员点击"发布" | 设置createdAt、加入列表索引 |
| pending -> revealed | 管理员点击"开奖" 或 到达revealTime | 设置result、生成Report、结算Guess、发放积分 |
| revealed -> archived | 开奖30天后自动 | 从列表移除、保留详情页可访问 |

### 7.3 前端状态映射

```typescript
// types/index.ts 中 Melon.status 扩展

// 前端只展示 pending 和 revealed 的瓜
// archived 的瓜仅在历史页面/个人猜瓜记录中可见
```

### 8.1 最新排序

```typescript
melons.sort((a, b) => new Date(b.createdAt).getTime() - new Date(a.createdAt).getTime())
```

### 8.2 最热排序（简化版）

```typescript
function calculateHotScore(melon: Melon): number {
  const ageHours = (Date.now() - new Date(melon.createdAt).getTime()) / 3600000
  const participation = melon.totalParticipants * 3
  const evidence = melon.evidenceCount * 5
  const comments = melon.commentCount * 2
  // 时间衰减：每过6小时热度减半
  const decay = Math.pow(0.5, ageHours / 6)
  return (participation + evidence + comments) * decay
}
```

权重说明：
- 参与人数权重3：核心指标，代表瓜的吸引力
- 佐证数权重5：高质量互动，代表用户深度参与
- 评论数权重2：轻量互动
- 时间衰减：确保新瓜有机会浮上来，老瓜自然沉底

## 九、异常场景应对

### 9.1 网络中断时猜瓜

```
场景：用户在弱网环境下点击"选真/选假"
处理：
1. 乐观更新：点击后立即更新UI（trueCount+1），显示"提交中..."
2. 后台重试：最多3次，间隔1s/2s/4s（指数退避）
3. 全部失败：回滚UI，toast提示"网络异常，请重试"
4. 离线队列：将操作存入localStorage，恢复网络后自动重放
5. 冲突检测：重放时服务端检查是否重复提交（幂等key = userId + melonId）
```

### 9.2 瓜在详情页开奖

```
场景：用户正在浏览瓜详情页，管理员此时开奖
处理：
1. 短轮询：详情页每30秒检查一次瓜状态（GET /api/melons/{id}/status）
2. 状态变化时：
   - 页面顶部弹出"刚刚开奖！"横幅
   - 3秒后自动刷新页面数据
   - 展示实锤报告 + 结算用户积分
3. 如果用户已离开页面：下次进入时直接展示revealed状态
```

### 9.3 多设备操作

```
场景：用户在手机和电脑同时登录，在不同设备上猜同一个瓜
处理：
1. 服务端幂等：userId + melonId 唯一约束，第二次提交返回已有结果
2. 前端检测：API返回409时，提示"你已在其他设备猜过此瓜"
3. 积分不重复发放：以第一次提交为准
```

### 9.4 LLM生成种子瓜失败

```
场景：调用书生模型生成种子瓜时超时或返回异常
处理：
1. 超时控制：单次请求15秒超时
2. 自动重试：最多2次，换用Groq作为fallback
3. JSON解析失败：重试时temperature降低到0.3
4. 全部失败：管理员手动填写瓜内容（AdminPage的MelonForm支持手动模式）
5. 降级方案：预制50个静态瓜作为最终兜底
```

### 9.5 开奖时积分并发

```
场景：多个用户同时触发开奖结算
处理：
1. 开奖是管理员操作，单线程执行，不存在并发开奖
2. 积分发放逐用户处理，每个用户的积分变动是原子操作
3. 后端用数据库事务保证：Guess.isCorrect + User.points 同时更新
```

## 十一、资源需求

### 11.1 LLM调用估算

| 场景 | 调用频率 | 单次token | 月总量 |
|------|---------|----------|--------|
| 种子瓜生成 | 30个（一次性） | ~1000 input + 800 output | ~54,000 tokens |
| 实锤报告生成 | 30次开奖 | ~500 input + 600 output | ~33,000 tokens |
| 热点瓜生成 | 30个/月 | ~1000 input + 800 output | ~54,000 tokens |
| **合计** | | | **~141,000 tokens/月** |

书生模型配额：9,000 token/月，远不够。
解决方案：
- 种子瓜和报告生成用Groq（llama-3.3-70b），无token月限（有rpm限制）
- 书生模型仅用于实时交互场景（如AI竞技场）
- Groq rpm限制30次/分钟，单次生成<15秒，满足需求

### 11.2 存储空间

## 十二、实现路径

### Phase 1：内容管线（第1-2天）

```
交付物：
1. melonSeedService.ts — LLM生成种子瓜，支持批量生成+单条生成

- 管理员能在后台看到30个待发布的瓜
```
