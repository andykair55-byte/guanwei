## 观微项目开发思路全景复盘

本文档记录观微(JianWei)从"一无是处"到"产品成型"的完整思考过程。不是技术文档，是决策日志——为什么这样做，为什么不那样做，以及中间走过的弯路。

---

### 一、起点：一个参赛作品的自我定位

观微参加的是 TRAE AI 创造力大赛。初始状态很朴素：一个信息求证工具，有工具箱，有基本的信息展示。问题在于——它只是一个"工具"，不是一个"产品"。

工具和产品之间差的是什么？是用户留下来的理由。工具箱再好，用完就走，没有留存，没有社区，没有身份认同。这是整个项目重构的出发点。

核心理念在这时候确立：**"AI辅助人判断，不替人决定"**。这句话不是随便写的，它是整个产品的锚。市面上求真类产品的思路是"AI告诉你真假"，观微的思路是"AI把资料摆在你面前，你可以参考AI的分析，但最终判断权在人"。这多出来的一步思考过程，就是差异化。

---

### 二、产品哲学：过程即娱乐

和 AI 讨论产品定位时，达成了一个关键共识：

大厂卷的是"客观题准确率"——谁判断得更准、更快、更权威。这条路观微走不通，也没有资源走。观微卷的是"过程即娱乐"。具体来说：

AI的思考过程、辩论的节奏感、悬念的营造、角色之间的人格碰撞——这些才是看点。LLM在主观题上的随机性不是缺陷，是特性。一场辩论的结果不确定，过程有戏剧性，观众有参与感，这就够了。

另一个核心认知：**"prompt工程本身就是对模型的微调"**。与其追求用更好的模型，不如把精力放在角色人格设计、话术模板、辩论节奏控制上。白泽和獬豸两个AI人格的对话，比一个模型正经地输出分析有趣得多。

---

### 三、从工具到社区的演进路径

#### 3.1 工具箱的定位调整

7个工具箱（截图EXIF鉴定、情绪操控检测、反向搜图、时间线重建、洗稿检测、多源验证、一键求证）最初是产品的核心卖点。但在讨论中意识到：

工具箱不是核心竞争力，是佐证辅助。大部分用户做判断是因为信息差，工具箱的功能本质上是帮用户缩小信息差——获取佐证资料、分析整理、时间线梳理。它们是"AI辅助人判断"理念的具体落地，但不是用户每天打开 app 的理由。

用户每天打开 app 的理由是：吃瓜、辩论、社交。

#### 3.2 辩论系统的三代演进

**第一代：双阵营投票**
最早的辩论场设计很传统——正方/反方两个阵营，用户选边站，提交论据，投票。这是"论坛模式"，本质上还是发帖+评论的变体。问题很明显：只有6个人能辩论，其他人只能看，参与度极低。

**第二代：AI斗蛐蛐**
加入AI辩论观赏模式。白泽vs獬豸，双人格AI对战，12轮回合制，带思考步骤揭示和实时投票。这一代的价值在于验证了"过程即娱乐"的可行性——看两个AI吵架确实有意思。但它只是观赏内容，用户没有参与感。

后来又扩展出三个子模式：AI斗蛐蛐（纯观赛）、人机对决（用户vs AI，4回合制）、圆桌局（狼人杀式多人混合）。角色体系共享 characters.ts，评分逻辑复用。

**第三代：全真人辩论场**
这是质变。用户提出关键问题："一个很热的帖子只能6个人辩论吗？如果场上的人辩论水平平庸，下面的人只会心急。"

解法：积分制换人机制。场下的用户消耗积分申请替换场上辩手，场上的人表现不好可以被80%投票"抬走"。阶梯递增积分（30→60→120→240）保证越往后门槛越高，防止无限换人导致辩论无法推进。发言字数限制也分阶段放开（前期150字→中期300字→后期500字），模拟真实辩论赛的节奏感。

这套设计的本质是：**把辩论从"静态帖子"变成"实时竞技"**。类似抖音PK的观赏性+竞技性，但载体是文字辩论而非直播。

### 四、关键架构决策

#### 4.1 多Agent独立服务，避免记忆污染

用户明确要求：AI的每个职能用独立agent，避免相互污染记忆。这和用户对 SonettoHere 记忆引擎的教训直接相关——那个项目每轮全量加载YAML，O(n) token爆炸，被评价为"最烂设计"。

落地方式：4个独立service文件，各自维护自己的prompt和上下文。

- `aiJudgeService.ts`：裁判agent，评分+逻辑谬误检测，temperature 0.3
- `aiTranslatorService.ts`：翻译agent，方言/黑话→标准中文，temperature 0.2
- `aiSummaryService.ts`：总结agent，全局复盘+MVP评选，temperature 0.4
- `debateRoomService.ts`：房间生命周期管理，串联以上三个agent

每个service都是 Groq API → mock fallback 的 withFallback 模式，保证离线也能跑。

#### 4.2 API层：withFallback模式

整个项目的API层采用统一模式：先调真实API，失败则fallback到mock数据。这不是偷懒，是参赛策略——评委演示时网络不稳定，mock保证demo永远能跑通。同时mock数据本身也是精心设计的产品原型，包含预写的思考步骤、角色话术、辩论节奏。

#### 4.3 路由级代码分割

App.tsx 所有21个页面全部用 React.lazy + Suspense 做代码分割。每个页面独立chunk，首屏只加载当前路由的代码。构建时间600ms，产物分割成70+个chunk。这不是性能强迫症，是实际体验需要——移动端首屏加载速度直接影响评委第一印象。

#### 4.4 瓜田+辩论融合（最终版首页）

这是讨论最久的架构决策。三个选项：

1. 抖音式视频流全屏切换——被否决，"抄抖音抄的太明显"
2. 保留瓜田卡片形式，每张卡片叠加实时辩论状态——**选中**
3. 混合流（卡片+全屏交替）——被否决，理由同上

选方案2的逻辑：观微本质是文字辩论，不是视频流。卡片形式信息密度高，适合文字内容。叠加辩论状态（激辩中/等待中/已结束 + 观众数 + 座位 + 积分）后，每张卡片从"静态帖子"变成了"实时赛况"。点击卡片直接进入辩论房间，不是详情页。

这个改动让首页从"信息流"变成了"竞技场入口"，用户的第一视觉焦点不再是"发生了什么"，而是"正在吵什么"。

#### 4.5 辩论评判机制：AI评分 + 人气票双轨制

这是辩论系统的核心设计，也是和其他"AI判断对错"产品的本质区别。

**问题：主观题怎么判输赢？**

辩论题目分三类，评判标准完全不同：

- **事实型**："地球是圆的吗"——看论据准确性，有客观标准
- **价值型**："该不该躺平"、"吕布vs项羽谁更强"——没有标准答案，比的是论证深度和价值观自洽
- **政策型**："该不该收社保"——看方案可行性和成本分析

如果用同一套标准评判所有题目，价值型题目会显得很荒谬——AI裁判不可能"判定"吕布一定打得过或打不过项羽。

**解法：AI打专业分，观众打人气分**

最终得分 = AI评分 × 权重 + 人气票 × 权重。AI裁判评估论证质量（逻辑链完整性、论据可靠性、反驳有效性），观众投票反映说服力。没有人气票就变成纯AI评判，失去了"人"的维度。

**AI裁判的"倾向性表达"**

AI裁判不只是冷冰冰地打分，可以说"我更倾向您的观点，因为..."，给出具体解释。这让评判过程本身成为内容——观众看到AI的分析后会想"我同意/不同意AI的判断"，这又是一层讨论空间。

**红字免责声明**

最后用红字标识："AI也会犯错"。这不是bug，是feature。它做两件事：一是降低用户对AI评判的过度信任，二是给"争议性判决"留退路——如果AI判错了，用户不会骂平台，因为平台已经说了AI可能犯错。

**为什么这是核心区别？**

求真类产品让AI做最终裁判（"这是假的"），观微让AI做参与者（"我觉得这个论证有道理，但你看这个反例..."）。裁判权始终在人手里。AI是辩手之一、是解说员、是分析师，但不是最终法官。最终法官是观众投票 + 用户自己的判断。

### 五、产品诚实评估

这段对话对项目走向影响最大。用户要求以"多年市场经验的项目总监"视角做客观分析：这个作品在阿里或投资人眼里是垃圾还是大垃圾？

核心结论：

**优势：** 产品完整度高（工具箱+社区+辩论+AI多角色），场景落地真实（信息求证有真实需求），差异化明确（过程即娱乐 vs 客观准确率）。

**劣势：** 没有真实用户验证（100个测试用户可以靠AI模拟，但和真实用户行为完全不同），没有增长策略，没有盈利模型的具体数据支撑。

**关键建议：**

1. 盈利模型要提前设计——积分购买（竞技性辩论的入场费）、流量加热（类似抖+）、打赏系统
2. 工具箱是佐证不是核心——核心竞争力在社区活跃度和辩论内容质量
3. 好牌一张一张打——不要一次展示所有功能，先打"瓜田+辩论"这张核心牌
4. 多Agent是成熟模式无壁垒——工具箱需要技术深度（图像取证/NLP分析），不是简单调API

用户对此全盘接受，并补充了关键的盈利设计思路：阶梯积分制类似抖音PK已被验证，观众打赏增加话题热度类似免费送抖+。

### 六、UI设计演进

#### 6.1 去emoji化

用户明确讨厌UI中出现emoji，认为是"AI生成前端的典型特征"。现实软件很少用emoji做UI元素。

落地方式：角色头像从emoji改为 CharacterIcon.tsx SVG组件（龙角/独角/竖瞳/翼翅），通过 currentColor 继承颜色。思考步骤中的emoji保留在数据层作为key，渲染层通过 THINKING_ICON_MAP 转为Lucide图标组件。

#### 6.2 白底现代风

从宣纸暖色系重构为纯白干净系。24个文件改动，核心是 index.css 的 @theme token 改名（bg-paper/text-ink-900 等保留Tailwind class名，变更自动级联全站）。圆角从 rounded-2xl 缩减到 rounded-xl（卡片）和 rounded-lg（按钮），更接近 Twitter/X 的干净风格。

#### 6.3 设计原则

- 颜色心理学+渐变光谱（cyan→blue→purple→magenta→red），拒绝红绿灯式配色
- 粒子物理特效+质感，重视触觉反馈
- Mobile-first 480px，Twitter/X式高互动感
- 品牌色：红色（seal），辅助色：金色（gold）、竹绿（bamboo）

### 七、技术债务与已知问题

1. **mock数据与真实API的切换**：目前 withFallback 模式在演示时很稳，但上线时需要完整的后端。mock数据中预写的思考步骤和辩论节奏是手工设计的，真实LLM输出质量不可控。

2. **辩论房间状态管理**：debateStore 目前用 Zustand 管理，但房间生命周期（创建→加入→辩论→结束→清理）的边界情况还没完全覆盖。特别是"80%抬走"后的候选人递补逻辑。

3. **积分经济平衡**：阶梯递增（30 * 2^n）在n=4时已经是480积分，需要验证经济系统是否会通胀或通缩。

4. **AI成本**：一场完整辩论（裁判+翻译+总结）需要3次Groq API调用。书生模型（9kw token/月，30rpm）加入后可以分担，但需要设计调度策略。

### 八、开发方法论提炼

回顾整个过程，有几个反复出现的思维模式：

**"这个功能解决了谁的什么问题？"** ——每个功能都要回答这个问题。工具箱解决"信息差"，辩论场解决"参与感"，积分制解决"质量筛选"。回答不了的功能不做。

**"好牌一张一张打"** ——不要一次展示所有功能。先打核心牌（瓜田+辩论），再打辅助牌（工具箱），最后打彩蛋牌（圆桌局、打赏）。

**"大厂做什么，我就不做什么"** ——差异化不是在别人的赛道上跑得更快，是换一个赛道。大厂卷准确率，我卷过程娱乐。大厂做AI判断，我做AI辅助人判断。

**"O(n)不可接受"** ——任何随输入规模线性增长的token消耗都要警惕。多agent独立服务、分页加载、mock fallback都是这个原则的体现。

**"产品完整度 > 技术架构"** ——核心竞争力不在技术架构本身，在于场景落地和产品完整度。一个架构再优雅，如果用户用不起来，就是零。

### 九、如果重来一次

1. 先做辩论场，后做工具箱。辩论场是留存引擎，工具箱是锦上添花。
2. 第一天就设计积分经济。不要等功能做完再想盈利，积分系统渗透在辩论、换人、打赏的每个环节。
3. 更早引入真实用户测试。AI模拟的用户和真实用户行为差异巨大，尤其是"什么样的内容会让人想辩论"这个问题，只有真实数据能回答。
4. 减少UI重构次数。从暖色到白底改了一次，去emoji改了一次，每次都是20+文件的改动。应该在项目初期就确定设计规范并锁定。
