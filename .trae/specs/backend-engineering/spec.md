# 见微后端工程化 Spec

## Why
项目已完成 MVP 核心功能开发，需要系统性地规划后端工程化改造，包括：技术栈确认、数据库选型、性能优化、鲁棒性设计、耦合性降低、测试覆盖等，为验证期（500-1000 DAU）做好准备。

## 核心约束（基于用户反馈）

### 目标用户群体优先级

| 阶段 | 目标用户 | 说明 |
|------|---------|------|
| **MVP/验证期** | 青年、大学生（18-35岁） | 互联网原住民 + AI原住民，最爱吃瓜，高质量用户群体 |
| **成长期** | 扩展到中年群体（35-50岁） | 稳定用户后向外拓展 |
| **爆发期** | 青少年、老年人等特殊群体 | 扩展功能，非主流决策方向 |

**设计原则**：
- 前期聚焦核心用户，确保高质量用户群体
- 特殊人群关怀作为扩展项，不影响核心架构决策
- 先有用户，再拓展；先有质量，再扩大规模

### Agent 模块决策约束

- **Agent 模块暂不设计**：由用户做主要决策
- **证据链体系需要设计**：核心功能，影响数据库设计
- **工具箱模块功能筛选**：确认必要功能

### 测试覆盖

## ADDED Requirements

### Requirement: 证据链体系设计
系统 SHALL 设计证据链数据结构，支持多源证据的采集、存储、展示和可信度评估。证据链是 AI 求证的核心产出，需支持可视化展示。

#### Scenario: 证据链数据结构
- **WHEN** Agent 完成信息搜集
- **THEN** 生成结构化证据链，包含：
  - 证据 ID
  - 来源 URL
  - 来源类型（官方/媒体/社交/论坛）
  - 可信度等级（1-5星）
  - 时间戳
  - 内容摘要
  - 相关性评分

#### Scenario: 证据链存储
- **WHEN** 求证完成
- **THEN** 证据链存储至数据库，与瓜/求证记录关联

#### Scenario: 证据链可视化
- **WHEN** 用户查看实锤报告
- **THEN** 按可信度排序展示证据链，高可信度证据优先显示

---

### Requirement: 工具箱模块功能筛选
系统 SHALL 提供工具箱功能，但需筛选必要功能，避免过度开发。

#### 工具箱功能优先级

| 功能 | 优先级 | 积分消耗 | 说明 |
|------|--------|---------|------|
| 一键求证 | P0 | 50积分/次 | 核心功能，已实现 |
| 深度调查 | P1 | 200积分/次 | 多 Agent 协作，暂不开发 |
| 谣言粉碎机 | P2 | 免费 | 后续版本 |
| 专题追踪 | P2 | 100积分/次 | 后续版本 |

**MVP 必要功能**：仅"一键求证"

---

### Requirement: 数据库选型与迁移
系统 SHALL 从 SQLite 迁移到 PostgreSQL，满足验证期（500-1000 DAU）需求。

#### Scenario: PostgreSQL 配置
- **WHEN** 系统进入验证期
- **THEN** 使用 PostgreSQL 15，配置连接池（最小 5，最大 20）

#### Scenario: 数据迁移
- **WHEN** 从 SQLite 迁移数据
- **THEN** 使用 SQLAlchemy 导出/导入，保持数据一致性

#### 数据库选型对比

| 维度 | SQLite | PostgreSQL | 决策 |
|------|--------|------------|------|
| 并发写入 | 单写多读 | 多写多读 | PostgreSQL |
| 数据量 | < 1GB | 无上限 | PostgreSQL |
| 运维成本 | 零 | 低 | SQLite MVP，PG 验证期 |
| 迁移成本 | - | 1小时 | 可接受 |

---

### Requirement: 缓存策略设计
系统 SHALL 实现热点数据缓存，提升 API 响应速度。

#### 缓存优先级

| 数据类型 | 缓存策略 | TTL | 说明 |
|---------|---------|-----|------|
| 瓜田列表 | Redis 缓存 | 5分钟 | 高频访问 |
| 用户积分 | Redis 缓存 | 1分钟 | 实时性要求 |
| 段位配置 | 内存缓存 | 永久 | 静态数据 |
| 实锤报告 | 不缓存 | - | 动态生成 |

#### Scenario: 缓存失效
- **WHEN** 用户猜瓜后积分变动
- **THEN** 立即失效该用户积分缓存

---

### Requirement: API 性能目标
系统 SHALL 满足以下性能目标（验证期）。

| 端点 | 目标响应时间 | 说明 |
|------|-------------|------|
| 瓜田列表 | < 200ms | 高频访问 |
| 猜瓜详情 | < 300ms | 包含佐证数据 |
| 提交判断 | < 100ms | 快速响应 |
| AI 求证 | < 5s | LLM 调用耗时 |
| 用户信息 | < 100ms | 缓存优先 |

---

### Requirement: 鲁棒性设计
系统 SHALL 实现多层容错机制，确保服务可用性。

#### LLM 调用降级（已实现）
- 主模型失败自动降级
- 降级顺序：DeepSeek → Groq → OpenAI → Claude
- 全部失败返回友好错误提示

#### Scenario: 数据库连接池
- **WHEN** 数据库连接池耗尽
- **THEN** 等待连接释放，超时后返回 503 错误

#### Scenario: API 错误统一处理
- **WHEN** 任何 API 端点发生错误
- **THEN** 返回统一错误格式：`{ code, message, details }`

---

### Requirement: 模块耦合性降低
系统 SHALL 降低模块耦合度，便于后续扩展。

#### 前端 API 层解耦
- 当前：`api.ts` 包含 `withFallback` 逻辑
- 改进：分离真实 API 和 Mock API，通过环境变量切换

#### 后端服务层模块化

```
backend/services/
├── llm.py          # LLM 调用（独立）
├── evidence.py     # 证据链服务（新增）
├── credibility.py  # 可信度评估（独立）
├── points.py       # 积分服务（独立）
├── rank.py         # 段位服务（独立）
└── cache.py        # 缓存服务（新增）
```

#### Agent 模块约束
- **暂不改动**：Agent 模块由用户决策
- 当前状态：`backend/agents/` 已实现，保持现状

|------|------|------|
