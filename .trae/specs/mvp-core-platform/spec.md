# 见微 MVP 核心平台 Spec

## Why
见微是一个"理性吃瓜"的信息求证社区，需要从零构建 MVP 版本。核心使命是提升全民信息素养，不是简单的"猜对游戏"，而是让用户在参与过程中学会思考和判断。

## What Changes
- 创建 React + TypeScript 前端应用，移动端优先的底部 Tab 导航架构
- 实现瓜田信息流 + 猜瓜核心玩法（包含佐证机制，让用户学会思考）
- 实现 AI 一键求证工具（基础版），积分可兑换工具箱额度
- 实现积分系统（每日获取上限）和段位成长体系（前期快后期慢，顶级按年计）
- 实现佐证机制，用户可提交自己的判断依据，被点赞/选为最佳可获额外奖励
- 实现 AI 内容初筛模块
- 后端：FastAPI + LangGraph 多 Agent 编排（搜集→验证→分析→报告）
- 所有数据现阶段使用 Mock 数据，预留真实 API 接口

| 层级 | MVP 阶段 | 验证期 | 成长期 | 爆发期 |

### 各阶段详细规划

#### Phase 1: MVP（当前）
**目标**：快速验证产品核心假设

| 指标 | 目标 |
|------|------|
| 日活用户 | 0 - 1,000 |
| 同时在线 | 0 - 200 |
| 数据库 | SQLite（本地文件） |
| 服务器 | 开发机即可 |
| 预估成本 | ¥0 |

**决策点**：
- SQLite 单写多读，200 并发无压力
- 无运维负担，所有精力投入产品迭代
- 迁移到 PostgreSQL 仅需改连接字符串

#### Phase 2: 验证期
**触发条件**：用户反馈"加载慢" 或 日活突破 1,000

| 指标 | 目标 |
|------|------|
| 日活用户 | 1,000 - 10,000 |
| 同时在线 | 200 - 1,000 |
| 数据库 | PostgreSQL 15 |
| 服务器 | 1台云服务器（2核4G） |
| 预估成本 | ¥200-500/月 |

**改动清单**：
1. SQLite → PostgreSQL 迁移（数据导出/导入，< 1小时）
2. 添加 Redis 缓存热点数据（瓜田列表、用户积分）
3. 添加 JWT 无状态认证
4. 服务器部署到云（阿里云/腾讯云/HC Cloud）

#### Phase 3: 成长期
**触发条件**：日活突破 10,000 或 并发超过 1,000

| 指标 | 目标 |
|------|------|
| 日活用户 | 10,000 - 100,000 |
| 同时在线 | 1,000 - 5,000 |
| 数据库 | PostgreSQL + 只读副本 |
| 服务器 | 2-4台云服务器 |
| 预估成本 | ¥1,000-3,000/月 |

**改动清单**：
1. 数据库读写分离
2. 添加 Redis Session 和缓存
3. 后端服务水平扩展（多实例 + 负载均衡）
4. 添加异步任务队列（Celery + RabbitMQ）
5. LLM 请求增加重试和降级逻辑
6. 添加请求限流和熔断

#### Phase 4: 爆发期
**触发条件**：日活突破 100,000 或 需要 99.9% 可用性

| 指标 | 目标 |
|------|------|
| 日活用户 | 100,000+ |
| 同时在线 | 5,000+ |
| 数据库 | PostgreSQL 分片 + 跨区副本 |
| 服务器 | K8s 容器编排 |
| 预估成本 | ¥5,000+/月 |

**改动清单**：
1. K8s 容器化部署
2. 数据库分片（按用户 ID 或地域）
3. 消息队列解耦非核心流程
4. 多云/跨区部署
5. 全链路监控和告警

### 扩容决策树

## ADDED Requirements

### Requirement: 底部 Tab 导航
系统 SHALL 提供底部 Tab 导航栏，包含五个 Tab：瓜田、求证、发布、社区、我的。MVP 阶段"发布"和"社区"Tab 显示占位页面。

#### Scenario: 用户切换 Tab
- **WHEN** 用户点击底部 Tab
- **THEN** 对应页面内容切换显示，当前 Tab 图标高亮

#### Scenario: 移动端适配
- **WHEN** 在移动设备上访问
- **THEN** 底部导航固定在屏幕底部，内容区域可滚动，最大宽度 480px 居中显示

---

### Requirement: 瓜田信息流
系统 SHALL 以卡片列表形式展示"待求证"内容，每个卡片包含标题、封面图、参与人数、剩余开奖时间。

#### Scenario: 浏览瓜田
- **WHEN** 用户进入瓜田 Tab
- **THEN** 展示列表形式的待求证内容卡片

#### Scenario: 分类筛选
- **WHEN** 用户点击分类标签（娱乐/科技/生活科普/社会热点/历史/财经）
- **THEN** 信息流仅展示对应分类的内容

#### Scenario: 下拉刷新
- **WHEN** 用户下拉页面
- **THEN** 刷新信息流内容

---

### Requirement: 猜瓜（核心玩法）
系统 SHALL 允许用户对瓜田中的内容做出"真"或"假"的判断，并可选提交佐证。猜对获积分奖励，猜错看科普视频。核心驱动是段位升级——用户通过不断猜瓜积累战绩，提升段位、解锁称号，获得成长成就感。

#### Scenario: 进入猜瓜详情
- **WHEN** 用户点击瓜田卡片
- **THEN** 进入详情页，展示完整内容描述、真/假判断按钮、当前参与人数和双方比例

#### Scenario: 提交判断
- **WHEN** 用户选择"真"或"假"并确认
- **THEN** 记录用户的判断结果，显示"已参与"状态，不扣除任何积分

#### Scenario: 提交佐证（核心机制）
- **WHEN** 用户提交判断后，可填写"你的佐证"（选填）
- **THEN** 佐证内容与判断一起保存，提示"被点赞多的佐证会获得额外奖励"
- **佐证设计理念**：不是猜对就行，而是让用户学会如何思考。佐证内容可被其他用户查看和点赞，形成讨论氛围

#### Scenario: 已参与状态
- **WHEN** 用户查看已参与的内容
- **THEN** 显示"已参与"标签、所选方向、已提交的佐证（如有）

#### Scenario: 辅助功能-邀请好友
- **WHEN** 用户在详情页点击"邀请好友来猜"
- **THEN** 生成分享链接，复制到剪贴板，好友点击后进入同一猜瓜页面

---

### Requirement: 佐证机制（核心差异化）
系统 SHALL 实现用户佐证功能，让用户不只是"猜"，而是学会"如何思考"。佐证被点赞/选为最佳可获额外奖励，形成正向学习循环。

#### Scenario: 查看佐证排行榜
- **WHEN** 开奖后用户进入结果页
- **THEN** 展示佐证排行榜：最佳佐证置顶 + 按点赞数排序
- **佐证卡片内容**：用户名、头像、判断方向、佐证内容、点赞数、踩数

#### Scenario: 点赞/踩佐证
- **WHEN** 用户对佐证点击点赞或踩
- **THEN** 实时更新点赞/踩数，被点赞的佐证作者获得积分奖励（+2积分/赞）

#### Scenario: 最佳佐证判定
- **WHEN** 系统判定某条佐证为最佳（综合点赞数、相关性、准确率）
- **THEN** 该用户获得额外奖励：+50积分、段位经验×1.5

#### Scenario: 佐证引导提示
- **WHEN** 用户填写佐证时
- **THEN** 显示引导文案，如"说说你的判断依据？查到了什么资料？"

---

### Requirement: AI 开奖与实锤报告
系统 SHALL 在开奖时间到达后，展示 AI 生成的实锤报告，包含证据链、来源可信度、倾向性判断。

#### Scenario: 开奖结果展示
- **WHEN** 开奖时间到达
- **THEN** 猜瓜页面变为"已开奖"状态，显示 AI 实锤报告

#### Scenario: 实锤报告结构
- **WHEN** 用户查看实锤报告
- **THEN** 报告包含：信息时间线、证据链列表、来源可信度标注（⭐1-5）、关键疑点分析、倾向性判断、免责声明

#### Scenario: 猜对奖励
- **WHEN** 用户判断方向与开奖结果一致
- **THEN** 获得积分奖励（+20~50），显示奖励动画。每日通过猜瓜获取的积分上限为 200

#### Scenario: 猜错处理
- **WHEN** 用户判断方向与开奖结果不一致
- **THEN** 不扣积分，弹出防骗科普小视频（约1分钟），观看倒计时结束后可关闭。视频内容与被猜内容主题相关

#### Scenario: 开奖后结果页
- **WHEN** 用户查看开奖结果
- **THEN** 同时展示：AI实锤报告 + 佐证排行榜 + 段位进度反馈

---

### Requirement: AI 一键求证
系统 SHALL 提供一键求证工具，用户输入文字或链接，AI 快速核查信息真伪。MVP 阶段每日免费 3 次，积分可兑换额外次数。

#### Scenario: 提交求证请求
- **WHEN** 用户在求证 Tab 输入文字或链接并提交
- **THEN** 显示加载动画，模拟 AI 分析过程，返回简版求证结果

#### Scenario: 免费额度用尽
- **WHEN** 用户当日已使用 3 次免费求证
- **THEN** 提示"今日免费次数已用完，可用积分兑换额外次数"，展示兑换入口（50 积分/次）

#### Scenario: 积分兑换次数
- **WHEN** 用户用积分兑换求证次数
- **THEN** 扣除对应积分，增加当日可用次数，继续提交求证

#### Scenario: 求证结果展示
- **WHEN** AI 返回求证结果
- **THEN** 展示结果摘要：可信度等级、关键证据、倾向性判断

---

### Requirement: 积分系统
系统 SHALL 实现积分获取与消耗机制，每日获取有上限，防止刷分和滥用。积分形成"猜瓜赚积分→佐证获积分→积分换工具→工具帮猜瓜→猜瓜更准"的闭环。

#### Scenario: 获取积分
- **WHEN** 用户完成以下行为
- **THEN** 积分余额自动增加：
  - 每日登录：+10（每日限1次）
  - 猜对瓜：+20~50（根据难度，每日上限 200）
  - 提交佐证：+5（每日上限 50）
  - 佐证被点赞：+2/赞（每日上限 100）
  - 佐证被选为最佳：+50
  - 发布高质量内容：+20~100
  - 邀请好友注册：+50

#### Scenario: 积分明细
- **WHEN** 用户在"我的"页面查看积分
- **THEN** 显示当前余额、积分变动记录（时间、来源、数量）

#### Scenario: 积分使用（MVP 先做第1项）
- **WHEN** 用户消耗积分
- **THEN** 可用于：
  - ✅ 兑换工具箱额外求证次数（50积分/次）
  - ⏳ 深度调查（200积分/次）— 后续版本
  - ⏳ 帖子曝光度加成（100积分/次）— 后续版本
  - ⏳ 头像框/称号装饰（100-1000积分）— 后续版本
  - ⏳ 公益捐赠 — 后续版本
  - ⏳ RMB 充值（¥1=10积分）— 二期实现

#### Scenario: 猜错不扣积分
- **WHEN** 用户猜错瓜
- **THEN** 积分余额不变，仅需观看科普视频

### Requirement: 段位成长体系（核心留存引擎）
系统 SHALL 根据用户表现计算段位，从低到高共 7 级。前期升级快获得成就感，中后期需要持续参与才能升级，顶级按年计。禁止 VIP 加速，保持公平。

#### Scenario: 段位数值
- **WHEN** 系统计算段位
- **THEN** 使用以下标准：

| 段位 | 等级 | 猜对次数 | 准确率 | 参与次数 | 预估耗时 |
|------|------|---------|--------|---------|---------|
| 吃瓜群众 | Lv.0 | 0 | - | 0 | 初始 |
| 瓜田新手 | Lv.1 | 10 | 25% | 30 | 1-2周 |
| 鉴瓜学徒 | Lv.2 | 30 | 35% | 80 | 1个月 |
| 瓜田侦探 | Lv.3 | 80 | 45% | 180 | 3个月 |
| 鉴瓜达人 | Lv.4 | 200 | 50% | 400 | 6个月 |
| 鉴瓜大师 | Lv.5 | 400 | 55% | 800 | 12个月 |
| 见微先知 | Lv.6 | 800 | 60% | 1500 | 1.5年+ |

#### Scenario: 段位展示
- **WHEN** 用户查看"我的"页面
- **THEN** 显示当前段位名称、专属段位图标、距离下一级进度条，激励用户持续参与

#### Scenario: 段位升级
- **WHEN** 用户综合表现（准确率、参与次数、猜对总数）达标
- **THEN** 段位自动升级，全屏播放升级动画，展示新段位名称和图标

#### Scenario: 段位列表
- **WHEN** 用户查看段位说明
- **THEN** 展示全部 7 级段位：吃瓜群众→瓜田新手→鉴瓜学徒→瓜田侦探→鉴瓜达人→鉴瓜大师→见微先知

#### Scenario: 鉴瓜大师称号
- **WHEN** 用户累计猜对次数达到 50 次
- **THEN** 自动获得"鉴瓜大师"专属称号和徽章，在个人主页和社区中展示

#### Scenario: 猜瓜后段位进度反馈
- **WHEN** 用户猜瓜开奖后
- **THEN** 在结果页展示段位进度变化（如"段位经验 +15，距离鉴瓜侦探还差 8 次"），强化成长反馈

#### Scenario: 佐证加速升级
- **WHEN** 用户的佐证被选为最佳
- **THEN** 获得段位经验×1.5 加速奖励，鼓励用户积极思考

### 后端架构

#### Requirement: Agent Pipeline
系统 SHALL 实现多 Agent 协作系统，通过 LangGraph 编排完成信息求证任务。

##### Agent 角色
| Agent | 职责 |
|-------|------|
| **审核 Agent** | 内容安全初筛，检测敏感内容 |
| **搜集 Agent** | 全网检索，抓取相关信息 |
| **验证 Agent** | 交叉验证信息来源，评估可信度 |
| **分析 Agent** | 整理证据链，生成结构化报告 |

##### Pipeline 流程
```
用户请求 → 审核 Agent → [通过] → 搜集 Agent → 验证 Agent → 分析 Agent → 返回报告
                    → [拦截] → 返回审核失败
```

##### API 端点
| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/verify` | POST | 一键求证（调用完整 Pipeline） |
| `/api/v1/moderate` | POST | 内容审核 |
| `/api/v1/health` | GET | 健康检查 |
| `/api/v1/models/providers` | GET | 列出所有可用模型提供商 |
| `/api/v1/models/set-provider` | POST | 切换当前使用的模型 |

#### Requirement: 多模型支持
系统 SHALL 支持主流大模型提供商，国产模型全覆盖，可随时切换主模型。

##### 切换机制

- 系统默认主模型通过 `LLM_PROVIDER` 环境变量设置（默认：deepseek）
- 支持运行时通过 API 切换主模型
- 主模型调用失败时自动降级到备用模型（Groq → OpenAI → Claude）
- 所有模型统一使用 OpenAI 兼容 SDK 调用，无需单独适配

##### 设计原则
- **国产优先**：默认使用 DeepSeek，国产模型全部优先支持
- **不偏不倚**：7 家国产模型平等对待，接口完全一致
- **可扩展性**：新增模型只需添加配置，无需改代码
- **容错保障**：多模型降级确保服务可用，不依赖单一供应商
