## 观微项目落地计划

> 目标：从 demo 变成评委能上手操作、有真实功能的产品。
> 约束：零预算，Cloudflare Pages 免费部署，比赛阶段不备案。

---

### 一、当前状态

已完成的部分：React 前端全套页面（瓜田/社区/发布/求证/我的）、国风 UI 设计系统、Python FastAPI 后端（LangGraph 多 Agent 管线 + 12 家 LLM 供应商 + SQLite + JWT 鉴权）、EXIF 元数据鉴定工具（纯前端，已可用）。

核心问题：前端和后端是割裂的。评委打开部署的 app，看到的是 mock 数据；后端跑在本地 localhost，没人能访问。工具箱六个入口只有一个能用。

---

### 二、部署方案（零成本）

前端部署到 Cloudflare Pages，这没有争议。问题在后端。

Python FastAPI 后端无法跑在 Cloudflare 上。免费选项有三个：

**方案 A：Render.com 免费层**
推荐。支持 Python，免费层 750 小时/月，自带 HTTPS，部署时连 GitHub 仓库自动构建。缺点是冷启动慢（闲置 15 分钟后首次请求约 30 秒），但比赛演示场景可以接受——评委点一下等一等就行。数据库用 Render 自带的免费 PostgreSQL（90 天限制，比赛够用）或继续用 SQLite（配合持久化磁盘 $0.25/GB/月，几乎免费）。

**方案 B：Hugging Face Spaces**
免费 GPU/CPU，适合有 AI 标签的项目，比赛叙事加分。但 Python 环境定制灵活度不如 Render，且冷启动更慢。

**方案 C：前端纯静态 + Groq 免费 API 直连**
不部署 Python 后端，把关键工具（情绪操控检测、一键求证）做成前端直调 Groq 免费 API。Groq 免费层每分钟 30 请求、每天 14400 请求，比赛演示绰绰有余。好处是零运维，坏处是 LangGraph 管线用不上，Agent 流程只能在前端模拟。

**建议：方案 A + C 混合。** 后端部署到 Render 保持完整架构（评委看代码时能展示 LangGraph），同时前端关键工具支持 Groq 直连作为 fallback（确保演示时不依赖后端存活状态）。EXIF 工具已经纯前端，不受影响。

### 三、工具箱落地优先级

七个工具按"投入产出比"排序：

#### 第一梯队：比赛前必须做真

**1. EXIF 元数据鉴定** ✅ 已完成
纯前端，exifr 库，风险评估引擎。评委上传照片立刻看到结果。

**2. 情绪操控检测** ⏳ 下一步
这是最能打动评委的工具。技术上不复杂——本质是 LLM prompt engineering 做话术拆解。

实现方案：
- 前端输入一段文字（新闻标题、营销号文案、聊天记录）
- 调用 Groq 免费 API（llama-3.3-70b 或 mixtral-8x7b），用精心设计的 system prompt 做修辞分析
- 输出：识别出的操控手法（恐惧诉求/虚假两难/人身攻击/诉诸权威/偷换概念/数据操纵/情绪绑架）+ 原文逐句标注 + 客观事实重述

关键在 prompt 设计。不是让 LLM "判断真假"，而是让它像语文老师一样拆解话术结构——"这句话用了什么修辞手法？试图激发什么情绪？隐藏了什么前提？"这完美契合"AI 辅助人做判断"的理念。

前端展示：文字输入 → 逐句高亮标注（红色=高风险操控、橙色=轻微操控、绿色=客观陈述）→ 右侧面板列出识别到的操控手法和解释 → 底部给出"客观重述"（去掉操控话术后的事实版本）。

预计工期：3-5 天（prompt 调试 2 天 + 前端 UI 2 天）。

**3. 一键求证** ⏳ 简化版
完整的多 Agent 求证管线依赖后端 LangGraph。

后端 LangGraph 管线保持部署在 Render 上，作为"完整架构展示"，评委看代码时能看到真正的多 Agent 协作。

#### 第二梯队：有时间就做

**4. 反向搜图**
前端上传图片 → 提取图片哈希（pHash/dHash）→ 生成 Google Images / Bing Visual Search 搜索链接 → 在新标签页打开搜索结果。不需要自己建图片数据库，只是帮用户自动化"用图搜图"这个动作。技术上用 Canvas API 算哈希，纯前端，半天能做完。

**5. 时间线重建**
输入一段事件描述或新闻 → LLM 提取关键时间节点 → 前端渲染为可视化时间线。本质还是 prompt engineering + 前端展示。可以复用已有的 EvidenceTimeline 组件。

#### 第三梯队：比赛后迭代

**6. 洗稿检测**
需要文本相似度算法（SimHash 或余弦相似度），以及一个对比文本库。短期内没有数据源，先做 UI 占位。

**7. 截图伪造检测（ELA）**
需要 OpenCV 做 JPEG 重压缩分析，计算量大，纯前端不现实（WebAssembly 编译 OpenCV 体积太大）。需要后端支持，等基础设施稳定后再做。

### 四、内容策略

关于 mock 数据：完全没问题，但要做对。

**瓜田页面：** 12 条 mock 瓜没问题，picsum.photos 做封面图也行。但建议做两个优化：

第一，把标题和描述换成更贴近真实热点事件的风格（不要用"某顶流男星"这种明显的占位符，换成有具体细节但虚构的内容，比如"某新能源车品牌被曝冬季续航虚标 40%，车主集体维权"）。评委不会去搜这些事件是否真实，但标题的质感直接影响第一印象。

第二，给每条瓜加一个"已揭晓"状态和结果。评委点进去能看到完整的"实锤报告"，而不是一个 pending 状态的空壳。报告数据可以是 mock 的，但要有。

**社区页面：** 22 条 mock 帖子没问题。建议把帖子类型分布调一下——不要全是 normal，加几条 charity（公益求助）和 hot（热帖），让瀑布流看起来有多样性。picsum.photos 的随机图高度差异已经做了，视觉上不会穿帮。

**求证页面：** 一键求证的结果目前调的是后端 mock。如果接了 Groq API，结果就是真实生成的，不再是 mock。这是关键转变——评委输入一段文字，真的返回了 LLM 分析结果，哪怕只是单次调用而非多 Agent 流程，体验也完全不同。

**用户系统：** 登录注册保持 mock 或接后端都行。评委不太会去测用户系统，但如果有后端的话接上更完整。

### 五、前端打磨清单

这些不是新功能，是让现有功能看起来不像 demo 的细节：

1. **空状态处理**：瓜田加载失败时显示"网络异常，点击重试"而不是白屏；社区帖子为空时显示引导文案。

2. **错误边界**：API 调用失败时 fallback 到 mock 数据而不是崩溃。目前 verificationStore 里已经有 catch fallback，但其他页面没有。

3. **瓜田详情页完善**：目前点进瓜的详情，如果是 mock 数据，佐证和报告可能加载不出来。确保 mock 模式下也能展示完整的猜测 + 佐证 + 报告流程。

4. **发布页的 AI 预检**：目前调的是 `api.moderate`，后端不在线时会失败。改成 fallback 到前端本地正则检测（contentFilter.ts 已经有框架，填充关键词即可）。

5. **TabBar 状态**：当前页面的 tab 高亮应该和路由同步，确保从工具页返回时 tab 状态正确。

6. **页面转场**：工具箱页面（/tools/exif）没有 TabBar，返回时应该平滑过渡。考虑给工具页加一个统一的顶部导航栏样式。

### 六、比赛展示策略

评委大概会花 3-5 分钟看你的项目。设计一条演示路线：

**第一步（30 秒）：** 打开 app 首页，瓜田大图卡片流，视觉第一印象——"这个做得好看"。国风设计在这里加分。

**第二步（1 分钟）：** 点进一条瓜，参与猜测，看佐证，看实锤报告。展示产品的核心玩法和社区互动机制。

**第三步（2 分钟）：** 进入求证页面，打开工具箱。先演示 EXIF 鉴定——上传一张截图，实时返回分析结果。然后演示情绪操控检测——输入一段营销号文案，逐句高亮标注操控话术。这两个是"点下去真的能用"的工具。

**第四步（30 秒）：** 展示一键求证的简化版，输入一段文字，LLM 实时返回分析。说明完整版是多 Agent 协作流程（展示架构图或后端代码）。

**第五步（30 秒）：** 展示后端代码——LangGraph 管线、12 家 LLM 供应商、Agent 编排。评委不一定会看，但放在 README 里能证明技术深度。

这条路线的逻辑是：好看 → 好玩 → 真的能用 → 有深度。每一步都有锚点。

### 八、关于 Groq API Key

Groq 免费注册就能拿 API key，不需要信用卡。注册地址 https://console.groq.com，免费层限制：每分钟 30 请求，每天 14400 请求。比赛演示完全够用。

如果担心评委多人同时测导致触发限流，可以准备两个 key 做轮换。或者用 Cloudflare Workers AI 做备用——Cloudflare 账号免费，Workers AI 每天 10000 次推理免费，但模型选择不如 Groq 多。

前端直调 Groq API 有个安全问题：API key 暴露在浏览器端。但比赛场景下这不是问题——这不是一个要上线的产品，评委不会去扒你的 key。如果实在介意，可以用 Cloudflare Workers 做一层代理转发，key 放在 Worker 环境变量里。
